38/// Takes ownership and "forgets" about the value **without running its destructor**.
39///
40/// Any resources the value manages, such as heap memory or a file handle, will linger
41/// forever in an unreachable state. However, it does not guarantee that pointers
42/// to this memory will remain valid.
43///
44/// * If you want to leak memory, see [`Box::leak`].
45/// * If you want to obtain a raw pointer to the memory, see [`Box::into_raw`].
46/// * If you want to dispose of a value properly, running its destructor, see
47/// [`mem::drop`].
48///
49/// # Safety
50///
51/// `forget` is not marked as `unsafe`, because Rust's safety guarantees
52/// do not include a guarantee that destructors will always run. For example,
53/// a program can create a reference cycle using [`Rc`][rc], or call
54/// [`process::exit`][exit] to exit without running destructors. Thus, allowing
55/// `mem::forget` from safe code does not fundamentally change Rust's safety
56/// guarantees.
57///
58/// That said, leaking resources such as memory or I/O objects is usually undesirable.
59/// The need comes up in some specialized use cases for FFI or unsafe code, but even
60/// then, [`ManuallyDrop`] is typically preferred.
61///
62/// Because forgetting a value is allowed, any `unsafe` code you write must
63/// allow for this possibility. You cannot return a value and expect that the
64/// caller will necessarily run the value's destructor.
65///
66/// [rc]: ../../std/rc/struct.Rc.html
67/// [exit]: ../../std/process/fn.exit.html
68///
69/// # Examples
70///
71/// The canonical safe use of `mem::forget` is to circumvent a value's destructor
72/// implemented by the `Drop` trait. For example, this will leak a `File`, i.e. reclaim
73/// the space taken by the variable but never close the underlying system resource:
74///
75/// ```no_run
76/// use std::mem;
77/// use std::fs::File;
78///
79/// let file = File::open("foo.txt").unwrap();
80/// mem::forget(file);
81/// ```
82///
83/// This is useful when the ownership of the underlying resource was previously
84/// transferred to code outside of Rust, for example by transmitting the raw
85/// file descriptor to C code.
86///
87/// # Relationship with `ManuallyDrop`
88///
89/// While `mem::forget` can also be used to transfer *memory* ownership, doing so is error-prone.
90/// [`ManuallyDrop`] should be used instead. Consider, for example, this code:
91///
92/// ```
93/// use std::mem;
94///
95/// let mut v = vec![65, 122];
96/// // Build a `String` using the contents of `v`
97/// let s = unsafe { String::from_raw_parts(v.as_mut_ptr(), v.len(), v.capacity()) };
98/// // leak `v` because its memory is now managed by `s`
99/// mem::forget(v);  // ERROR - v is invalid and must not be passed to a function
100/// assert_eq!(s, "Az");
101/// // `s` is implicitly dropped and its memory deallocated.
102/// ```
103///
104/// There are two issues with the above example:
105///
106/// * If more code were added between the construction of `String` and the invocation of
107///   `mem::forget()`, a panic within it would cause a double free because the same memory
108///   is handled by both `v` and `s`.
109/// * After calling `v.as_mut_ptr()` and transmitting the ownership of the data to `s`,
110///   the `v` value is invalid. Even when a value is just moved to `mem::forget` (which won't
111///   inspect it), some types have strict requirements on their values that
112///   make them invalid when dangling or no longer owned. Using invalid values in any
113///   way, including passing them to or returning them from functions, constitutes
114///   undefined behavior and may break the assumptions made by the compiler.
115///
116/// Switching to `ManuallyDrop` avoids both issues:
117///
118/// ```
119/// use std::mem::ManuallyDrop;
120///
121/// let v = vec![65, 122];
122/// // Before we disassemble `v` into its raw parts, make sure it
123/// // does not get dropped!
124/// let mut v = ManuallyDrop::new(v);
125/// // Now disassemble `v`. These operations cannot panic, so there cannot be a leak.
126/// let (ptr, len, cap) = (v.as_mut_ptr(), v.len(), v.capacity());
127/// // Finally, build a `String`.
128/// let s = unsafe { String::from_raw_parts(ptr, len, cap) };
129/// assert_eq!(s, "Az");
130/// // `s` is implicitly dropped and its memory deallocated.
131/// ```
132///
133/// `ManuallyDrop` robustly prevents double-free because we disable `v`'s destructor
134/// before doing anything else. `mem::forget()` doesn't allow this because it consumes its
135/// argument, forcing us to call it only after extracting anything we need from `v`. Even
136/// if a panic were introduced between construction of `ManuallyDrop` and building the
137/// string (which cannot happen in the code as shown), it would result in a leak and not a
138/// double free. In other words, `ManuallyDrop` errs on the side of leaking instead of
139/// erring on the side of (double-)dropping.
140///
141/// Also, `ManuallyDrop` prevents us from having to "touch" `v` after transferring the
142/// ownership to `s` â€” the final step of interacting with `v` to dispose of it without
143/// running its destructor is entirely avoided.
144///
145/// [`Box`]: ../../std/boxed/struct.Box.html
146/// [`Box::leak`]: ../../std/boxed/struct.Box.html#method.leak
147/// [`Box::into_raw`]: ../../std/boxed/struct.Box.html#method.into_raw
148/// [`mem::drop`]: drop
149/// [ub]: ../../reference/behavior-considered-undefined.html
150#[inline]
151#[rustc_const_stable(feature = "const_forget", since = "1.46.0")]
152#[stable(feature = "rust1", since = "1.0.0")]
153#[rustc_diagnostic_item = "mem_forget"]
154#[cfg(not(feature = "ferrocene_certified"))]
155pub const fn forget<T>(t: T) {
156    let _ = ManuallyDrop::new(t);
157}

170/// Returns the size of a type in bytes.
171///
172/// More specifically, this is the offset in bytes between successive elements
173/// in an array with that item type including alignment padding. Thus, for any
174/// type `T` and length `n`, `[T; n]` has a size of `n * size_of::<T>()`.
175///
176/// In general, the size of a type is not stable across compilations, but
177/// specific types such as primitives are.
178///
179/// The following table gives the size for primitives.
180///
181/// Type | `size_of::<Type>()`
182/// ---- | ---------------
183/// () | 0
184/// bool | 1
185/// u8 | 1
186/// u16 | 2
187/// u32 | 4
188/// u64 | 8
189/// u128 | 16
190/// i8 | 1
191/// i16 | 2
192/// i32 | 4
193/// i64 | 8
194/// i128 | 16
195/// f32 | 4
196/// f64 | 8
197/// char | 4
198///
199/// Furthermore, `usize` and `isize` have the same size.
200///
201/// The types [`*const T`], `&T`, [`Box<T>`], [`Option<&T>`], and `Option<Box<T>>` all have
202/// the same size. If `T` is `Sized`, all of those types have the same size as `usize`.
203///
204/// The mutability of a pointer does not change its size. As such, `&T` and `&mut T`
205/// have the same size. Likewise for `*const T` and `*mut T`.
206///
207/// # Size of `#[repr(C)]` items
208///
209/// The `C` representation for items has a defined layout. With this layout,
210/// the size of items is also stable as long as all fields have a stable size.
211///
212/// ## Size of Structs
213///
214/// For `struct`s, the size is determined by the following algorithm.
215///
216/// For each field in the struct ordered by declaration order:
217///
218/// 1. Add the size of the field.
219/// 2. Round up the current size to the nearest multiple of the next field's [alignment].
220///
221/// Finally, round the size of the struct to the nearest multiple of its [alignment].
222/// The alignment of the struct is usually the largest alignment of all its
223/// fields; this can be changed with the use of `repr(align(N))`.
224///
225/// Unlike `C`, zero sized structs are not rounded up to one byte in size.
226///
227/// ## Size of Enums
228///
229/// Enums that carry no data other than the discriminant have the same size as C enums
230/// on the platform they are compiled for.
231///
232/// ## Size of Unions
233///
234/// The size of a union is the size of its largest field.
235///
236/// Unlike `C`, zero sized unions are not rounded up to one byte in size.
237///
238/// # Examples
239///
240/// ```
241/// // Some primitives
242/// assert_eq!(4, size_of::<i32>());
243/// assert_eq!(8, size_of::<f64>());
244/// assert_eq!(0, size_of::<()>());
245///
246/// // Some arrays
247/// assert_eq!(8, size_of::<[i32; 2]>());
248/// assert_eq!(12, size_of::<[i32; 3]>());
249/// assert_eq!(0, size_of::<[i32; 0]>());
250///
251///
252/// // Pointer size equality
253/// assert_eq!(size_of::<&i32>(), size_of::<*const i32>());
254/// assert_eq!(size_of::<&i32>(), size_of::<Box<i32>>());
255/// assert_eq!(size_of::<&i32>(), size_of::<Option<&i32>>());
256/// assert_eq!(size_of::<Box<i32>>(), size_of::<Option<Box<i32>>>());
257/// ```
258///
259/// Using `#[repr(C)]`.
260///
261/// ```
262/// #[repr(C)]
263/// struct FieldStruct {
264///     first: u8,
265///     second: u16,
266///     third: u8
267/// }
268///
269/// // The size of the first field is 1, so add 1 to the size. Size is 1.
270/// // The alignment of the second field is 2, so add 1 to the size for padding. Size is 2.
271/// // The size of the second field is 2, so add 2 to the size. Size is 4.
272/// // The alignment of the third field is 1, so add 0 to the size for padding. Size is 4.
273/// // The size of the third field is 1, so add 1 to the size. Size is 5.
274/// // Finally, the alignment of the struct is 2 (because the largest alignment amongst its
275/// // fields is 2), so add 1 to the size for padding. Size is 6.
276/// assert_eq!(6, size_of::<FieldStruct>());
277///
278/// #[repr(C)]
279/// struct TupleStruct(u8, u16, u8);
280///
281/// // Tuple structs follow the same rules.
282/// assert_eq!(6, size_of::<TupleStruct>());
283///
284/// // Note that reordering the fields can lower the size. We can remove both padding bytes
285/// // by putting `third` before `second`.
286/// #[repr(C)]
287/// struct FieldStructOptimized {
288///     first: u8,
289///     third: u8,
290///     second: u16
291/// }
292///
293/// assert_eq!(4, size_of::<FieldStructOptimized>());
294///
295/// // Union size is the size of the largest field.
296/// #[repr(C)]
297/// union ExampleUnion {
298///     smaller: u8,
299///     larger: u16
300/// }
301///
302/// assert_eq!(2, size_of::<ExampleUnion>());
303/// ```
304///
305/// [alignment]: align_of
306/// [`*const T`]: primitive@pointer
307/// [`Box<T>`]: ../../std/boxed/struct.Box.html
308/// [`Option<&T>`]: crate::option::Option
309///
310#[inline(always)]
311#[must_use]

313#[rustc_promotable]
314#[rustc_const_stable(feature = "const_mem_size_of", since = "1.24.0")]
315#[rustc_diagnostic_item = "mem_size_of"]
316pub const fn size_of<T>() -> usize {
317    intrinsics::size_of::<T>()
318}

350/// Returns the size of the pointed-to value in bytes.
351///
352/// This is usually the same as [`size_of::<T>()`]. However, when `T` *has* no
353/// statically-known size, e.g., a slice [`[T]`][slice] or a [trait object],
354/// then `size_of_val_raw` can be used to get the dynamically-known size.
355///
356/// # Safety
357///
358/// This function is only safe to call if the following conditions hold:
359///
360/// - If `T` is `Sized`, this function is always safe to call.
361/// - If the unsized tail of `T` is:
362///     - a [slice], then the length of the slice tail must be an initialized
363///       integer, and the size of the *entire value*
364///       (dynamic tail length + statically sized prefix) must fit in `isize`.
365///       For the special case where the dynamic tail length is 0, this function
366///       is safe to call.
367//        NOTE: the reason this is safe is that if an overflow were to occur already with size 0,
368//        then we would stop compilation as even the "statically known" part of the type would
369//        already be too big (or the call may be in dead code and optimized away, but then it
370//        doesn't matter).
371///     - a [trait object], then the vtable part of the pointer must point
372///       to a valid vtable acquired by an unsizing coercion, and the size
373///       of the *entire value* (dynamic tail length + statically sized prefix)
374///       must fit in `isize`.
375///     - an (unstable) [extern type], then this function is always safe to
376///       call, but may panic or otherwise return the wrong value, as the
377///       extern type's layout is not known. This is the same behavior as
378///       [`size_of_val`] on a reference to a type with an extern type tail.
379///     - otherwise, it is conservatively not allowed to call this function.
380///
381/// [`size_of::<T>()`]: size_of
382/// [trait object]: ../../book/ch17-02-trait-objects.html
383/// [extern type]: ../../unstable-book/language-features/extern-types.html
384///
385/// # Examples
386///
387/// ```
388/// #![feature(layout_for_ptr)]
389/// use std::mem;
390///
391/// assert_eq!(4, size_of_val(&5i32));
392///
393/// let x: [u8; 13] = [0; 13];
394/// let y: &[u8] = &x;
395/// assert_eq!(13, unsafe { mem::size_of_val_raw(y) });
396/// ```
397#[inline]
398#[must_use]
399#[unstable(feature = "layout_for_ptr", issue = "69835")]
400#[cfg(not(feature = "ferrocene_certified"))]
401pub const unsafe fn size_of_val_raw<T: ?Sized>(val: *const T) -> usize {
402    // SAFETY: the caller must provide a valid raw pointer
403    unsafe { intrinsics::size_of_val(val) }
404}

549/// Returns `true` if dropping values of type `T` matters.
550///
551/// This is purely an optimization hint, and may be implemented conservatively:
552/// it may return `true` for types that don't actually need to be dropped.
553/// As such always returning `true` would be a valid implementation of
554/// this function. However if this function actually returns `false`, then you
555/// can be certain dropping `T` has no side effect.
556///
557/// Low level implementations of things like collections, which need to manually
558/// drop their data, should use this function to avoid unnecessarily
559/// trying to drop all their contents when they are destroyed. This might not
560/// make a difference in release builds (where a loop that has no side-effects
561/// is easily detected and eliminated), but is often a big win for debug builds.
562///
563/// Note that [`drop_in_place`] already performs this check, so if your workload
564/// can be reduced to some small number of [`drop_in_place`] calls, using this is
565/// unnecessary. In particular note that you can [`drop_in_place`] a slice, and that
566/// will do a single needs_drop check for all the values.
567///
568/// Types like Vec therefore just `drop_in_place(&mut self[..])` without using
569/// `needs_drop` explicitly. Types like [`HashMap`], on the other hand, have to drop
570/// values one at a time and should use this API.
571///
572/// [`drop_in_place`]: crate::ptr::drop_in_place
573/// [`HashMap`]: ../../std/collections/struct.HashMap.html
574///
575/// # Examples
576///
577/// Here's an example of how a collection might make use of `needs_drop`:
578///
579/// ```
580/// use std::{mem, ptr};
581///
582/// pub struct MyCollection<T> {
583/// #   data: [T; 1],
584///     /* ... */
585/// }
586/// # impl<T> MyCollection<T> {
587/// #   fn iter_mut(&mut self) -> &mut [T] { &mut self.data }
588/// #   fn free_buffer(&mut self) {}
589/// # }
590///
591/// impl<T> Drop for MyCollection<T> {
592///     fn drop(&mut self) {
593///         unsafe {
594///             // drop the data
595///             if mem::needs_drop::<T>() {
596///                 for x in self.iter_mut() {
597///                     ptr::drop_in_place(x);
598///                 }
599///             }
600///             self.free_buffer();
601///         }
602///     }
603/// }
604/// ```
605#[inline]
606#[must_use]
607#[stable(feature = "needs_drop", since = "1.21.0")]
608#[rustc_const_stable(feature = "const_mem_needs_drop", since = "1.36.0")]
609#[rustc_diagnostic_item = "needs_drop"]
610#[cfg(not(feature = "ferrocene_certified"))]
611pub const fn needs_drop<T: ?Sized>() -> bool {
612    intrinsics::needs_drop::<T>()
613}

615/// Returns the value of type `T` represented by the all-zero byte-pattern.
616///
617/// This means that, for example, the padding byte in `(u8, u16)` is not
618/// necessarily zeroed.
619///
620/// There is no guarantee that an all-zero byte-pattern represents a valid value
621/// of some type `T`. For example, the all-zero byte-pattern is not a valid value
622/// for reference types (`&T`, `&mut T`) and function pointers. Using `zeroed`
623/// on such types causes immediate [undefined behavior][ub] because [the Rust
624/// compiler assumes][inv] that there always is a valid value in a variable it
625/// considers initialized.
626///
627/// This has the same effect as [`MaybeUninit::zeroed().assume_init()`][zeroed].
628/// It is useful for FFI sometimes, but should generally be avoided.
629///
630/// [zeroed]: MaybeUninit::zeroed
631/// [ub]: ../../reference/behavior-considered-undefined.html
632/// [inv]: MaybeUninit#initialization-invariant
633///
634/// # Examples
635///
636/// Correct usage of this function: initializing an integer with zero.
637///
638/// ```
639/// use std::mem;
640///
641/// let x: i32 = unsafe { mem::zeroed() };
642/// assert_eq!(0, x);
643/// ```
644///
645/// *Incorrect* usage of this function: initializing a reference with zero.
646///
647/// ```rust,no_run
648/// # #![allow(invalid_value)]
649/// use std::mem;
650///
651/// let _x: &i32 = unsafe { mem::zeroed() }; // Undefined behavior!
652/// let _y: fn() = unsafe { mem::zeroed() }; // And again!
653/// ```
654#[inline(always)]
655#[must_use]
656#[stable(feature = "rust1", since = "1.0.0")]
657#[allow(deprecated_in_future)]
658#[allow(deprecated)]
659#[rustc_diagnostic_item = "mem_zeroed"]
660#[track_caller]
661#[rustc_const_stable(feature = "const_mem_zeroed", since = "1.75.0")]
662#[cfg(not(feature = "ferrocene_certified"))]
663pub const unsafe fn zeroed<T>() -> T {
664    // SAFETY: the caller must guarantee that an all-zero value is valid for `T`.
665    unsafe {
666        intrinsics::assert_zero_valid::<T>();
667        MaybeUninit::zeroed().assume_init()
668    }
669}

671/// Bypasses Rust's normal memory-initialization checks by pretending to
672/// produce a value of type `T`, while doing nothing at all.
673///
674/// **This function is deprecated.** Use [`MaybeUninit<T>`] instead.
675/// It also might be slower than using `MaybeUninit<T>` due to mitigations that were put in place to
676/// limit the potential harm caused by incorrect use of this function in legacy code.
677///
678/// The reason for deprecation is that the function basically cannot be used
679/// correctly: it has the same effect as [`MaybeUninit::uninit().assume_init()`][uninit].
680/// As the [`assume_init` documentation][assume_init] explains,
681/// [the Rust compiler assumes][inv] that values are properly initialized.
682///
683/// Truly uninitialized memory like what gets returned here
684/// is special in that the compiler knows that it does not have a fixed value.
685/// This makes it undefined behavior to have uninitialized data in a variable even
686/// if that variable has an integer type.
687///
688/// Therefore, it is immediate undefined behavior to call this function on nearly all types,
689/// including integer types and arrays of integer types, and even if the result is unused.
690///
691/// [uninit]: MaybeUninit::uninit
692/// [assume_init]: MaybeUninit::assume_init
693/// [inv]: MaybeUninit#initialization-invariant
694#[inline(always)]
695#[must_use]
696#[deprecated(since = "1.39.0", note = "use `mem::MaybeUninit` instead")]
697#[stable(feature = "rust1", since = "1.0.0")]
698#[allow(deprecated_in_future)]
699#[allow(deprecated)]
700#[rustc_diagnostic_item = "mem_uninitialized"]
701#[track_caller]
702#[cfg(not(feature = "ferrocene_certified"))]
703pub unsafe fn uninitialized<T>() -> T {
704    // SAFETY: the caller must guarantee that an uninitialized value is valid for `T`.
705    unsafe {
706        intrinsics::assert_mem_uninitialized_valid::<T>();
707        let mut val = MaybeUninit::<T>::uninit();
708
709        // Fill memory with 0x01, as an imperfect mitigation for old code that uses this function on
710        // bool, nonnull, and noundef types. But don't do this if we actively want to detect UB.
711        if !cfg!(any(miri, sanitize = "memory")) {
712            val.as_mut_ptr().write_bytes(0x01, 1);
713        }
714
715        val.assume_init()
716    }
717}

719/// Swaps the values at two mutable locations, without deinitializing either one.
720///
721/// * If you want to swap with a default or dummy value, see [`take`].
722/// * If you want to swap with a passed value, returning the old value, see [`replace`].
723///
724/// # Examples
725///
726/// ```
727/// use std::mem;
728///
729/// let mut x = 5;
730/// let mut y = 42;
731///
732/// mem::swap(&mut x, &mut y);
733///
734/// assert_eq!(42, x);
735/// assert_eq!(5, y);
736/// ```
737#[inline]
738#[stable(feature = "rust1", since = "1.0.0")]
739#[rustc_const_stable(feature = "const_swap", since = "1.85.0")]
740#[rustc_diagnostic_item = "mem_swap"]
741#[cfg(not(feature = "ferrocene_certified"))]
742pub const fn swap<T>(x: &mut T, y: &mut T) {
743    // SAFETY: `&mut` guarantees these are typed readable and writable
744    // as well as non-overlapping.
745    unsafe { intrinsics::typed_swap_nonoverlapping(x, y) }
746}

748/// Replaces `dest` with the default value of `T`, returning the previous `dest` value.
749///
750/// * If you want to replace the values of two variables, see [`swap`].
751/// * If you want to replace with a passed value instead of the default value, see [`replace`].
752///
753/// # Examples
754///
755/// A simple example:
756///
757/// ```
758/// use std::mem;
759///
760/// let mut v: Vec<i32> = vec![1, 2];
761///
762/// let old_v = mem::take(&mut v);
763/// assert_eq!(vec![1, 2], old_v);
764/// assert!(v.is_empty());
765/// ```
766///
767/// `take` allows taking ownership of a struct field by replacing it with an "empty" value.
768/// Without `take` you can run into issues like these:
769///
770/// ```compile_fail,E0507
771/// struct Buffer<T> { buf: Vec<T> }
772///
773/// impl<T> Buffer<T> {
774///     fn get_and_reset(&mut self) -> Vec<T> {
775///         // error: cannot move out of dereference of `&mut`-pointer
776///         let buf = self.buf;
777///         self.buf = Vec::new();
778///         buf
779///     }
780/// }
781/// ```
782///
783/// Note that `T` does not necessarily implement [`Clone`], so it can't even clone and reset
784/// `self.buf`. But `take` can be used to disassociate the original value of `self.buf` from
785/// `self`, allowing it to be returned:
786///
787/// ```
788/// use std::mem;
789///
790/// # struct Buffer<T> { buf: Vec<T> }
791/// impl<T> Buffer<T> {
792///     fn get_and_reset(&mut self) -> Vec<T> {
793///         mem::take(&mut self.buf)
794///     }
795/// }
796///
797/// let mut buffer = Buffer { buf: vec![0, 1] };
798/// assert_eq!(buffer.buf.len(), 2);
799///
800/// assert_eq!(buffer.get_and_reset(), vec![0, 1]);
801/// assert_eq!(buffer.buf.len(), 0);
802/// ```
803#[inline]
804#[stable(feature = "mem_take", since = "1.40.0")]
805#[cfg(not(feature = "ferrocene_certified"))]
806pub fn take<T: Default>(dest: &mut T) -> T {
807    replace(dest, T::default())
808}

810/// Moves `src` into the referenced `dest`, returning the previous `dest` value.
811///
812/// Neither value is dropped.
813///
814/// * If you want to replace the values of two variables, see [`swap`].
815/// * If you want to replace with a default value, see [`take`].
816///
817/// # Examples
818///
819/// A simple example:
820///
821/// ```
822/// use std::mem;
823///
824/// let mut v: Vec<i32> = vec![1, 2];
825///
826/// let old_v = mem::replace(&mut v, vec![3, 4, 5]);
827/// assert_eq!(vec![1, 2], old_v);
828/// assert_eq!(vec![3, 4, 5], v);
829/// ```
830///
831/// `replace` allows consumption of a struct field by replacing it with another value.
832/// Without `replace` you can run into issues like these:
833///
834/// ```compile_fail,E0507
835/// struct Buffer<T> { buf: Vec<T> }
836///
837/// impl<T> Buffer<T> {
838///     fn replace_index(&mut self, i: usize, v: T) -> T {
839///         // error: cannot move out of dereference of `&mut`-pointer
840///         let t = self.buf[i];
841///         self.buf[i] = v;
842///         t
843///     }
844/// }
845/// ```
846///
847/// Note that `T` does not necessarily implement [`Clone`], so we can't even clone `self.buf[i]` to
848/// avoid the move. But `replace` can be used to disassociate the original value at that index from
849/// `self`, allowing it to be returned:
850///
851/// ```
852/// # #![allow(dead_code)]
853/// use std::mem;
854///
855/// # struct Buffer<T> { buf: Vec<T> }
856/// impl<T> Buffer<T> {
857///     fn replace_index(&mut self, i: usize, v: T) -> T {
858///         mem::replace(&mut self.buf[i], v)
859///     }
860/// }
861///
862/// let mut buffer = Buffer { buf: vec![0, 1] };
863/// assert_eq!(buffer.buf[0], 0);
864///
865/// assert_eq!(buffer.replace_index(0, 2), 0);
866/// assert_eq!(buffer.buf[0], 2);
867/// ```
868#[inline]
869#[stable(feature = "rust1", since = "1.0.0")]
870#[must_use = "if you don't need the old value, you can just assign the new value directly"]
871#[rustc_const_stable(feature = "const_replace", since = "1.83.0")]
872#[rustc_diagnostic_item = "mem_replace"]
873#[cfg(not(feature = "ferrocene_certified"))]
874pub const fn replace<T>(dest: &mut T, src: T) -> T {
875    // It may be tempting to use `swap` to avoid `unsafe` here. Don't!
876    // The compiler optimizes the implementation below to two `memcpy`s
877    // while `swap` would require at least three. See PR#83022 for details.
878
879    // SAFETY: We read from `dest` but directly write `src` into it afterwards,
880    // such that the old value is not duplicated. Nothing is dropped and
881    // nothing here can panic.
882    unsafe {
883        // Ideally we wouldn't use the intrinsics here, but going through the
884        // `ptr` methods introduces two unnecessary UbChecks, so until we can
885        // remove those for pointers that come from references, this uses the
886        // intrinsics instead so this stays very cheap in MIR (and debug).
887
888        let result = crate::intrinsics::read_via_copy(dest);
889        crate::intrinsics::write_via_move(dest, src);
890        result
891    }
892}

894/// Disposes of a value.
895///
896/// This does so by calling the argument's implementation of [`Drop`][drop].
897///
898/// This effectively does nothing for types which implement `Copy`, e.g.
899/// integers. Such values are copied and _then_ moved into the function, so the
900/// value persists after this function call.
901///
902/// This function is not magic; it is literally defined as
903///
904/// ```
905/// pub fn drop<T>(_x: T) {}
906/// ```
907///
908/// Because `_x` is moved into the function, it is automatically dropped before
909/// the function returns.
910///
911/// [drop]: Drop
912///
913/// # Examples
914///
915/// Basic usage:
916///
917/// ```
918/// let v = vec![1, 2, 3];
919///
920/// drop(v); // explicitly drop the vector
921/// ```
922///
923/// Since [`RefCell`] enforces the borrow rules at runtime, `drop` can
924/// release a [`RefCell`] borrow:
925///
926/// ```
927/// use std::cell::RefCell;
928///
929/// let x = RefCell::new(1);
930///
931/// let mut mutable_borrow = x.borrow_mut();
932/// *mutable_borrow = 1;
933///
934/// drop(mutable_borrow); // relinquish the mutable borrow on this slot
935///
936/// let borrow = x.borrow();
937/// println!("{}", *borrow);
938/// ```
939///
940/// Integers and other types implementing [`Copy`] are unaffected by `drop`.
941///
942/// ```
943/// # #![allow(dropping_copy_types)]
944/// #[derive(Copy, Clone)]
945/// struct Foo(u8);
946///
947/// let x = 1;
948/// let y = Foo(2);
949/// drop(x); // a copy of `x` is moved and dropped
950/// drop(y); // a copy of `y` is moved and dropped
951///
952/// println!("x: {}, y: {}", x, y.0); // still available
953/// ```
954///
955/// [`RefCell`]: crate::cell::RefCell
956#[inline]
957#[stable(feature = "rust1", since = "1.0.0")]
958#[rustc_diagnostic_item = "mem_drop"]
959pub fn drop<T>(_x: T) {}

984/// Interprets `src` as having type `&Dst`, and then reads `src` without moving
985/// the contained value.
986///
987/// This function will unsafely assume the pointer `src` is valid for [`size_of::<Dst>`][size_of]
988/// bytes by transmuting `&Src` to `&Dst` and then reading the `&Dst` (except that this is done
989/// in a way that is correct even when `&Dst` has stricter alignment requirements than `&Src`).
990/// It will also unsafely create a copy of the contained value instead of moving out of `src`.
991///
992/// It is not a compile-time error if `Src` and `Dst` have different sizes, but it
993/// is highly encouraged to only invoke this function where `Src` and `Dst` have the
994/// same size. This function triggers [undefined behavior][ub] if `Dst` is larger than
995/// `Src`.
996///
997/// [ub]: ../../reference/behavior-considered-undefined.html
998///
999/// # Examples
1000///
1001/// ```
1002/// use std::mem;
1003///
1004/// #[repr(packed)]
1005/// struct Foo {
1006///     bar: u8,
1007/// }
1008///
1009/// let foo_array = [10u8];
1010///
1011/// unsafe {
1012///     // Copy the data from 'foo_array' and treat it as a 'Foo'
1013///     let mut foo_struct: Foo = mem::transmute_copy(&foo_array);
1014///     assert_eq!(foo_struct.bar, 10);
1015///
1016///     // Modify the copied data
1017///     foo_struct.bar = 20;
1018///     assert_eq!(foo_struct.bar, 20);
1019/// }
1020///
1021/// // The contents of 'foo_array' should not have changed
1022/// assert_eq!(foo_array, [10]);
1023/// ```
1024#[inline]
1025#[must_use]
1026#[track_caller]
1027#[stable(feature = "rust1", since = "1.0.0")]
1028#[rustc_const_stable(feature = "const_transmute_copy", since = "1.74.0")]
1029#[cfg(not(feature = "ferrocene_certified"))]
1030pub const unsafe fn transmute_copy<Src, Dst>(src: &Src) -> Dst {
1031    assert!(
1032        size_of::<Src>() >= size_of::<Dst>(),
1033        "cannot transmute_copy if Dst is larger than Src"
1034    );
1035
1036    // If Dst has a higher alignment requirement, src might not be suitably aligned.
1037    if align_of::<Dst>() > align_of::<Src>() {
1038        // SAFETY: `src` is a reference which is guaranteed to be valid for reads.
1039        // The caller must guarantee that the actual transmutation is safe.
1040        unsafe { ptr::read_unaligned(src as *const Src as *const Dst) }
1041    } else {
1042        // SAFETY: `src` is a reference which is guaranteed to be valid for reads.
1043        // We just checked that `src as *const Dst` was properly aligned.
1044        // The caller must guarantee that the actual transmutation is safe.
1045        unsafe { ptr::read(src as *const Src as *const Dst) }
1046    }
1047}

1098/// Returns a value uniquely identifying the enum variant in `v`.
1099///
1100/// If `T` is not an enum, calling this function will not result in undefined behavior, but the
1101/// return value is unspecified.
1102///
1103/// # Stability
1104///
1105/// The discriminant of an enum variant may change if the enum definition changes. A discriminant
1106/// of some variant will not change between compilations with the same compiler. See the [Reference]
1107/// for more information.
1108///
1109/// [Reference]: ../../reference/items/enumerations.html#custom-discriminant-values-for-fieldless-enumerations
1110///
1111/// The value of a [`Discriminant<T>`] is independent of any *free lifetimes* in `T`. As such,
1112/// reading or writing a `Discriminant<Foo<'a>>` as a `Discriminant<Foo<'b>>` (whether via
1113/// [`transmute`] or otherwise) is always sound. Note that this is **not** true for other kinds
1114/// of generic parameters and for higher-ranked lifetimes; `Discriminant<Foo<A>>` and
1115/// `Discriminant<Foo<B>>` as well as `Discriminant<Bar<dyn for<'a> Trait<'a>>>` and
1116/// `Discriminant<Bar<dyn Trait<'static>>>` may be incompatible.
1117///
1118/// # Examples
1119///
1120/// This can be used to compare enums that carry data, while disregarding
1121/// the actual data:
1122///
1123/// ```
1124/// use std::mem;
1125///
1126/// enum Foo { A(&'static str), B(i32), C(i32) }
1127///
1128/// assert_eq!(mem::discriminant(&Foo::A("bar")), mem::discriminant(&Foo::A("baz")));
1129/// assert_eq!(mem::discriminant(&Foo::B(1)), mem::discriminant(&Foo::B(2)));
1130/// assert_ne!(mem::discriminant(&Foo::B(3)), mem::discriminant(&Foo::C(3)));
1131/// ```
1132///
1133/// ## Accessing the numeric value of the discriminant
1134///
1135/// Note that it is *undefined behavior* to [`transmute`] from [`Discriminant`] to a primitive!
1136///
1137/// If an enum has only unit variants, then the numeric value of the discriminant can be accessed
1138/// with an [`as`] cast:
1139///
1140/// ```
1141/// enum Enum {
1142///     Foo,
1143///     Bar,
1144///     Baz,
1145/// }
1146///
1147/// assert_eq!(0, Enum::Foo as isize);
1148/// assert_eq!(1, Enum::Bar as isize);
1149/// assert_eq!(2, Enum::Baz as isize);
1150/// ```
1151///
1152/// If an enum has opted-in to having a [primitive representation] for its discriminant,
1153/// then it's possible to use pointers to read the memory location storing the discriminant.
1154/// That **cannot** be done for enums using the [default representation], however, as it's
1155/// undefined what layout the discriminant has and where it's stored â€” it might not even be
1156/// stored at all!
1157///
1158/// [`as`]: ../../std/keyword.as.html
1159/// [primitive representation]: ../../reference/type-layout.html#primitive-representations
1160/// [default representation]: ../../reference/type-layout.html#the-default-representation
1161/// ```
1162/// #[repr(u8)]
1163/// enum Enum {
1164///     Unit,
1165///     Tuple(bool),
1166///     Struct { a: bool },
1167/// }
1168///
1169/// impl Enum {
1170///     fn discriminant(&self) -> u8 {
1171///         // SAFETY: Because `Self` is marked `repr(u8)`, its layout is a `repr(C)` `union`
1172///         // between `repr(C)` structs, each of which has the `u8` discriminant as its first
1173///         // field, so we can read the discriminant without offsetting the pointer.
1174///         unsafe { *<*const _>::from(self).cast::<u8>() }
1175///     }
1176/// }
1177///
1178/// let unit_like = Enum::Unit;
1179/// let tuple_like = Enum::Tuple(true);
1180/// let struct_like = Enum::Struct { a: false };
1181/// assert_eq!(0, unit_like.discriminant());
1182/// assert_eq!(1, tuple_like.discriminant());
1183/// assert_eq!(2, struct_like.discriminant());
1184///
1185/// // âš ï¸ This is undefined behavior. Don't do this. âš ï¸
1186/// // assert_eq!(0, unsafe { std::mem::transmute::<_, u8>(std::mem::discriminant(&unit_like)) });
1187/// ```
1188#[stable(feature = "discriminant_value", since = "1.21.0")]
1189#[rustc_const_stable(feature = "const_discriminant", since = "1.75.0")]

1197/// Returns the number of variants in the enum type `T`.
1198///
1199/// If `T` is not an enum, calling this function will not result in undefined behavior, but the
1200/// return value is unspecified. Equally, if `T` is an enum with more variants than `usize::MAX`
1201/// the return value is unspecified. Uninhabited variants will be counted.
1202///
1203/// Note that an enum may be expanded with additional variants in the future
1204/// as a non-breaking change, for example if it is marked `#[non_exhaustive]`,
1205/// which will change the result of this function.
1206///
1207/// # Examples
1208///
1209/// ```
1210/// # #![feature(never_type)]
1211/// # #![feature(variant_count)]
1212///
1213/// use std::mem;
1214///
1215/// enum Void {}
1216/// enum Foo { A(&'static str), B(i32), C(i32) }
1217///
1218/// assert_eq!(mem::variant_count::<Void>(), 0);
1219/// assert_eq!(mem::variant_count::<Foo>(), 3);
1220///
1221/// assert_eq!(mem::variant_count::<Option<!>>(), 2);
1222/// assert_eq!(mem::variant_count::<Result<!, !>>(), 2);
1223/// ```
1224#[inline(always)]
1225#[must_use]
1226#[unstable(feature = "variant_count", issue = "73662")]
1227#[rustc_const_unstable(feature = "variant_count", issue = "73662")]
1228#[rustc_diagnostic_item = "mem_variant_count"]
1229#[cfg(not(feature = "ferrocene_certified"))]
1230pub const fn variant_count<T>() -> usize {
1231    intrinsics::variant_count::<T>()
1232}

1234/// Provides associated constants for various useful properties of types,
1235/// to give them a canonical form in our code and make them easier to read.
1236///
1237/// This is here only to simplify all the ZST checks we need in the library.
1238/// It's not on a stabilization track right now.
1239#[doc(hidden)]
1240#[unstable(feature = "sized_type_properties", issue = "none")]
1241pub trait SizedTypeProperties: Sized {
1242    /// `true` if this type requires no storage.
1243    /// `false` if its [size](size_of) is greater than zero.
1244    ///
1245    /// # Examples
1246    ///
1247    /// ```
1248    /// #![feature(sized_type_properties)]
1249    /// use core::mem::SizedTypeProperties;
1250    ///
1251    /// fn do_something_with<T>() {
1252    ///     if T::IS_ZST {
1253    ///         // ... special approach ...
1254    ///     } else {
1255    ///         // ... the normal thing ...
1256    ///     }
1257    /// }
1258    ///
1259    /// struct MyUnit;
1260    /// assert!(MyUnit::IS_ZST);
1261    ///
1262    /// // For negative checks, consider using UFCS to emphasize the negation
1263    /// assert!(!<i32>::IS_ZST);
1264    /// // As it can sometimes hide in the type otherwise
1265    /// assert!(!String::IS_ZST);
1266    /// ```
1267    #[doc(hidden)]
1268    #[unstable(feature = "sized_type_properties", issue = "none")]
1269    const IS_ZST: bool = size_of::<Self>() == 0;
1270
1271    #[doc(hidden)]
1272    #[unstable(feature = "sized_type_properties", issue = "none")]
1273    const LAYOUT: Layout = Layout::new::<Self>();
1274
1275    /// The largest safe length for a `[Self]`.
1276    ///
1277    /// Anything larger than this would make `size_of_val` overflow `isize::MAX`,
1278    /// which is never allowed for a single object.
1279    #[doc(hidden)]
1280    #[unstable(feature = "sized_type_properties", issue = "none")]
1281    const MAX_SLICE_LEN: usize = match size_of::<Self>() {
1282        0 => usize::MAX,
1283        n => (isize::MAX as usize) / n,
1284    };
1285}
1286#[doc(hidden)]
1287#[unstable(feature = "sized_type_properties", issue = "none")]
1288impl<T> SizedTypeProperties for T {}

1290/// Expands to the offset in bytes of a field from the beginning of the given type.
1291///
1292/// The type may be a `struct`, `enum`, `union`, or tuple.
1293///
1294/// The field may be a nested field (`field1.field2`), but not an array index.
1295/// The field must be visible to the call site.
1296///
1297/// The offset is returned as a [`usize`].
1298///
1299/// # Offsets of, and in, dynamically sized types
1300///
1301/// The fieldâ€™s type must be [`Sized`], but it may be located in a [dynamically sized] container.
1302/// If the field type is dynamically sized, then you cannot use `offset_of!` (since the field's
1303/// alignment, and therefore its offset, may also be dynamic) and must take the offset from an
1304/// actual pointer to the container instead.
1305///
1306/// ```
1307/// # use core::mem;
1308/// # use core::fmt::Debug;
1309/// #[repr(C)]
1310/// pub struct Struct<T: ?Sized> {
1311///     a: u8,
1312///     b: T,
1313/// }
1314///
1315/// #[derive(Debug)]
1316/// #[repr(C, align(4))]
1317/// struct Align4(u32);
1318///
1319/// assert_eq!(mem::offset_of!(Struct<dyn Debug>, a), 0); // OK â€” Sized field
1320/// assert_eq!(mem::offset_of!(Struct<Align4>, b), 4); // OK â€” not DST
1321///
1322/// // assert_eq!(mem::offset_of!(Struct<dyn Debug>, b), 1);
1323/// // ^^^ error[E0277]: ... cannot be known at compilation time
1324///
1325/// // To obtain the offset of a !Sized field, examine a concrete value
1326/// // instead of using offset_of!.
1327/// let value: Struct<Align4> = Struct { a: 1, b: Align4(2) };
1328/// let ref_unsized: &Struct<dyn Debug> = &value;
1329/// let offset_of_b = unsafe {
1330///     (&raw const ref_unsized.b).byte_offset_from_unsigned(ref_unsized)
1331/// };
1332/// assert_eq!(offset_of_b, 4);
1333/// ```
1334///
1335/// If you need to obtain the offset of a field of a `!Sized` type, then, since the offset may
1336/// depend on the particular value being stored (in particular, `dyn Trait` values have a
1337/// dynamically-determined alignment), you must retrieve the offset from a specific reference
1338/// or pointer, and so you cannot use `offset_of!` to work without one.
1339///
1340/// # Layout is subject to change
1341///
1342/// Note that type layout is, in general, [subject to change and
1343/// platform-specific](https://doc.rust-lang.org/reference/type-layout.html). If
1344/// layout stability is required, consider using an [explicit `repr` attribute].
1345///
1346/// Rust guarantees that the offset of a given field within a given type will not
1347/// change over the lifetime of the program. However, two different compilations of
1348/// the same program may result in different layouts. Also, even within a single
1349/// program execution, no guarantees are made about types which are *similar* but
1350/// not *identical*, e.g.:
1351///
1352/// ```
1353/// struct Wrapper<T, U>(T, U);
1354///
1355/// type A = Wrapper<u8, u8>;
1356/// type B = Wrapper<u8, i8>;
1357///
1358/// // Not necessarily identical even though `u8` and `i8` have the same layout!
1359/// // assert_eq!(mem::offset_of!(A, 1), mem::offset_of!(B, 1));
1360///
1361/// #[repr(transparent)]
1362/// struct U8(u8);
1363///
1364/// type C = Wrapper<u8, U8>;
1365///
1366/// // Not necessarily identical even though `u8` and `U8` have the same layout!
1367/// // assert_eq!(mem::offset_of!(A, 1), mem::offset_of!(C, 1));
1368///
1369/// struct Empty<T>(core::marker::PhantomData<T>);
1370///
1371/// // Not necessarily identical even though `PhantomData` always has the same layout!
1372/// // assert_eq!(mem::offset_of!(Empty<u8>, 0), mem::offset_of!(Empty<i8>, 0));
1373/// ```
1374///
1375/// [explicit `repr` attribute]: https://doc.rust-lang.org/reference/type-layout.html#representations
1376///
1377/// # Unstable features
1378///
1379/// The following unstable features expand the functionality of `offset_of!`:
1380///
1381/// * [`offset_of_enum`] â€” allows `enum` variants to be traversed as if they were fields.
1382/// * [`offset_of_slice`] â€” allows getting the offset of a field of type `[T]`.
1383///
1384/// # Examples
1385///
1386/// ```
1387/// use std::mem;
1388/// #[repr(C)]
1389/// struct FieldStruct {
1390///     first: u8,
1391///     second: u16,
1392///     third: u8
1393/// }
1394///
1395/// assert_eq!(mem::offset_of!(FieldStruct, first), 0);
1396/// assert_eq!(mem::offset_of!(FieldStruct, second), 2);
1397/// assert_eq!(mem::offset_of!(FieldStruct, third), 4);
1398///
1399/// #[repr(C)]
1400/// struct NestedA {
1401///     b: NestedB
1402/// }
1403///
1404/// #[repr(C)]
1405/// struct NestedB(u8);
1406///
1407/// assert_eq!(mem::offset_of!(NestedA, b.0), 0);
1408/// ```
1409///
1410/// [dynamically sized]: https://doc.rust-lang.org/reference/dynamically-sized-types.html
1411/// [`offset_of_enum`]: https://doc.rust-lang.org/nightly/unstable-book/language-features/offset-of-enum.html
1412/// [`offset_of_slice`]: https://doc.rust-lang.org/nightly/unstable-book/language-features/offset-of-slice.html
1413#[stable(feature = "offset_of", since = "1.77.0")]
1414#[allow_internal_unstable(builtin_syntax)]
1415#[cfg(not(feature = "ferrocene_certified"))]
1416pub macro offset_of($Container:ty, $($fields:expr)+ $(,)?) {
1417    // The `{}` is for better error messages
1418    {builtin # offset_of($Container, $($fields)+)}
1419}
