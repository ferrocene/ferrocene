1//! Manually manage memory through raw pointers.
2//!
3//! *[See also the pointer primitive types](pointer).*
4//!
5//! # Safety
6//!
7//! Many functions in this module take raw pointers as arguments and read from or write to them. For
8//! this to be safe, these pointers must be *valid* for the given access. Whether a pointer is valid
9//! depends on the operation it is used for (read or write), and the extent of the memory that is
10//! accessed (i.e., how many bytes are read/written) -- it makes no sense to ask "is this pointer
11//! valid"; one has to ask "is this pointer valid for a given access". Most functions use `*mut T`
12//! and `*const T` to access only a single value, in which case the documentation omits the size and
13//! implicitly assumes it to be `size_of::<T>()` bytes.
14//!
15//! The precise rules for validity are not determined yet. The guarantees that are
16//! provided at this point are very minimal:
17//!
18//! * For memory accesses of [size zero][zst], *every* pointer is valid, including the [null]
19//!   pointer. The following points are only concerned with non-zero-sized accesses.
20//! * A [null] pointer is *never* valid.
21//! * For a pointer to be valid, it is necessary, but not always sufficient, that the pointer be
22//!   *dereferenceable*. The [provenance] of the pointer is used to determine which [allocated
23//!   object] it is derived from; a pointer is dereferenceable if the memory range of the given size
24//!   starting at the pointer is entirely contained within the bounds of that allocated object. Note
25//!   that in Rust, every (stack-allocated) variable is considered a separate allocated object.
26//! * All accesses performed by functions in this module are *non-atomic* in the sense
27//!   of [atomic operations] used to synchronize between threads. This means it is
28//!   undefined behavior to perform two concurrent accesses to the same location from different
29//!   threads unless both accesses only read from memory. Notice that this explicitly
30//!   includes [`read_volatile`] and [`write_volatile`]: Volatile accesses cannot
31//!   be used for inter-thread synchronization.
32//! * The result of casting a reference to a pointer is valid for as long as the
33//!   underlying object is live and no reference (just raw pointers) is used to
34//!   access the same memory. That is, reference and pointer accesses cannot be
35//!   interleaved.
36//!
37//! These axioms, along with careful use of [`offset`] for pointer arithmetic,
38//! are enough to correctly implement many useful things in unsafe code. Stronger guarantees
39//! will be provided eventually, as the [aliasing] rules are being determined. For more
40//! information, see the [book] as well as the section in the reference devoted
41//! to [undefined behavior][ub].
42//!
43//! We say that a pointer is "dangling" if it is not valid for any non-zero-sized accesses. This
44//! means out-of-bounds pointers, pointers to freed memory, null pointers, and pointers created with
45//! [`NonNull::dangling`] are all dangling.
46//!
47//! ## Alignment
48//!
49//! Valid raw pointers as defined above are not necessarily properly aligned (where
50//! "proper" alignment is defined by the pointee type, i.e., `*const T` must be
51//! aligned to `align_of::<T>()`). However, most functions require their
52//! arguments to be properly aligned, and will explicitly state
53//! this requirement in their documentation. Notable exceptions to this are
54//! [`read_unaligned`] and [`write_unaligned`].
55//!
56//! When a function requires proper alignment, it does so even if the access
57//! has size 0, i.e., even if memory is not actually touched. Consider using
58//! [`NonNull::dangling`] in such cases.
59//!
60//! ## Pointer to reference conversion
61//!
62//! When converting a pointer to a reference (e.g. via `&*ptr` or `&mut *ptr`),
63//! there are several rules that must be followed:
64//!
65//! * The pointer must be properly aligned.
66//!
67//! * It must be non-null.
68//!
69//! * It must be "dereferenceable" in the sense defined above.
70//!
71//! * The pointer must point to a [valid value] of type `T`.
72//!
73//! * You must enforce Rust's aliasing rules. The exact aliasing rules are not decided yet, so we
74//!   only give a rough overview here. The rules also depend on whether a mutable or a shared
75//!   reference is being created.
76//!   * When creating a mutable reference, then while this reference exists, the memory it points to
77//!     must not get accessed (read or written) through any other pointer or reference not derived
78//!     from this reference.
79//!   * When creating a shared reference, then while this reference exists, the memory it points to
80//!     must not get mutated (except inside `UnsafeCell`).
81//!
82//! If a pointer follows all of these rules, it is said to be
83//! *convertible to a (mutable or shared) reference*.
84// ^ we use this term instead of saying that the produced reference must
85// be valid, as the validity of a reference is easily confused for the
86// validity of the thing it refers to, and while the two concepts are
87// closely related, they are not identical.
88//!
89//! These rules apply even if the result is unused!
90//! (The part about being initialized is not yet fully decided, but until
91//! it is, the only safe approach is to ensure that they are indeed initialized.)
92//!
93//! An example of the implications of the above rules is that an expression such
94//! as `unsafe { &*(0 as *const u8) }` is Immediate Undefined Behavior.
95//!
96//! [valid value]: ../../reference/behavior-considered-undefined.html#invalid-values
97//!
98//! ## Allocated object
99//!
100//! An *allocated object* is a subset of program memory which is addressable
101//! from Rust, and within which pointer arithmetic is possible. Examples of
102//! allocated objects include heap allocations, stack-allocated variables,
103//! statics, and consts. The safety preconditions of some Rust operations -
104//! such as `offset` and field projections (`expr.field`) - are defined in
105//! terms of the allocated objects on which they operate.
106//!
107//! An allocated object has a base address, a size, and a set of memory
108//! addresses. It is possible for an allocated object to have zero size, but
109//! such an allocated object will still have a base address. The base address
110//! of an allocated object is not necessarily unique. While it is currently the
111//! case that an allocated object always has a set of memory addresses which is
112//! fully contiguous (i.e., has no "holes"), there is no guarantee that this
113//! will not change in the future.
114//!
115//! For any allocated object with `base` address, `size`, and a set of
116//! `addresses`, the following are guaranteed:
117//! - For all addresses `a` in `addresses`, `a` is in the range `base .. (base +
118//!   size)` (note that this requires `a < base + size`, not `a <= base + size`)
119//! - `base` is not equal to [`null()`] (i.e., the address with the numerical
120//!   value 0)
121//! - `base + size <= usize::MAX`
122//! - `size <= isize::MAX`
123//!
124//! As a consequence of these guarantees, given any address `a` within the set
125//! of addresses of an allocated object:
126//! - It is guaranteed that `a - base` does not overflow `isize`
127//! - It is guaranteed that `a - base` is non-negative
128//! - It is guaranteed that, given `o = a - base` (i.e., the offset of `a` within
129//!   the allocated object), `base + o` will not wrap around the address space (in
130//!   other words, will not overflow `usize`)
131//!
132//! [`null()`]: null

134//! # Provenance
135//!
136//! Pointers are not *simply* an "integer" or "address". For instance, it's uncontroversial
137//! to say that a Use After Free is clearly Undefined Behavior, even if you "get lucky"
138//! and the freed memory gets reallocated before your read/write (in fact this is the
139//! worst-case scenario, UAFs would be much less concerning if this didn't happen!).
140//! As another example, consider that [`wrapping_offset`] is documented to "remember"
141//! the allocated object that the original pointer points to, even if it is offset far
142//! outside the memory range occupied by that allocated object.
143//! To rationalize claims like this, pointers need to somehow be *more* than just their addresses:
144//! they must have **provenance**.
145//!
146//! A pointer value in Rust semantically contains the following information:
147//!
148//! * The **address** it points to, which can be represented by a `usize`.
149//! * The **provenance** it has, defining the memory it has permission to access. Provenance can be
150//!   absent, in which case the pointer does not have permission to access any memory.
151//!
152//! The exact structure of provenance is not yet specified, but the permission defined by a
153//! pointer's provenance have a *spatial* component, a *temporal* component, and a *mutability*
154//! component:
155//!
156//! * Spatial: The set of memory addresses that the pointer is allowed to access.
157//! * Temporal: The timespan during which the pointer is allowed to access those memory addresses.
158//! * Mutability: Whether the pointer may only access the memory for reads, or also access it for
159//!   writes. Note that this can interact with the other components, e.g. a pointer might permit
160//!   mutation only for a subset of addresses, or only for a subset of its maximal timespan.
161//!
162//! When an [allocated object] is created, it has a unique Original Pointer. For alloc
163//! APIs this is literally the pointer the call returns, and for local variables and statics,
164//! this is the name of the variable/static. (This is mildly overloading the term "pointer"
165//! for the sake of brevity/exposition.)
166//!
167//! The Original Pointer for an allocated object has provenance that constrains the *spatial*
168//! permissions of this pointer to the memory range of the allocation, and the *temporal*
169//! permissions to the lifetime of the allocation. Provenance is implicitly inherited by all
170//! pointers transitively derived from the Original Pointer through operations like [`offset`],
171//! borrowing, and pointer casts. Some operations may *shrink* the permissions of the derived
172//! provenance, limiting how much memory it can access or how long it's valid for (i.e. borrowing a
173//! subfield and subslicing can shrink the spatial component of provenance, and all borrowing can
174//! shrink the temporal component of provenance). However, no operation can ever *grow* the
175//! permissions of the derived provenance: even if you "know" there is a larger allocation, you
176//! can't derive a pointer with a larger provenance. Similarly, you cannot "recombine" two
177//! contiguous provenances back into one (i.e. with a `fn merge(&[T], &[T]) -> &[T]`).
178//!
179//! A reference to a place always has provenance over at least the memory that place occupies.
180//! A reference to a slice always has provenance over at least the range that slice describes.
181//! Whether and when exactly the provenance of a reference gets "shrunk" to *exactly* fit
182//! the memory it points to is not yet determined.
183//!
184//! A *shared* reference only ever has provenance that permits reading from memory,
185//! and never permits writes, except inside [`UnsafeCell`].
186//!
187//! Provenance can affect whether a program has undefined behavior:
188//!
189//! * It is undefined behavior to access memory through a pointer that does not have provenance over
190//!   that memory. Note that a pointer "at the end" of its provenance is not actually outside its
191//!   provenance, it just has 0 bytes it can load/store. Zero-sized accesses do not require any
192//!   provenance since they access an empty range of memory.
193//!
194//! * It is undefined behavior to [`offset`] a pointer across a memory range that is not contained
195//!   in the allocated object it is derived from, or to [`offset_from`] two pointers not derived
196//!   from the same allocated object. Provenance is used to say what exactly "derived from" even
197//!   means: the lineage of a pointer is traced back to the Original Pointer it descends from, and
198//!   that identifies the relevant allocated object. In particular, it's always UB to offset a
199//!   pointer derived from something that is now deallocated, except if the offset is 0.
200//!
201//! But it *is* still sound to:
202//!
203//! * Create a pointer without provenance from just an address (see [`without_provenance`]). Such a
204//!   pointer cannot be used for memory accesses (except for zero-sized accesses). This can still be
205//!   useful for sentinel values like `null` *or* to represent a tagged pointer that will never be
206//!   dereferenceable. In general, it is always sound for an integer to pretend to be a pointer "for
207//!   fun" as long as you don't use operations on it which require it to be valid (non-zero-sized
208//!   offset, read, write, etc).
209//!
210//! * Forge an allocation of size zero at any sufficiently aligned non-null address.
211//!   i.e. the usual "ZSTs are fake, do what you want" rules apply.
212//!
213//! * [`wrapping_offset`] a pointer outside its provenance. This includes pointers
214//!   which have "no" provenance. In particular, this makes it sound to do pointer tagging tricks.
215//!
216//! * Compare arbitrary pointers by address. Pointer comparison ignores provenance and addresses
217//!   *are* just integers, so there is always a coherent answer, even if the pointers are dangling
218//!   or from different provenances. Note that if you get "lucky" and notice that a pointer at the
219//!   end of one allocated object is the "same" address as the start of another allocated object,
220//!   anything you do with that fact is *probably* going to be gibberish. The scope of that
221//!   gibberish is kept under control by the fact that the two pointers *still* aren't allowed to
222//!   access the other's allocation (bytes), because they still have different provenance.
223//!
224//! Note that the full definition of provenance in Rust is not decided yet, as this interacts
225//! with the as-yet undecided [aliasing] rules.
226//!
227//! ## Pointers Vs Integers
228//!
229//! From this discussion, it becomes very clear that a `usize` *cannot* accurately represent a pointer,
230//! and converting from a pointer to a `usize` is generally an operation which *only* extracts the
231//! address. Converting this address back into pointer requires somehow answering the question:
232//! which provenance should the resulting pointer have?
233//!
234//! Rust provides two ways of dealing with this situation: *Strict Provenance* and *Exposed Provenance*.
235//!
236//! Note that a pointer *can* represent a `usize` (via [`without_provenance`]), so the right type to
237//! use in situations where a value is "sometimes a pointer and sometimes a bare `usize`" is a
238//! pointer type.
239//!
240//! ## Strict Provenance
241//!
242//! "Strict Provenance" refers to a set of APIs designed to make working with provenance more
243//! explicit. They are intended as substitutes for casting a pointer to an integer and back.
244//!
245//! Entirely avoiding integer-to-pointer casts successfully side-steps the inherent ambiguity of
246//! that operation. This benefits compiler optimizations, and it is pretty much a requirement for
247//! using tools like [Miri] and architectures like [CHERI] that aim to detect and diagnose pointer
248//! misuse.
249//!
250//! The key insight to making programming without integer-to-pointer casts *at all* viable is the
251//! [`with_addr`] method:
252//!
253//! ```text
254//!     /// Creates a new pointer with the given address.
255//!     ///
256//!     /// This performs the same operation as an `addr as ptr` cast, but copies
257//!     /// the *provenance* of `self` to the new pointer.
258//!     /// This allows us to dynamically preserve and propagate this important
259//!     /// information in a way that is otherwise impossible with a unary cast.
260//!     ///
261//!     /// This is equivalent to using `wrapping_offset` to offset `self` to the
262//!     /// given address, and therefore has all the same capabilities and restrictions.
263//!     pub fn with_addr(self, addr: usize) -> Self;
264//! ```
265//!
266//! So you're still able to drop down to the address representation and do whatever
267//! clever bit tricks you want *as long as* you're able to keep around a pointer
268//! into the allocation you care about that can "reconstitute" the provenance.
269//! Usually this is very easy, because you only are taking a pointer, messing with the address,
270//! and then immediately converting back to a pointer. To make this use case more ergonomic,
271//! we provide the [`map_addr`] method.
272//!
273//! To help make it clear that code is "following" Strict Provenance semantics, we also provide an
274//! [`addr`] method which promises that the returned address is not part of a
275//! pointer-integer-pointer roundtrip. In the future we may provide a lint for pointer<->integer
276//! casts to help you audit if your code conforms to strict provenance.
277//!
278//! ### Using Strict Provenance
279//!
280//! Most code needs no changes to conform to strict provenance, as the only really concerning
281//! operation is casts from `usize` to a pointer. For code which *does* cast a `usize` to a pointer,
282//! the scope of the change depends on exactly what you're doing.
283//!
284//! In general, you just need to make sure that if you want to convert a `usize` address to a
285//! pointer and then use that pointer to read/write memory, you need to keep around a pointer
286//! that has sufficient provenance to perform that read/write itself. In this way all of your
287//! casts from an address to a pointer are essentially just applying offsets/indexing.
288//!
289//! This is generally trivial to do for simple cases like tagged pointers *as long as you
290//! represent the tagged pointer as an actual pointer and not a `usize`*. For instance:
291//!
292//! ```
293//! unsafe {
294//!     // A flag we want to pack into our pointer
295//!     static HAS_DATA: usize = 0x1;
296//!     static FLAG_MASK: usize = !HAS_DATA;
297//!
298//!     // Our value, which must have enough alignment to have spare least-significant-bits.
299//!     let my_precious_data: u32 = 17;
300//!     assert!(align_of::<u32>() > 1);
301//!
302//!     // Create a tagged pointer
303//!     let ptr = &my_precious_data as *const u32;
304//!     let tagged = ptr.map_addr(|addr| addr | HAS_DATA);
305//!
306//!     // Check the flag:
307//!     if tagged.addr() & HAS_DATA != 0 {
308//!         // Untag and read the pointer
309//!         let data = *tagged.map_addr(|addr| addr & FLAG_MASK);
310//!         assert_eq!(data, 17);
311//!     } else {
312//!         unreachable!()
313//!     }
314//! }
315//! ```
316//!
317//! (Yes, if you've been using [`AtomicUsize`] for pointers in concurrent datastructures, you should
318//! be using [`AtomicPtr`] instead. If that messes up the way you atomically manipulate pointers,
319//! we would like to know why, and what needs to be done to fix it.)
320//!
321//! Situations where a valid pointer *must* be created from just an address, such as baremetal code
322//! accessing a memory-mapped interface at a fixed address, cannot currently be handled with strict
323//! provenance APIs and should use [exposed provenance](#exposed-provenance).
324//!
325//! ## Exposed Provenance
326//!
327//! As discussed above, integer-to-pointer casts are not possible with Strict Provenance APIs.
328//! This is by design: the goal of Strict Provenance is to provide a clear specification that we are
329//! confident can be formalized unambiguously and can be subject to precise formal reasoning.
330//! Integer-to-pointer casts do not (currently) have such a clear specification.
331//!
332//! However, there exist situations where integer-to-pointer casts cannot be avoided, or
333//! where avoiding them would require major refactoring. Legacy platform APIs also regularly assume
334//! that `usize` can capture all the information that makes up a pointer.
335//! Bare-metal platforms can also require the synthesis of a pointer "out of thin air" without
336//! anywhere to obtain proper provenance from.
337//!
338//! Rust's model for dealing with integer-to-pointer casts is called *Exposed Provenance*. However,
339//! the semantics of Exposed Provenance are on much less solid footing than Strict Provenance, and
340//! at this point it is not yet clear whether a satisfying unambiguous semantics can be defined for
341//! Exposed Provenance. (If that sounds bad, be reassured that other popular languages that provide
342//! integer-to-pointer casts are not faring any better.) Furthermore, Exposed Provenance will not
343//! work (well) with tools like [Miri] and [CHERI].
344//!
345//! Exposed Provenance is provided by the [`expose_provenance`] and [`with_exposed_provenance`] methods,
346//! which are equivalent to `as` casts between pointers and integers.
347//! - [`expose_provenance`] is a lot like [`addr`], but additionally adds the provenance of the
348//!   pointer to a global list of 'exposed' provenances. (This list is purely conceptual, it exists
349//!   for the purpose of specifying Rust but is not materialized in actual executions, except in
350//!   tools like [Miri].)
351//!   Memory which is outside the control of the Rust abstract machine (MMIO registers, for example)
352//!   is always considered to be exposed, so long as this memory is disjoint from memory that will
353//!   be used by the abstract machine such as the stack, heap, and statics.
354//! - [`with_exposed_provenance`] can be used to construct a pointer with one of these previously
355//!   'exposed' provenances. [`with_exposed_provenance`] takes only `addr: usize` as arguments, so
356//!   unlike in [`with_addr`] there is no indication of what the correct provenance for the returned
357//!   pointer is -- and that is exactly what makes integer-to-pointer casts so tricky to rigorously
358//!   specify! The compiler will do its best to pick the right provenance for you, but currently we
359//!   cannot provide any guarantees about which provenance the resulting pointer will have. Only one
360//!   thing is clear: if there is *no* previously 'exposed' provenance that justifies the way the
361//!   returned pointer will be used, the program has undefined behavior.
362//!
363//! If at all possible, we encourage code to be ported to [Strict Provenance] APIs, thus avoiding
364//! the need for Exposed Provenance. Maximizing the amount of such code is a major win for avoiding
365//! specification complexity and to facilitate adoption of tools like [CHERI] and [Miri] that can be
366//! a big help in increasing the confidence in (unsafe) Rust code. However, we acknowledge that this
367//! is not always possible, and offer Exposed Provenance as a way to explicit "opt out" of the
368//! well-defined semantics of Strict Provenance, and "opt in" to the unclear semantics of
369//! integer-to-pointer casts.

455///
456/// This is almost the same as calling [`ptr::read`] and discarding
457/// the result, but has the following advantages:
458// FIXME: say something more useful than "almost the same"?
459// There are open questions here: `read` requires the value to be fully valid, e.g. if `T` is a
460// `bool` it must be 0 or 1, if it is a reference then it must be dereferenceable. `drop_in_place`
461// only requires that `*to_drop` be "valid for dropping" and we have not defined what that means. In
462// Miri it currently (May 2024) requires nothing at all for types without drop glue.
463///
464/// * It is *required* to use `drop_in_place` to drop unsized types like
465///   trait objects, because they can't be read out onto the stack and
466///   dropped normally.
467///
468/// * It is friendlier to the optimizer to do this over [`ptr::read`] when
469///   dropping manually allocated memory (e.g., in the implementations of
470///   `Box`/`Rc`/`Vec`), as the compiler doesn't need to prove that it's
471///   sound to elide the copy.
472///
473/// * It can be used to drop [pinned] data when `T` is not `repr(packed)`
474///   (pinned data must not be moved before it is dropped).
475///
476/// Unaligned values cannot be dropped in place, they must be copied to an aligned
477/// location first using [`ptr::read_unaligned`]. For packed structs, this move is
478/// done automatically by the compiler. This means the fields of packed structs
479/// are not dropped in-place.
480///
481/// [`ptr::read`]: self::read
482/// [`ptr::read_unaligned`]: self::read_unaligned
483/// [pinned]: crate::pin
484///
485/// # Safety
486///
487/// Behavior is undefined if any of the following conditions are violated:
488///
489/// * `to_drop` must be [valid] for both reads and writes.
490///
491/// * `to_drop` must be properly aligned, even if `T` has size 0.
492///
493/// * `to_drop` must be nonnull, even if `T` has size 0.
494///

496///   it must uphold additional invariants. These invariants depend on the type
497///   of the value being dropped. For instance, when dropping a Box, the box's
498///   pointer to the heap must be valid.
499///

501///   `to_drop` is through the `&mut self` references supplied to the

505/// calling `drop_in_place` can cause undefined behavior. Note that `*to_drop =

776/// will end up dangling.
777///
778/// The caller must also ensure that the memory the pointer (non-transitively) points to is never
779/// written to (except inside an `UnsafeCell`) using this pointer or any pointer derived from it. If
780/// you need to mutate the pointee, use [`from_mut`]. Specifically, to turn a mutable reference `m:

785///
786/// Note that this has subtle interactions with the rules for lifetime extension of temporaries in

789/// # type T = i32;
790/// # fn foo() -> T { 42 }
791/// // The temporary holding the return value of `foo` has its lifetime extended,

965/// Swaps the values at two mutable locations of the same type, without
966/// deinitializing either.
967///
968/// But for the following exceptions, this function is semantically
969/// equivalent to [`mem::swap`]:
970///
971/// * It operates on raw pointers instead of references. When references are
972///   available, [`mem::swap`] should be preferred.
973///
974/// * The two pointed-to values may overlap. If the values do overlap, then the
975///   overlapping region of memory from `x` will be used. This is demonstrated
976///   in the second example below.
977///
978/// * The operation is "untyped" in the sense that data may be uninitialized or otherwise violate
979///   the requirements of `T`. The initialization state is preserved exactly.
980///
981/// # Safety
982///
983/// Behavior is undefined if any of the following conditions are violated:
984///
985/// * Both `x` and `y` must be [valid] for both reads and writes. They must remain valid even when the
986///   other pointer is written. (This means if the memory ranges overlap, the two pointers must not
987///   be subject to aliasing restrictions relative to each other.)
988///
989/// * Both `x` and `y` must be properly aligned.
990///
991/// Note that even if `T` has size `0`, the pointers must be properly aligned.
992///
993/// [valid]: self#safety
994///
995/// # Examples
996///
997/// Swapping two non-overlapping regions:
998///
999/// ```
1000/// use std::ptr;
1001///
1002/// let mut array = [0, 1, 2, 3];
1003///
1004/// let (x, y) = array.split_at_mut(2);
1005/// let x = x.as_mut_ptr().cast::<[u32; 2]>(); // this is `array[0..2]`
1006/// let y = y.as_mut_ptr().cast::<[u32; 2]>(); // this is `array[2..4]`
1007///
1008/// unsafe {
1009///     ptr::swap(x, y);
1010///     assert_eq!([2, 3, 0, 1], array);
1011/// }
1012/// ```
1013///
1014/// Swapping two overlapping regions:
1015///
1016/// ```
1017/// use std::ptr;
1018///
1019/// let mut array: [i32; 4] = [0, 1, 2, 3];
1020///
1021/// let array_ptr: *mut i32 = array.as_mut_ptr();
1022///
1023/// let x = array_ptr as *mut [i32; 3]; // this is `array[0..3]`
1024/// let y = unsafe { array_ptr.add(1) } as *mut [i32; 3]; // this is `array[1..4]`
1025///
1026/// unsafe {
1027///     ptr::swap(x, y);
1028///     // The indices `1..3` of the slice overlap between `x` and `y`.
1029///     // Reasonable results would be for to them be `[2, 3]`, so that indices `0..3` are
1030///     // `[1, 2, 3]` (matching `y` before the `swap`); or for them to be `[0, 1]`
1031///     // so that indices `1..4` are `[0, 1, 2]` (matching `x` before the `swap`).
1032///     // This implementation is defined to make the latter choice.
1033///     assert_eq!([1, 0, 1, 2], array);
1034/// }
1035/// ```
1036#[inline]
1037#[stable(feature = "rust1", since = "1.0.0")]
1038#[rustc_const_stable(feature = "const_swap", since = "1.85.0")]
1039#[rustc_diagnostic_item = "ptr_swap"]
1040#[cfg(not(feature = "ferrocene_certified"))]
1041pub const unsafe fn swap<T>(x: *mut T, y: *mut T) {
1042    // Give ourselves some scratch space to work with.
1043    // We do not have to worry about drops: `MaybeUninit` does nothing when dropped.
1044    let mut tmp = MaybeUninit::<T>::uninit();
1045
1046    // Perform the swap
1047    // SAFETY: the caller must guarantee that `x` and `y` are
1048    // valid for writes and properly aligned. `tmp` cannot be
1049    // overlapping either `x` or `y` because `tmp` was just allocated
1050    // on the stack as a separate allocated object.
1051    unsafe {
1052        copy_nonoverlapping(x, tmp.as_mut_ptr(), 1);
1053        copy(y, x, 1); // `x` and `y` may overlap
1054        copy_nonoverlapping(tmp.as_ptr(), y, 1);
1055    }
1056}

1058/// Swaps `count * size_of::<T>()` bytes between the two regions of memory
1059/// beginning at `x` and `y`. The two regions must *not* overlap.
1060///
1061/// The operation is "untyped" in the sense that data may be uninitialized or otherwise violate the
1062/// requirements of `T`. The initialization state is preserved exactly.
1063///
1064/// # Safety
1065///
1066/// Behavior is undefined if any of the following conditions are violated:
1067///
1068/// * Both `x` and `y` must be [valid] for both reads and writes of `count *
1069///   size_of::<T>()` bytes.
1070///
1071/// * Both `x` and `y` must be properly aligned.
1072///
1073/// * The region of memory beginning at `x` with a size of `count *
1074///   size_of::<T>()` bytes must *not* overlap with the region of memory
1075///   beginning at `y` with the same size.
1076///
1077/// Note that even if the effectively copied size (`count * size_of::<T>()`) is `0`,
1078/// the pointers must be properly aligned.
1079///
1080/// [valid]: self#safety
1081///
1082/// # Examples
1083///
1084/// Basic usage:
1085///
1086/// ```
1087/// use std::ptr;
1088///
1089/// let mut x = [1, 2, 3, 4];
1090/// let mut y = [7, 8, 9];
1091///
1092/// unsafe {
1093///     ptr::swap_nonoverlapping(x.as_mut_ptr(), y.as_mut_ptr(), 2);
1094/// }
1095///
1096/// assert_eq!(x, [7, 8, 3, 4]);
1097/// assert_eq!(y, [1, 2, 9]);
1098/// ```
1099///
1100/// # Const evaluation limitations
1101///
1102/// If this function is invoked during const-evaluation, the current implementation has a small (and
1103/// rarely relevant) limitation: if `count` is at least 2 and the data pointed to by `x` or `y`
1104/// contains a pointer that crosses the boundary of two `T`-sized chunks of memory, the function may
1105/// fail to evaluate (similar to a panic during const-evaluation). This behavior may change in the
1106/// future.
1107///
1108/// The limitation is illustrated by the following example:
1109///
1110/// ```
1111/// use std::mem::size_of;
1112/// use std::ptr;
1113///
1114/// const { unsafe {
1115///     const PTR_SIZE: usize = size_of::<*const i32>();
1116///     let mut data1 = [0u8; PTR_SIZE];
1117///     let mut data2 = [0u8; PTR_SIZE];
1118///     // Store a pointer in `data1`.
1119///     data1.as_mut_ptr().cast::<*const i32>().write_unaligned(&42);
1120///     // Swap the contents of `data1` and `data2` by swapping `PTR_SIZE` many `u8`-sized chunks.
1121///     // This call will fail, because the pointer in `data1` crosses the boundary
1122///     // between several of the 1-byte chunks that are being swapped here.
1123///     //ptr::swap_nonoverlapping(data1.as_mut_ptr(), data2.as_mut_ptr(), PTR_SIZE);
1124///     // Swap the contents of `data1` and `data2` by swapping a single chunk of size
1125///     // `[u8; PTR_SIZE]`. That works, as there is no pointer crossing the boundary between
1126///     // two chunks.
1127///     ptr::swap_nonoverlapping(&mut data1, &mut data2, 1);
1128///     // Read the pointer from `data2` and dereference it.
1129///     let ptr = data2.as_ptr().cast::<*const i32>().read_unaligned();
1130///     assert!(*ptr == 42);
1131/// } }
1132/// ```
1133#[inline]
1134#[stable(feature = "swap_nonoverlapping", since = "1.27.0")]
1135#[rustc_const_stable(feature = "const_swap_nonoverlapping", since = "1.88.0")]
1136#[rustc_diagnostic_item = "ptr_swap_nonoverlapping"]
1137#[rustc_allow_const_fn_unstable(const_eval_select)] // both implementations behave the same
1138#[cfg(not(feature = "ferrocene_certified"))]
1139pub const unsafe fn swap_nonoverlapping<T>(x: *mut T, y: *mut T, count: usize) {
1140    ub_checks::assert_unsafe_precondition!(
1141        check_library_ub,
1142        "ptr::swap_nonoverlapping requires that both pointer arguments are aligned and non-null \
1143        and the specified memory ranges do not overlap",
1144        (
1145            x: *mut () = x as *mut (),
1146            y: *mut () = y as *mut (),
1147            size: usize = size_of::<T>(),
1148            align: usize = align_of::<T>(),
1149            count: usize = count,
1150        ) => {
1151            let zero_size = size == 0 || count == 0;
1152            ub_checks::maybe_is_aligned_and_not_null(x, align, zero_size)
1153                && ub_checks::maybe_is_aligned_and_not_null(y, align, zero_size)
1154                && ub_checks::maybe_is_nonoverlapping(x, y, size, count)
1155        }
1156    );
1157
1158    const_eval_select!(
1159        @capture[T] { x: *mut T, y: *mut T, count: usize }:
1160        if const {
1161            // At compile-time we want to always copy this in chunks of `T`, to ensure that if there
1162            // are pointers inside `T` we will copy them in one go rather than trying to copy a part
1163            // of a pointer (which would not work).
1164            // SAFETY: Same preconditions as this function
1165            unsafe { swap_nonoverlapping_const(x, y, count) }
1166        } else {
1167            // Going though a slice here helps codegen know the size fits in `isize`
1168            let slice = slice_from_raw_parts_mut(x, count);
1169            // SAFETY: This is all readable from the pointer, meaning it's one
1170            // allocated object, and thus cannot be more than isize::MAX bytes.
1171            let bytes = unsafe { mem::size_of_val_raw::<[T]>(slice) };
1172            if let Some(bytes) = NonZero::new(bytes) {
1173                // SAFETY: These are the same ranges, just expressed in a different
1174                // type, so they're still non-overlapping.
1175                unsafe { swap_nonoverlapping_bytes(x.cast(), y.cast(), bytes) };
1176            }
1177        }
1178    )
1179}

1181/// Same behavior and safety conditions as [`swap_nonoverlapping`]
1182#[inline]
1183#[cfg(not(feature = "ferrocene_certified"))]
1184const unsafe fn swap_nonoverlapping_const<T>(x: *mut T, y: *mut T, count: usize) {
1185    let mut i = 0;
1186    while i < count {
1187        // SAFETY: By precondition, `i` is in-bounds because it's below `n`
1188        let x = unsafe { x.add(i) };
1189        // SAFETY: By precondition, `i` is in-bounds because it's below `n`
1190        // and it's distinct from `x` since the ranges are non-overlapping
1191        let y = unsafe { y.add(i) };
1192
1193        // SAFETY: we're only ever given pointers that are valid to read/write,
1194        // including being aligned, and nothing here panics so it's drop-safe.
1195        unsafe {
1196            // Note that it's critical that these use `copy_nonoverlapping`,
1197            // rather than `read`/`write`, to avoid #134713 if T has padding.
1198            let mut temp = MaybeUninit::<T>::uninit();
1199            copy_nonoverlapping(x, temp.as_mut_ptr(), 1);
1200            copy_nonoverlapping(y, x, 1);
1201            copy_nonoverlapping(temp.as_ptr(), y, 1);
1202        }
1203
1204        i += 1;
1205    }
1206}

1208// Don't let MIR inline this, because we really want it to keep its noalias metadata
1209#[rustc_no_mir_inline]
1210#[inline]
1211#[cfg(not(feature = "ferrocene_certified"))]
1212fn swap_chunk<const N: usize>(x: &mut MaybeUninit<[u8; N]>, y: &mut MaybeUninit<[u8; N]>) {
1213    let a = *x;
1214    let b = *y;
1215    *x = b;
1216    *y = a;
1217}
1218
1219#[inline]
1220#[cfg(not(feature = "ferrocene_certified"))]
1221unsafe fn swap_nonoverlapping_bytes(x: *mut u8, y: *mut u8, bytes: NonZero<usize>) {
1222    // Same as `swap_nonoverlapping::<[u8; N]>`.
1223    unsafe fn swap_nonoverlapping_chunks<const N: usize>(
1224        x: *mut MaybeUninit<[u8; N]>,
1225        y: *mut MaybeUninit<[u8; N]>,
1226        chunks: NonZero<usize>,
1227    ) {
1228        let chunks = chunks.get();
1229        for i in 0..chunks {
1230            // SAFETY: i is in [0, chunks) so the adds and dereferences are in-bounds.
1231            unsafe { swap_chunk(&mut *x.add(i), &mut *y.add(i)) };
1232        }
1233    }
1234
1235    // Same as `swap_nonoverlapping_bytes`, but accepts at most 1+2+4=7 bytes
1236    #[inline]
1237    unsafe fn swap_nonoverlapping_short(x: *mut u8, y: *mut u8, bytes: NonZero<usize>) {
1238        // Tail handling for auto-vectorized code sometimes has element-at-a-time behaviour,
1239        // see <https://github.com/rust-lang/rust/issues/134946>.
1240        // By swapping as different sizes, rather than as a loop over bytes,
1241        // we make sure not to end up with, say, seven byte-at-a-time copies.
1242
1243        let bytes = bytes.get();
1244        let mut i = 0;
1245        macro_rules! swap_prefix {
1246            ($($n:literal)+) => {$(
1247                if (bytes & $n) != 0 {
1248                    // SAFETY: `i` can only have the same bits set as those in bytes,
1249                    // so these `add`s are in-bounds of `bytes`.  But the bit for
1250                    // `$n` hasn't been set yet, so the `$n` bytes that `swap_chunk`
1251                    // will read and write are within the usable range.
1252                    unsafe { swap_chunk::<$n>(&mut*x.add(i).cast(), &mut*y.add(i).cast()) };
1253                    i |= $n;
1254                }
1255            )+};
1256        }
1257        swap_prefix!(4 2 1);
1258        debug_assert_eq!(i, bytes);
1259    }
1260
1261    const CHUNK_SIZE: usize = size_of::<*const ()>();
1262    let bytes = bytes.get();
1263
1264    let chunks = bytes / CHUNK_SIZE;
1265    let tail = bytes % CHUNK_SIZE;
1266    if let Some(chunks) = NonZero::new(chunks) {
1267        // SAFETY: this is bytes/CHUNK_SIZE*CHUNK_SIZE bytes, which is <= bytes,
1268        // so it's within the range of our non-overlapping bytes.
1269        unsafe { swap_nonoverlapping_chunks::<CHUNK_SIZE>(x.cast(), y.cast(), chunks) };
1270    }
1271    if let Some(tail) = NonZero::new(tail) {
1272        const { assert!(CHUNK_SIZE <= 8) };
1273        let delta = chunks * CHUNK_SIZE;
1274        // SAFETY: the tail length is below CHUNK SIZE because of the remainder,
1275        // and CHUNK_SIZE is at most 8 by the const assert, so tail <= 7
1276        unsafe { swap_nonoverlapping_short(x.add(delta), y.add(delta), tail) };
1277    }
1278}

1289///

1291///
1292/// * `dst` must be [valid] for both reads and writes.

1295///
1296/// * `dst` must point to a properly initialized value of type `T`.

1323pub const unsafe fn replace<T>(dst: *mut T, src: T) -> T {
1324    // SAFETY: the caller must guarantee that `dst` is valid to be
1325    // cast to a mutable reference (valid for writes, aligned, initialized),
1326    // and cannot overlap `src` since `dst` must point to a distinct
1327    // allocated object.
1328    unsafe {
1329        ub_checks::assert_unsafe_precondition!(

1331            "ptr::replace requires that the pointer argument is aligned and non-null",

1339    }

1342/// Reads the value from `src` without moving it. This leaves the
1343/// memory in `src` unchanged.
1344///
1345/// # Safety
1346///
1347/// Behavior is undefined if any of the following conditions are violated:
1348///
1349/// * `src` must be [valid] for reads.
1350///
1351/// * `src` must be properly aligned. Use [`read_unaligned`] if this is not the
1352///   case.
1353///
1354/// * `src` must point to a properly initialized value of type `T`.
1355///
1356/// Note that even if `T` has size `0`, the pointer must be properly aligned.
1357///
1358/// # Examples
1359///
1360/// Basic usage:
1361///
1362/// ```
1363/// let x = 12;
1364/// let y = &x as *const i32;
1365///
1366/// unsafe {
1367///     assert_eq!(std::ptr::read(y), 12);
1368/// }
1369/// ```
1370///
1371/// Manually implement [`mem::swap`]:
1372///
1373/// ```
1374/// use std::ptr;
1375///
1376/// fn swap<T>(a: &mut T, b: &mut T) {
1377///     unsafe {
1378///         // Create a bitwise copy of the value at `a` in `tmp`.
1379///         let tmp = ptr::read(a);
1380///
1381///         // Exiting at this point (either by explicitly returning or by
1382///         // calling a function which panics) would cause the value in `tmp` to
1383///         // be dropped while the same value is still referenced by `a`. This
1384///         // could trigger undefined behavior if `T` is not `Copy`.
1385///
1386///         // Create a bitwise copy of the value at `b` in `a`.
1387///         // This is safe because mutable references cannot alias.
1388///         ptr::copy_nonoverlapping(b, a, 1);
1389///
1390///         // As above, exiting here could trigger undefined behavior because
1391///         // the same value is referenced by `a` and `b`.
1392///
1393///         // Move `tmp` into `b`.
1394///         ptr::write(b, tmp);
1395///
1396///         // `tmp` has been moved (`write` takes ownership of its second argument),
1397///         // so nothing is dropped implicitly here.
1398///     }
1399/// }
1400///
1401/// let mut foo = "foo".to_owned();
1402/// let mut bar = "bar".to_owned();
1403///
1404/// swap(&mut foo, &mut bar);
1405///
1406/// assert_eq!(foo, "bar");
1407/// assert_eq!(bar, "foo");
1408/// ```
1409///
1410/// ## Ownership of the Returned Value
1411///
1412/// `read` creates a bitwise copy of `T`, regardless of whether `T` is [`Copy`].
1413/// If `T` is not [`Copy`], using both the returned value and the value at
1414/// `*src` can violate memory safety. Note that assigning to `*src` counts as a
1415/// use because it will attempt to drop the value at `*src`.
1416///
1417/// [`write()`] can be used to overwrite data without causing it to be dropped.
1418///
1419/// ```
1420/// use std::ptr;
1421///
1422/// let mut s = String::from("foo");
1423/// unsafe {
1424///     // `s2` now points to the same underlying memory as `s`.
1425///     let mut s2: String = ptr::read(&s);
1426///
1427///     assert_eq!(s2, "foo");
1428///
1429///     // Assigning to `s2` causes its original value to be dropped. Beyond
1430///     // this point, `s` must no longer be used, as the underlying memory has
1431///     // been freed.
1432///     s2 = String::default();
1433///     assert_eq!(s2, "");
1434///
1435///     // Assigning to `s` would cause the old value to be dropped again,
1436///     // resulting in undefined behavior.
1437///     // s = String::from("bar"); // ERROR
1438///
1439///     // `ptr::write` can be used to overwrite a value without dropping it.
1440///     ptr::write(&mut s, String::from("bar"));
1441/// }
1442///
1443/// assert_eq!(s, "bar");
1444/// ```
1445///
1446/// [valid]: self#safety
1447#[inline]
1448#[stable(feature = "rust1", since = "1.0.0")]
1449#[rustc_const_stable(feature = "const_ptr_read", since = "1.71.0")]
1450#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
1451#[rustc_diagnostic_item = "ptr_read"]
1452pub const unsafe fn read<T>(src: *const T) -> T {
1453    // It would be semantically correct to implement this via `copy_nonoverlapping`
1454    // and `MaybeUninit`, as was done before PR #109035. Calling `assume_init`
1455    // provides enough information to know that this is a typed operation.
1456
1457    // However, as of March 2023 the compiler was not capable of taking advantage
1458    // of that information. Thus, the implementation here switched to an intrinsic,
1459    // which lowers to `_0 = *src` in MIR, to address a few issues:
1460    //
1461    // - Using `MaybeUninit::assume_init` after a `copy_nonoverlapping` was not
1462    //   turning the untyped copy into a typed load. As such, the generated
1463    //   `load` in LLVM didn't get various metadata, such as `!range` (#73258),
1464    //   `!nonnull`, and `!noundef`, resulting in poorer optimization.
1465    // - Going through the extra local resulted in multiple extra copies, even
1466    //   in optimized MIR.  (Ignoring StorageLive/Dead, the intrinsic is one
1467    //   MIR statement, while the previous implementation was eight.)  LLVM
1468    //   could sometimes optimize them away, but because `read` is at the core
1469    //   of so many things, not having them in the first place improves what we
1470    //   hand off to the backend.  For example, `mem::replace::<Big>` previously
1471    //   emitted 4 `alloca` and 6 `memcpy`s, but is now 1 `alloc` and 3 `memcpy`s.
1472    // - In general, this approach keeps us from getting any more bugs (like
1473    //   #106369) that boil down to "`read(p)` is worse than `*p`", as this
1474    //   makes them look identical to the backend (or other MIR consumers).
1475    //
1476    // Future enhancements to MIR optimizations might well allow this to return
1477    // to the previous implementation, rather than using an intrinsic.
1478
1479    // SAFETY: the caller must guarantee that `src` is valid for reads.
1480    unsafe {
1481        #[cfg(debug_assertions)] // Too expensive to always enable (for now?)
1482        ub_checks::assert_unsafe_precondition!(
1483            check_language_ub,
1484            "ptr::read requires that the pointer argument is aligned and non-null",
1485            (
1486                addr: *const () = src as *const (),
1487                align: usize = align_of::<T>(),
1488                is_zst: bool = T::IS_ZST,
1489            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, is_zst)
1490        );
1491        crate::intrinsics::read_via_copy(src)
1492    }
1493}

1495/// Reads the value from `src` without moving it. This leaves the
1496/// memory in `src` unchanged.
1497///
1498/// Unlike [`read`], `read_unaligned` works with unaligned pointers.
1499///
1500/// # Safety
1501///
1502/// Behavior is undefined if any of the following conditions are violated:
1503///
1504/// * `src` must be [valid] for reads.
1505///
1506/// * `src` must point to a properly initialized value of type `T`.
1507///
1508/// Like [`read`], `read_unaligned` creates a bitwise copy of `T`, regardless of
1509/// whether `T` is [`Copy`]. If `T` is not [`Copy`], using both the returned
1510/// value and the value at `*src` can [violate memory safety][read-ownership].
1511///
1512/// [read-ownership]: read#ownership-of-the-returned-value
1513/// [valid]: self#safety
1514///
1515/// ## On `packed` structs
1516///
1517/// Attempting to create a raw pointer to an `unaligned` struct field with
1518/// an expression such as `&packed.unaligned as *const FieldType` creates an
1519/// intermediate unaligned reference before converting that to a raw pointer.
1520/// That this reference is temporary and immediately cast is inconsequential
1521/// as the compiler always expects references to be properly aligned.
1522/// As a result, using `&packed.unaligned as *const FieldType` causes immediate
1523/// *undefined behavior* in your program.
1524///
1525/// Instead you must use the `&raw const` syntax to create the pointer.
1526/// You may use that constructed pointer together with this function.
1527///
1528/// An example of what not to do and how this relates to `read_unaligned` is:
1529///
1530/// ```
1531/// #[repr(packed, C)]
1532/// struct Packed {
1533///     _padding: u8,
1534///     unaligned: u32,
1535/// }
1536///
1537/// let packed = Packed {
1538///     _padding: 0x00,
1539///     unaligned: 0x01020304,
1540/// };
1541///
1542/// // Take the address of a 32-bit integer which is not aligned.
1543/// // In contrast to `&packed.unaligned as *const _`, this has no undefined behavior.
1544/// let unaligned = &raw const packed.unaligned;
1545///
1546/// let v = unsafe { std::ptr::read_unaligned(unaligned) };
1547/// assert_eq!(v, 0x01020304);
1548/// ```
1549///
1550/// Accessing unaligned fields directly with e.g. `packed.unaligned` is safe however.
1551///
1552/// # Examples
1553///
1554/// Read a `usize` value from a byte buffer:
1555///
1556/// ```
1557/// fn read_usize(x: &[u8]) -> usize {
1558///     assert!(x.len() >= size_of::<usize>());
1559///
1560///     let ptr = x.as_ptr() as *const usize;
1561///
1562///     unsafe { ptr.read_unaligned() }
1563/// }
1564/// ```
1565#[inline]
1566#[stable(feature = "ptr_unaligned", since = "1.17.0")]
1567#[rustc_const_stable(feature = "const_ptr_read", since = "1.71.0")]
1568#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
1569#[rustc_diagnostic_item = "ptr_read_unaligned"]
1570#[cfg(not(feature = "ferrocene_certified"))]
1571pub const unsafe fn read_unaligned<T>(src: *const T) -> T {
1572    let mut tmp = MaybeUninit::<T>::uninit();
1573    // SAFETY: the caller must guarantee that `src` is valid for reads.
1574    // `src` cannot overlap `tmp` because `tmp` was just allocated on
1575    // the stack as a separate allocated object.
1576    //
1577    // Also, since we just wrote a valid value into `tmp`, it is guaranteed
1578    // to be properly initialized.
1579    unsafe {
1580        copy_nonoverlapping(src as *const u8, tmp.as_mut_ptr() as *mut u8, size_of::<T>());
1581        tmp.assume_init()
1582    }
1583}

1585/// Overwrites a memory location with the given value without reading or
1586/// dropping the old value.
1587///
1588/// `write` does not drop the contents of `dst`. This is safe, but it could leak
1589/// allocations or resources, so care should be taken not to overwrite an object
1590/// that should be dropped.
1591///
1592/// Additionally, it does not drop `src`. Semantically, `src` is moved into the
1593/// location pointed to by `dst`.
1594///
1595/// This is appropriate for initializing uninitialized memory, or overwriting
1596/// memory that has previously been [`read`] from.
1597///
1598/// # Safety
1599///
1600/// Behavior is undefined if any of the following conditions are violated:
1601///
1602/// * `dst` must be [valid] for writes.
1603///
1604/// * `dst` must be properly aligned. Use [`write_unaligned`] if this is not the
1605///   case.
1606///
1607/// Note that even if `T` has size `0`, the pointer must be properly aligned.
1608///
1609/// [valid]: self#safety
1610///
1611/// # Examples
1612///
1613/// Basic usage:
1614///
1615/// ```
1616/// let mut x = 0;
1617/// let y = &mut x as *mut i32;
1618/// let z = 12;
1619///
1620/// unsafe {
1621///     std::ptr::write(y, z);
1622///     assert_eq!(std::ptr::read(y), 12);
1623/// }
1624/// ```
1625///
1626/// Manually implement [`mem::swap`]:
1627///
1628/// ```
1629/// use std::ptr;
1630///
1631/// fn swap<T>(a: &mut T, b: &mut T) {
1632///     unsafe {
1633///         // Create a bitwise copy of the value at `a` in `tmp`.
1634///         let tmp = ptr::read(a);
1635///
1636///         // Exiting at this point (either by explicitly returning or by
1637///         // calling a function which panics) would cause the value in `tmp` to
1638///         // be dropped while the same value is still referenced by `a`. This
1639///         // could trigger undefined behavior if `T` is not `Copy`.
1640///
1641///         // Create a bitwise copy of the value at `b` in `a`.
1642///         // This is safe because mutable references cannot alias.
1643///         ptr::copy_nonoverlapping(b, a, 1);
1644///
1645///         // As above, exiting here could trigger undefined behavior because
1646///         // the same value is referenced by `a` and `b`.
1647///
1648///         // Move `tmp` into `b`.
1649///         ptr::write(b, tmp);
1650///
1651///         // `tmp` has been moved (`write` takes ownership of its second argument),
1652///         // so nothing is dropped implicitly here.
1653///     }
1654/// }
1655///
1656/// let mut foo = "foo".to_owned();
1657/// let mut bar = "bar".to_owned();
1658///
1659/// swap(&mut foo, &mut bar);
1660///
1661/// assert_eq!(foo, "bar");
1662/// assert_eq!(bar, "foo");
1663/// ```
1664#[inline]
1665#[stable(feature = "rust1", since = "1.0.0")]
1666#[rustc_const_stable(feature = "const_ptr_write", since = "1.83.0")]
1667#[rustc_diagnostic_item = "ptr_write"]
1668#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
1669pub const unsafe fn write<T>(dst: *mut T, src: T) {
1670    // Semantically, it would be fine for this to be implemented as a
1671    // `copy_nonoverlapping` and appropriate drop suppression of `src`.
1672
1673    // However, implementing via that currently produces more MIR than is ideal.
1674    // Using an intrinsic keeps it down to just the simple `*dst = move src` in
1675    // MIR (11 statements shorter, at the time of writing), and also allows
1676    // `src` to stay an SSA value in codegen_ssa, rather than a memory one.
1677
1678    // SAFETY: the caller must guarantee that `dst` is valid for writes.
1679    // `dst` cannot overlap `src` because the caller has mutable access
1680    // to `dst` while `src` is owned by this function.
1681    unsafe {
1682        #[cfg(debug_assertions)] // Too expensive to always enable (for now?)
1683        ub_checks::assert_unsafe_precondition!(
1684            check_language_ub,
1685            "ptr::write requires that the pointer argument is aligned and non-null",
1686            (
1687                addr: *mut () = dst as *mut (),
1688                align: usize = align_of::<T>(),
1689                is_zst: bool = T::IS_ZST,
1690            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, is_zst)
1691        );
1692        intrinsics::write_via_move(dst, src)
1693    }
1694}

1696/// Overwrites a memory location with the given value without reading or
1697/// dropping the old value.
1698///
1699/// Unlike [`write()`], the pointer may be unaligned.
1700///
1701/// `write_unaligned` does not drop the contents of `dst`. This is safe, but it
1702/// could leak allocations or resources, so care should be taken not to overwrite
1703/// an object that should be dropped.
1704///
1705/// Additionally, it does not drop `src`. Semantically, `src` is moved into the
1706/// location pointed to by `dst`.
1707///
1708/// This is appropriate for initializing uninitialized memory, or overwriting
1709/// memory that has previously been read with [`read_unaligned`].
1710///
1711/// # Safety
1712///
1713/// Behavior is undefined if any of the following conditions are violated:
1714///
1715/// * `dst` must be [valid] for writes.
1716///
1717/// [valid]: self#safety
1718///
1719/// ## On `packed` structs
1720///
1721/// Attempting to create a raw pointer to an `unaligned` struct field with
1722/// an expression such as `&packed.unaligned as *const FieldType` creates an
1723/// intermediate unaligned reference before converting that to a raw pointer.
1724/// That this reference is temporary and immediately cast is inconsequential
1725/// as the compiler always expects references to be properly aligned.
1726/// As a result, using `&packed.unaligned as *const FieldType` causes immediate
1727/// *undefined behavior* in your program.
1728///
1729/// Instead, you must use the `&raw mut` syntax to create the pointer.
1730/// You may use that constructed pointer together with this function.
1731///
1732/// An example of how to do it and how this relates to `write_unaligned` is:
1733///
1734/// ```
1735/// #[repr(packed, C)]
1736/// struct Packed {
1737///     _padding: u8,
1738///     unaligned: u32,
1739/// }
1740///
1741/// let mut packed: Packed = unsafe { std::mem::zeroed() };
1742///
1743/// // Take the address of a 32-bit integer which is not aligned.
1744/// // In contrast to `&packed.unaligned as *mut _`, this has no undefined behavior.
1745/// let unaligned = &raw mut packed.unaligned;
1746///
1747/// unsafe { std::ptr::write_unaligned(unaligned, 42) };
1748///
1749/// assert_eq!({packed.unaligned}, 42); // `{...}` forces copying the field instead of creating a reference.
1750/// ```
1751///
1752/// Accessing unaligned fields directly with e.g. `packed.unaligned` is safe however
1753/// (as can be seen in the `assert_eq!` above).
1754///
1755/// # Examples
1756///
1757/// Write a `usize` value to a byte buffer:
1758///
1759/// ```
1760/// fn write_usize(x: &mut [u8], val: usize) {
1761///     assert!(x.len() >= size_of::<usize>());
1762///
1763///     let ptr = x.as_mut_ptr() as *mut usize;
1764///
1765///     unsafe { ptr.write_unaligned(val) }
1766/// }
1767/// ```
1768#[inline]
1769#[stable(feature = "ptr_unaligned", since = "1.17.0")]
1770#[rustc_const_stable(feature = "const_ptr_write", since = "1.83.0")]
1771#[rustc_diagnostic_item = "ptr_write_unaligned"]
1772#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
1773#[cfg(not(feature = "ferrocene_certified"))]
1774pub const unsafe fn write_unaligned<T>(dst: *mut T, src: T) {
1775    // SAFETY: the caller must guarantee that `dst` is valid for writes.
1776    // `dst` cannot overlap `src` because the caller has mutable access
1777    // to `dst` while `src` is owned by this function.
1778    unsafe {
1779        copy_nonoverlapping((&raw const src) as *const u8, dst as *mut u8, size_of::<T>());
1780        // We are calling the intrinsic directly to avoid function calls in the generated code.
1781        intrinsics::forget(src);
1782    }
1783}

1785/// Performs a volatile read of the value from `src` without moving it. This
1786/// leaves the memory in `src` unchanged.
1787///
1788/// Volatile operations are intended to act on I/O memory, and are guaranteed
1789/// to not be elided or reordered by the compiler across other volatile
1790/// operations.
1791///
1792/// # Notes
1793///
1794/// Rust does not currently have a rigorously and formally defined memory model,
1795/// so the precise semantics of what "volatile" means here is subject to change
1796/// over time. That being said, the semantics will almost always end up pretty
1797/// similar to [C11's definition of volatile][c11].
1798///
1799/// The compiler shouldn't change the relative order or number of volatile
1800/// memory operations. However, volatile memory operations on zero-sized types
1801/// (e.g., if a zero-sized type is passed to `read_volatile`) are noops
1802/// and may be ignored.
1803///
1804/// [c11]: http://www.open-std.org/jtc1/sc22/wg14/www/docs/n1570.pdf
1805///
1806/// # Safety
1807///
1808/// Behavior is undefined if any of the following conditions are violated:
1809///
1810/// * `src` must be [valid] for reads.
1811///
1812/// * `src` must be properly aligned.
1813///
1814/// * `src` must point to a properly initialized value of type `T`.
1815///
1816/// Like [`read`], `read_volatile` creates a bitwise copy of `T`, regardless of
1817/// whether `T` is [`Copy`]. If `T` is not [`Copy`], using both the returned
1818/// value and the value at `*src` can [violate memory safety][read-ownership].
1819/// However, storing non-[`Copy`] types in volatile memory is almost certainly
1820/// incorrect.
1821///
1822/// Note that even if `T` has size `0`, the pointer must be properly aligned.
1823///
1824/// [valid]: self#safety
1825/// [read-ownership]: read#ownership-of-the-returned-value
1826///
1827/// Just like in C, whether an operation is volatile has no bearing whatsoever
1828/// on questions involving concurrent access from multiple threads. Volatile
1829/// accesses behave exactly like non-atomic accesses in that regard. In particular,
1830/// a race between a `read_volatile` and any write operation to the same location
1831/// is undefined behavior.
1832///
1833/// # Examples
1834///
1835/// Basic usage:
1836///
1837/// ```
1838/// let x = 12;
1839/// let y = &x as *const i32;
1840///
1841/// unsafe {
1842///     assert_eq!(std::ptr::read_volatile(y), 12);
1843/// }
1844/// ```
1845#[inline]
1846#[stable(feature = "volatile", since = "1.9.0")]
1847#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
1848#[rustc_diagnostic_item = "ptr_read_volatile"]
1849#[cfg(not(feature = "ferrocene_certified"))]
1850pub unsafe fn read_volatile<T>(src: *const T) -> T {
1851    // SAFETY: the caller must uphold the safety contract for `volatile_load`.
1852    unsafe {
1853        ub_checks::assert_unsafe_precondition!(
1854            check_language_ub,
1855            "ptr::read_volatile requires that the pointer argument is aligned and non-null",
1856            (
1857                addr: *const () = src as *const (),
1858                align: usize = align_of::<T>(),
1859                is_zst: bool = T::IS_ZST,
1860            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, is_zst)
1861        );
1862        intrinsics::volatile_load(src)
1863    }
1864}

1947/// Align pointer `p`.
1948///
1949/// Calculate offset (in terms of elements of `size_of::<T>()` stride) that has to be applied
1950/// to pointer `p` so that pointer `p` would get aligned to `a`.
1951///
1952/// # Safety
1953/// `a` must be a power of two.
1954///
1955/// # Notes
1956/// This implementation has been carefully tailored to not panic. It is UB for this to panic.
1957/// The only real change that can be made here is change of `INV_TABLE_MOD_16` and associated
1958/// constants.
1959///
1960/// If we ever decide to make it possible to call the intrinsic with `a` that is not a
1961/// power-of-two, it will probably be more prudent to just change to a naive implementation rather
1962/// than trying to adapt this to accommodate that change.
1963///
1964/// Any questions go to @nagisa.
1965#[allow(ptr_to_integer_transmute_in_consts)]
1966#[cfg(not(feature = "ferrocene_certified"))]
1967pub(crate) unsafe fn align_offset<T: Sized>(p: *const T, a: usize) -> usize {
1968    // FIXME(#75598): Direct use of these intrinsics improves codegen significantly at opt-level <=
1969    // 1, where the method versions of these operations are not inlined.
1970    use intrinsics::{
1971        assume, cttz_nonzero, exact_div, mul_with_overflow, unchecked_rem, unchecked_shl,
1972        unchecked_shr, unchecked_sub, wrapping_add, wrapping_mul, wrapping_sub,
1973    };
1974
1975    /// Calculate multiplicative modular inverse of `x` modulo `m`.
1976    ///
1977    /// This implementation is tailored for `align_offset` and has following preconditions:
1978    ///
1979    /// * `m` is a power-of-two;
1980    /// * `x < m`; (if `x  m`, pass in `x % m` instead)
1981    ///
1982    /// Implementation of this function shall not panic. Ever.
1983    #[inline]
1984    const unsafe fn mod_inv(x: usize, m: usize) -> usize {
1985        /// Multiplicative modular inverse table modulo 2 = 16.
1986        ///
1987        /// Note, that this table does not contain values where inverse does not exist (i.e., for
1988        /// `0 mod 16`, `2 mod 16`, etc.)
1989        const INV_TABLE_MOD_16: [u8; 8] = [1, 11, 13, 7, 9, 3, 5, 15];
1990        /// Modulo for which the `INV_TABLE_MOD_16` is intended.
1991        const INV_TABLE_MOD: usize = 16;
1992
1993        // SAFETY: `m` is required to be a power-of-two, hence non-zero.
1994        let m_minus_one = unsafe { unchecked_sub(m, 1) };
1995        let mut inverse = INV_TABLE_MOD_16[(x & (INV_TABLE_MOD - 1)) >> 1] as usize;
1996        let mut mod_gate = INV_TABLE_MOD;
1997        // We iterate "up" using the following formula:
1998        //
1999        // $$ xy  1 (mod 2)  xy (2 - xy)  1 (mod 2) $$
2000        //
2001        // This application needs to be applied at least until `2  m`, at which point we can
2002        // finally reduce the computation to our desired `m` by taking `inverse mod m`.
2003        //
2004        // This computation is `O(log log m)`, which is to say, that on 64-bit machines this loop
2005        // will always finish in at most 4 iterations.
2006        loop {
2007            // y = y * (2 - xy) mod n
2008            //
2009            // Note, that we use wrapping operations here intentionally  the original formula
2010            // uses e.g., subtraction `mod n`. It is entirely fine to do them `mod
2011            // usize::MAX` instead, because we take the result `mod n` at the end
2012            // anyway.
2013            if mod_gate >= m {
2014                break;
2015            }
2016            inverse = wrapping_mul(inverse, wrapping_sub(2usize, wrapping_mul(x, inverse)));
2017            let (new_gate, overflow) = mul_with_overflow(mod_gate, mod_gate);
2018            if overflow {
2019                break;
2020            }
2021            mod_gate = new_gate;
2022        }
2023        inverse & m_minus_one
2024    }
2025
2026    let stride = size_of::<T>();
2027
2028    let addr: usize = p.addr();
2029
2030    // SAFETY: `a` is a power-of-two, therefore non-zero.
2031    let a_minus_one = unsafe { unchecked_sub(a, 1) };
2032
2033    if stride == 0 {
2034        // SPECIAL_CASE: handle 0-sized types. No matter how many times we step, the address will
2035        // stay the same, so no offset will be able to align the pointer unless it is already
2036        // aligned. This branch _will_ be optimized out as `stride` is known at compile-time.
2037        let p_mod_a = addr & a_minus_one;
2038        return if p_mod_a == 0 { 0 } else { usize::MAX };
2039    }
2040
2041    // SAFETY: `stride == 0` case has been handled by the special case above.
2042    let a_mod_stride = unsafe { unchecked_rem(a, stride) };
2043    if a_mod_stride == 0 {
2044        // SPECIAL_CASE: In cases where the `a` is divisible by `stride`, byte offset to align a
2045        // pointer can be computed more simply through `-p (mod a)`. In the off-chance the byte
2046        // offset is not a multiple of `stride`, the input pointer was misaligned and no pointer
2047        // offset will be able to produce a `p` aligned to the specified `a`.
2048        //
2049        // The naive `-p (mod a)` equation inhibits LLVM's ability to select instructions
2050        // like `lea`. We compute `(round_up_to_next_alignment(p, a) - p)` instead. This
2051        // redistributes operations around the load-bearing, but pessimizing `and` instruction
2052        // sufficiently for LLVM to be able to utilize the various optimizations it knows about.
2053        //
2054        // LLVM handles the branch here particularly nicely. If this branch needs to be evaluated
2055        // at runtime, it will produce a mask `if addr_mod_stride == 0 { 0 } else { usize::MAX }`
2056        // in a branch-free way and then bitwise-OR it with whatever result the `-p mod a`
2057        // computation produces.
2058
2059        let aligned_address = wrapping_add(addr, a_minus_one) & wrapping_sub(0, a);
2060        let byte_offset = wrapping_sub(aligned_address, addr);
2061        // FIXME: Remove the assume after <https://github.com/llvm/llvm-project/issues/62502>
2062        // SAFETY: Masking by `-a` can only affect the low bits, and thus cannot have reduced
2063        // the value by more than `a-1`, so even though the intermediate values might have
2064        // wrapped, the byte_offset is always in `[0, a)`.
2065        unsafe { assume(byte_offset < a) };
2066
2067        // SAFETY: `stride == 0` case has been handled by the special case above.
2068        let addr_mod_stride = unsafe { unchecked_rem(addr, stride) };
2069
2070        return if addr_mod_stride == 0 {
2071            // SAFETY: `stride` is non-zero. This is guaranteed to divide exactly as well, because
2072            // addr has been verified to be aligned to the original types alignment requirements.
2073            unsafe { exact_div(byte_offset, stride) }
2074        } else {
2075            usize::MAX
2076        };
2077    }
2078
2079    // GENERAL_CASE: From here on were handling the very general case where `addr` may be
2080    // misaligned, there isnt an obvious relationship between `stride` and `a` that we can take an
2081    // advantage of, etc. This case produces machine code that isnt particularly high quality,
2082    // compared to the special cases above. The code produced here is still within the realm of
2083    // miracles, given the situations this case has to deal with.
2084
2085    // SAFETY: a is power-of-two hence non-zero. stride == 0 case is handled above.
2086    // FIXME(const-hack) replace with min
2087    let gcdpow = unsafe {
2088        let x = cttz_nonzero(stride);
2089        let y = cttz_nonzero(a);
2090        if x < y { x } else { y }
2091    };
2092    // SAFETY: gcdpow has an upper-bound thats at most the number of bits in a `usize`.
2093    let gcd = unsafe { unchecked_shl(1usize, gcdpow) };
2094    // SAFETY: gcd is always greater or equal to 1.
2095    if addr & unsafe { unchecked_sub(gcd, 1) } == 0 {
2096        // This branch solves for the following linear congruence equation:
2097        //
2098        // ` p + so = 0 mod a `
2099        //
2100        // `p` here is the pointer value, `s` - stride of `T`, `o` offset in `T`s, and `a` - the
2101        // requested alignment.
2102        //
2103        // With `g = gcd(a, s)`, and the above condition asserting that `p` is also divisible by
2104        // `g`, we can denote `a' = a/g`, `s' = s/g`, `p' = p/g`, then this becomes equivalent to:
2105        //
2106        // ` p' + s'o = 0 mod a' `
2107        // ` o = (a' - (p' mod a')) * (s'^-1 mod a') `
2108        //
2109        // The first term is "the relative alignment of `p` to `a`" (divided by the `g`), the
2110        // second term is "how does incrementing `p` by `s` bytes change the relative alignment of
2111        // `p`" (again divided by `g`). Division by `g` is necessary to make the inverse well
2112        // formed if `a` and `s` are not co-prime.
2113        //
2114        // Furthermore, the result produced by this solution is not "minimal", so it is necessary
2115        // to take the result `o mod lcm(s, a)`. This `lcm(s, a)` is the same as `a'`.
2116
2117        // SAFETY: `gcdpow` has an upper-bound not greater than the number of trailing 0-bits in
2118        // `a`.
2119        let a2 = unsafe { unchecked_shr(a, gcdpow) };
2120        // SAFETY: `a2` is non-zero. Shifting `a` by `gcdpow` cannot shift out any of the set bits
2121        // in `a` (of which it has exactly one).
2122        let a2minus1 = unsafe { unchecked_sub(a2, 1) };
2123        // SAFETY: `gcdpow` has an upper-bound not greater than the number of trailing 0-bits in
2124        // `a`.
2125        let s2 = unsafe { unchecked_shr(stride & a_minus_one, gcdpow) };
2126        // SAFETY: `gcdpow` has an upper-bound not greater than the number of trailing 0-bits in
2127        // `a`. Furthermore, the subtraction cannot overflow, because `a2 = a >> gcdpow` will
2128        // always be strictly greater than `(p % a) >> gcdpow`.
2129        let minusp2 = unsafe { unchecked_sub(a2, unchecked_shr(addr & a_minus_one, gcdpow)) };
2130        // SAFETY: `a2` is a power-of-two, as proven above. `s2` is strictly less than `a2`
2131        // because `(s % a) >> gcdpow` is strictly less than `a >> gcdpow`.
2132        return wrapping_mul(minusp2, unsafe { mod_inv(s2, a2) }) & a2minus1;
2133    }
2134
2135    // Cannot be aligned at all.
2136    usize::MAX
2137}

2139/// Compares raw pointers for equality.
2140///
2141/// This is the same as using the `==` operator, but less generic:
2142/// the arguments have to be `*const T` raw pointers,
2143/// not anything that implements `PartialEq`.
2144///
2145/// This can be used to compare `&T` references (which coerce to `*const T` implicitly)
2146/// by their address rather than comparing the values they point to
2147/// (which is what the `PartialEq for &T` implementation does).
2148///
2149/// When comparing wide pointers, both the address and the metadata are tested for equality.
2150/// However, note that comparing trait object pointers (`*const dyn Trait`) is unreliable: pointers
2151/// to values of the same underlying type can compare inequal (because vtables are duplicated in
2152/// multiple codegen units), and pointers to values of *different* underlying type can compare equal
2153/// (since identical vtables can be deduplicated within a codegen unit).
2154///
2155/// # Examples
2156///
2157/// ```
2158/// use std::ptr;
2159///
2160/// let five = 5;
2161/// let other_five = 5;
2162/// let five_ref = &five;
2163/// let same_five_ref = &five;
2164/// let other_five_ref = &other_five;
2165///
2166/// assert!(five_ref == same_five_ref);
2167/// assert!(ptr::eq(five_ref, same_five_ref));
2168///
2169/// assert!(five_ref == other_five_ref);
2170/// assert!(!ptr::eq(five_ref, other_five_ref));
2171/// ```
2172///
2173/// Slices are also compared by their length (fat pointers):
2174///
2175/// ```
2176/// let a = [1, 2, 3];
2177/// assert!(std::ptr::eq(&a[..3], &a[..3]));
2178/// assert!(!std::ptr::eq(&a[..2], &a[..3]));
2179/// assert!(!std::ptr::eq(&a[0..2], &a[1..3]));
2180/// ```
2181#[stable(feature = "ptr_eq", since = "1.17.0")]
2182#[inline(always)]
2183#[must_use = "pointer comparison produces a value"]
2184#[rustc_diagnostic_item = "ptr_eq"]
2185#[allow(ambiguous_wide_pointer_comparisons)] // it's actually clear here
2186#[cfg(not(feature = "ferrocene_certified"))]
2187pub fn eq<T: ?Sized>(a: *const T, b: *const T) -> bool {
2188    a == b
2189}

2192/// ignoring any metadata in fat pointers.

2196///
2197/// # Examples

2216/// Compares the *addresses* of the two function pointers for equality.
2217///
2218/// This is the same as `f == g`, but using this function makes clear that the potentially
2219/// surprising semantics of function pointer comparison are involved.
2220///
2221/// There are **very few guarantees** about how functions are compiled and they have no intrinsic
2222/// identity; in particular, this comparison:
2223///
2224/// * May return `true` unexpectedly, in cases where functions are equivalent.
2225///
2226///   For example, the following program is likely (but not guaranteed) to print `(true, true)`
2227///   when compiled with optimization:
2228///
2229///   ```
2230///   let f: fn(i32) -> i32 = |x| x;
2231///   let g: fn(i32) -> i32 = |x| x + 0;  // different closure, different body
2232///   let h: fn(u32) -> u32 = |x| x + 0;  // different signature too
2233///   dbg!(std::ptr::fn_addr_eq(f, g), std::ptr::fn_addr_eq(f, h)); // not guaranteed to be equal
2234///   ```
2235///
2236/// * May return `false` in any case.
2237///
2238///   This is particularly likely with generic functions but may happen with any function.
2239///   (From an implementation perspective, this is possible because functions may sometimes be
2240///   processed more than once by the compiler, resulting in duplicate machine code.)
2241///
2242/// Despite these false positives and false negatives, this comparison can still be useful.
2243/// Specifically, if
2244///
2245/// * `T` is the same type as `U`, `T` is a [subtype] of `U`, or `U` is a [subtype] of `T`, and
2246/// * `ptr::fn_addr_eq(f, g)` returns true,
2247///
2248/// then calling `f` and calling `g` will be equivalent.
2249///
2250///
2251/// # Examples
2252///
2253/// ```
2254/// use std::ptr;
2255///
2256/// fn a() { println!("a"); }
2257/// fn b() { println!("b"); }
2258/// assert!(!ptr::fn_addr_eq(a as fn(), b as fn()));
2259/// ```
2260///
2261/// [subtype]: https://doc.rust-lang.org/reference/subtyping.html
2262#[stable(feature = "ptr_fn_addr_eq", since = "1.85.0")]
2263#[inline(always)]
2264#[must_use = "function pointer comparison produces a value"]
2265#[cfg(not(feature = "ferrocene_certified"))]
2266pub fn fn_addr_eq<T: FnPtr, U: FnPtr>(f: T, g: U) -> bool {
2267    f.addr() == g.addr()
2268}

2355/// Creates a `const` raw pointer to a place, without creating an intermediate reference.
2356///
2357/// `addr_of!(expr)` is equivalent to `&raw const expr`. The macro is *soft-deprecated*;
2358/// use `&raw const` instead.
2359///
2360/// It is still an open question under which conditions writing through an `addr_of!`-created
2361/// pointer is permitted. If the place `expr` evaluates to is based on a raw pointer, then the
2362/// result of `addr_of!` inherits all permissions from that raw pointer. However, if the place is
2363/// based on a reference, local variable, or `static`, then until all details are decided, the same
2364/// rules as for shared references apply: it is UB to write through a pointer created with this
2365/// operation, except for bytes located inside an `UnsafeCell`. Use `&raw mut` (or [`addr_of_mut`])
2366/// to create a raw pointer that definitely permits mutation.
2367///
2368/// Creating a reference with `&`/`&mut` is only allowed if the pointer is properly aligned
2369/// and points to initialized data. For cases where those requirements do not hold,
2370/// raw pointers should be used instead. However, `&expr as *const _` creates a reference
2371/// before casting it to a raw pointer, and that reference is subject to the same rules
2372/// as all other references. This macro can create a raw pointer *without* creating
2373/// a reference first.
2374///
2375/// See [`addr_of_mut`] for how to create a pointer to uninitialized data.
2376/// Doing that with `addr_of` would not make much sense since one could only
2377/// read the data, and that would be Undefined Behavior.
2378///
2379/// # Safety
2380///
2381/// The `expr` in `addr_of!(expr)` is evaluated as a place expression, but never loads from the
2382/// place or requires the place to be dereferenceable. This means that `addr_of!((*ptr).field)`
2383/// still requires the projection to `field` to be in-bounds, using the same rules as [`offset`].
2384/// However, `addr_of!(*ptr)` is defined behavior even if `ptr` is null, dangling, or misaligned.
2385///
2386/// Note that `Deref`/`Index` coercions (and their mutable counterparts) are applied inside
2387/// `addr_of!` like everywhere else, in which case a reference is created to call `Deref::deref` or
2388/// `Index::index`, respectively. The statements above only apply when no such coercions are
2389/// applied.
2390///
2391/// [`offset`]: pointer::offset
2392///
2393/// # Example
2394///
2395/// **Correct usage: Creating a pointer to unaligned data**
2396///
2397/// ```
2398/// use std::ptr;
2399///
2400/// #[repr(packed)]
2401/// struct Packed {
2402///     f1: u8,
2403///     f2: u16,
2404/// }
2405///
2406/// let packed = Packed { f1: 1, f2: 2 };
2407/// // `&packed.f2` would create an unaligned reference, and thus be Undefined Behavior!
2408/// let raw_f2 = ptr::addr_of!(packed.f2);
2409/// assert_eq!(unsafe { raw_f2.read_unaligned() }, 2);
2410/// ```
2411///
2412/// **Incorrect usage: Out-of-bounds fields projection**
2413///
2414/// ```rust,no_run
2415/// use std::ptr;
2416///
2417/// #[repr(C)]
2418/// struct MyStruct {
2419///     field1: i32,
2420///     field2: i32,
2421/// }
2422///
2423/// let ptr: *const MyStruct = ptr::null();
2424/// let fieldptr = unsafe { ptr::addr_of!((*ptr).field2) }; // Undefined Behavior 
2425/// ```
2426///
2427/// The field projection `.field2` would offset the pointer by 4 bytes,
2428/// but the pointer is not in-bounds of an allocation for 4 bytes,
2429/// so this offset is Undefined Behavior.
2430/// See the [`offset`] docs for a full list of requirements for inbounds pointer arithmetic; the
2431/// same requirements apply to field projections, even inside `addr_of!`. (In particular, it makes
2432/// no difference whether the pointer is null or dangling.)
2433#[stable(feature = "raw_ref_macros", since = "1.51.0")]
2434#[rustc_macro_transparency = "semitransparent"]
2435#[cfg(not(feature = "ferrocene_certified"))]
2436pub macro addr_of($place:expr) {
2437    &raw const $place
2438}
