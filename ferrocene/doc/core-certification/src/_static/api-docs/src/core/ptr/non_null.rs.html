22/// `*mut T` but non-zero and [covariant].
23///
24/// This is often the correct thing to use when building data structures using
25/// raw pointers, but is ultimately more dangerous to use because of its additional
26/// properties. If you're not sure if you should use `NonNull<T>`, just use `*mut T`!
27///
28/// Unlike `*mut T`, the pointer must always be non-null, even if the pointer
29/// is never dereferenced. This is so that enums may use this forbidden value
30/// as a discriminant -- `Option<NonNull<T>>` has the same size as `*mut T`.
31/// However the pointer may still dangle if it isn't dereferenced.
32///
33/// Unlike `*mut T`, `NonNull<T>` was chosen to be covariant over `T`. This makes it
34/// possible to use `NonNull<T>` when building covariant types, but introduces the
35/// risk of unsoundness if used in a type that shouldn't actually be covariant.
36/// (The opposite choice was made for `*mut T` even though technically the unsoundness
37/// could only be caused by calling unsafe functions.)
38///
39/// Covariance is correct for most safe abstractions, such as `Box`, `Rc`, `Arc`, `Vec`,
40/// and `LinkedList`. This is the case because they provide a public API that follows the
41/// normal shared XOR mutable rules of Rust.
42///
43/// If your type cannot safely be covariant, you must ensure it contains some
44/// additional field to provide invariance. Often this field will be a [`PhantomData`]
45/// type like `PhantomData<Cell<T>>` or `PhantomData<&'a mut T>`.
46///
47/// Notice that `NonNull<T>` has a `From` instance for `&T`. However, this does
48/// not change the fact that mutating through a (pointer derived from a) shared
49/// reference is undefined behavior unless the mutation happens inside an
50/// [`UnsafeCell<T>`]. The same goes for creating a mutable reference from a shared
51/// reference. When using this `From` instance without an `UnsafeCell<T>`,
52/// it is your responsibility to ensure that `as_mut` is never called, and `as_ptr`
53/// is never used for mutation.
54///
55/// # Representation
56///
57/// Thanks to the [null pointer optimization],
58/// `NonNull<T>` and `Option<NonNull<T>>`
59/// are guaranteed to have the same size and alignment:
60///
61/// ```
62/// use std::ptr::NonNull;
63///
64/// assert_eq!(size_of::<NonNull<i16>>(), size_of::<Option<NonNull<i16>>>());
65/// assert_eq!(align_of::<NonNull<i16>>(), align_of::<Option<NonNull<i16>>>());
66///
67/// assert_eq!(size_of::<NonNull<str>>(), size_of::<Option<NonNull<str>>>());
68/// assert_eq!(align_of::<NonNull<str>>(), align_of::<Option<NonNull<str>>>());
69/// ```
70///
71/// [covariant]: https://doc.rust-lang.org/reference/subtyping.html
72/// [`PhantomData`]: crate::marker::PhantomData
73/// [`UnsafeCell<T>`]: crate::cell::UnsafeCell
74/// [null pointer optimization]: crate::option#representation
75#[stable(feature = "nonnull", since = "1.25.0")]
76#[repr(transparent)]
77#[rustc_layout_scalar_valid_range_start(1)]
78#[rustc_nonnull_optimization_guaranteed]
79#[rustc_diagnostic_item = "NonNull"]
80pub struct NonNull<T: ?Sized> {
81    // Remember to use `.as_ptr()` instead of `.pointer`, as field projecting to
82    // this is banned by <https://github.com/rust-lang/compiler-team/issues/807>.
83    pointer: *const T,
84}

503    /// Adds an offset to a pointer.
504    ///
505    /// `count` is in units of T; e.g., a `count` of 3 represents a pointer
506    /// offset of `3 * size_of::<T>()` bytes.
507    ///
508    /// # Safety
509    ///
510    /// If any of the following conditions are violated, the result is Undefined Behavior:
511    ///
512    /// * The computed offset, `count * size_of::<T>()` bytes, must not overflow `isize`.
513    ///
514    /// * If the computed offset is non-zero, then `self` must be derived from a pointer to some
515    ///   [allocated object], and the entire memory range between `self` and the result must be in
516    ///   bounds of that allocated object. In particular, this range must not "wrap around" the edge
517    ///   of the address space.
518    ///
519    /// Allocated objects can never be larger than `isize::MAX` bytes, so if the computed offset
520    /// stays in bounds of the allocated object, it is guaranteed to satisfy the first requirement.
521    /// This implies, for instance, that `vec.as_ptr().add(vec.len())` (for `vec: Vec<T>`) is always
522    /// safe.
523    ///
524    /// [allocated object]: crate::ptr#allocated-object
525    ///
526    /// # Examples
527    ///
528    /// ```
529    /// use std::ptr::NonNull;
530    ///
531    /// let mut s = [1, 2, 3];
532    /// let ptr: NonNull<u32> = NonNull::new(s.as_mut_ptr()).unwrap();
533    ///
534    /// unsafe {
535    ///     println!("{}", ptr.offset(1).read());
536    ///     println!("{}", ptr.offset(2).read());
537    /// }
538    /// ```
539    #[inline(always)]
540    #[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
541    #[must_use = "returns a new pointer rather than modifying its argument"]
542    #[stable(feature = "non_null_convenience", since = "1.80.0")]
543    #[rustc_const_stable(feature = "non_null_convenience", since = "1.80.0")]
544    pub const unsafe fn offset(self, count: isize) -> Self
545    where
546        T: Sized,
547    {
548        // SAFETY: the caller must uphold the safety contract for `offset`.
549        // Additionally safety contract of `offset` guarantees that the resulting pointer is

737    /// Calculates the distance between two pointers within the same allocation. The returned value is in
738    /// units of T: the distance in bytes divided by `size_of::<T>()`.
739    ///
740    /// This is equivalent to `(self as isize - origin as isize) / (size_of::<T>() as isize)`,
741    /// except that it has a lot more opportunities for UB, in exchange for the compiler
742    /// better understanding what you are doing.
743    ///
744    /// The primary motivation of this method is for computing the `len` of an array/slice
745    /// of `T` that you are currently representing as a "start" and "end" pointer
746    /// (and "end" is "one past the end" of the array).
747    /// In that case, `end.offset_from(start)` gets you the length of the array.
748    ///
749    /// All of the following safety requirements are trivially satisfied for this usecase.
750    ///
751    /// [`offset`]: #method.offset
752    ///
753    /// # Safety
754    ///
755    /// If any of the following conditions are violated, the result is Undefined Behavior:
756    ///
757    /// * `self` and `origin` must either
758    ///

760    ///   * both be *derived from* a pointer to the same [allocated object], and the memory range between
761    ///     the two pointers must be in bounds of that object. (See below for an example.)
762    ///
763    /// * The distance between the pointers, in bytes, must be an exact multiple
764    ///   of the size of `T`.

767    /// mathematical integers (without "wrapping around"), cannot overflow an `isize`. This is
768    /// implied by the in-bounds requirement, and the fact that no allocated object can be larger
769    /// than `isize::MAX` bytes.
770    ///
771    /// The requirement for pointers to be derived from the same allocated object is primarily
772    /// needed for `const`-compatibility: the distance between pointers into *different* allocated
773    /// objects is not known at compile-time. However, the requirement also exists at
774    /// runtime and may be exploited by optimizations. If you wish to compute the difference between
775    /// pointers that are not guaranteed to be from the same allocation, use `(self as isize -
776    /// origin as isize) / size_of::<T>()`.

779    /// [`add`]: #method.add
780    /// [allocated object]: crate::ptr#allocated-object

785    ///

1208    /// Computes the offset that needs to be applied to the pointer in order to make it aligned to
1209    /// `align`.
1210    ///
1211    /// If it is not possible to align the pointer, the implementation returns
1212    /// `usize::MAX`.
1213    ///
1214    /// The offset is expressed in number of `T` elements, and not bytes.
1215    ///
1216    /// There are no guarantees whatsoever that offsetting the pointer will not overflow or go
1217    /// beyond the allocation that the pointer points into. It is up to the caller to ensure that
1218    /// the returned offset is correct in all terms other than alignment.
1219    ///
1220    /// When this is called during compile-time evaluation (which is unstable), the implementation
1221    /// may return `usize::MAX` in cases where that can never happen at runtime. This is because the
1222    /// actual alignment of pointers is not known yet during compile-time, so an offset with
1223    /// guaranteed alignment can sometimes not be computed. For example, a buffer declared as `[u8;
1224    /// N]` might be allocated at an odd or an even address, but at compile-time this is not yet
1225    /// known, so the execution has to be correct for either choice. It is therefore impossible to
1226    /// find an offset that is guaranteed to be 2-aligned. (This behavior is subject to change, as usual
1227    /// for unstable APIs.)
1228    ///
1229    /// # Panics
1230    ///
1231    /// The function panics if `align` is not a power-of-two.
1232    ///
1233    /// # Examples
1234    ///
1235    /// Accessing adjacent `u8` as `u16`
1236    ///
1237    /// ```
1238    /// use std::ptr::NonNull;

1241    /// let x = [5_u8, 6, 7, 8, 9];
1242    /// let ptr = NonNull::new(x.as_ptr() as *mut u8).unwrap();
1243    /// let offset = ptr.align_offset(align_of::<u16>());
1244    ///
1245    /// if offset < x.len() - 1 {
1246    ///     let u16_ptr = ptr.add(offset).cast::<u16>();
1247    ///     assert!(u16_ptr.read() == u16::from_ne_bytes([5, 6]) || u16_ptr.read() == u16::from_ne_bytes([6, 7]));
1248    /// } else {
1249    ///     // while the pointer can be aligned via `offset`, it would point
1250    ///     // outside the allocation
1251    /// }
1252    /// # }
1253    /// ```
1254    #[inline]
