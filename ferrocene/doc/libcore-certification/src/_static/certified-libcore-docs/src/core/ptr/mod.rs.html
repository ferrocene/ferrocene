1//! Manually manage memory through raw pointers.
2//!
3//! *[See also the pointer primitive types](pointer).*
4//!
5//! # Safety
6//!
7//! Many functions in this module take raw pointers as arguments and read from or write to them. For
8//! this to be safe, these pointers must be *valid* for the given access. Whether a pointer is valid
9//! depends on the operation it is used for (read or write), and the extent of the memory that is
10//! accessed (i.e., how many bytes are read/written) -- it makes no sense to ask "is this pointer
11//! valid"; one has to ask "is this pointer valid for a given access". Most functions use `*mut T`
12//! and `*const T` to access only a single value, in which case the documentation omits the size and
13//! implicitly assumes it to be `size_of::<T>()` bytes.
14//!
15//! The precise rules for validity are not determined yet. The guarantees that are
16//! provided at this point are very minimal:
17//!
18//! * For memory accesses of [size zero][zst], *every* pointer is valid, including the [null]
19//!   pointer. The following points are only concerned with non-zero-sized accesses.
20//! * A [null] pointer is *never* valid.
21//! * For a pointer to be valid, it is necessary, but not always sufficient, that the pointer be
22//!   *dereferenceable*. The [provenance] of the pointer is used to determine which [allocation]
23//!   it is derived from; a pointer is dereferenceable if the memory range of the given size
24//!   starting at the pointer is entirely contained within the bounds of that allocation. Note
25//!   that in Rust, every (stack-allocated) variable is considered a separate allocation.
26//! * All accesses performed by functions in this module are *non-atomic* in the sense
27//!   of [atomic operations] used to synchronize between threads. This means it is
28//!   undefined behavior to perform two concurrent accesses to the same location from different
29//!   threads unless both accesses only read from memory. Notice that this explicitly
30//!   includes [`read_volatile`] and [`write_volatile`]: Volatile accesses cannot
31//!   be used for inter-thread synchronization.
32//! * The result of casting a reference to a pointer is valid for as long as the
33//!   underlying allocation is live and no reference (just raw pointers) is used to
34//!   access the same memory. That is, reference and pointer accesses cannot be
35//!   interleaved.
36//!
37//! These axioms, along with careful use of [`offset`] for pointer arithmetic,
38//! are enough to correctly implement many useful things in unsafe code. Stronger guarantees
39//! will be provided eventually, as the [aliasing] rules are being determined. For more
40//! information, see the [book] as well as the section in the reference devoted
41//! to [undefined behavior][ub].
42//!
43//! We say that a pointer is "dangling" if it is not valid for any non-zero-sized accesses. This
44//! means out-of-bounds pointers, pointers to freed memory, null pointers, and pointers created with
45//! [`NonNull::dangling`] are all dangling.
46//!
47//! ## Alignment
48//!
49//! Valid raw pointers as defined above are not necessarily properly aligned (where
50//! "proper" alignment is defined by the pointee type, i.e., `*const T` must be
51//! aligned to `align_of::<T>()`). However, most functions require their
52//! arguments to be properly aligned, and will explicitly state
53//! this requirement in their documentation. Notable exceptions to this are
54//! [`read_unaligned`] and [`write_unaligned`].
55//!
56//! When a function requires proper alignment, it does so even if the access
57//! has size 0, i.e., even if memory is not actually touched. Consider using
58//! [`NonNull::dangling`] in such cases.
59//!
60//! ## Pointer to reference conversion
61//!
62//! When converting a pointer to a reference (e.g. via `&*ptr` or `&mut *ptr`),
63//! there are several rules that must be followed:
64//!
65//! * The pointer must be properly aligned.
66//!
67//! * It must be non-null.
68//!
69//! * It must be "dereferenceable" in the sense defined above.
70//!
71//! * The pointer must point to a [valid value] of type `T`.
72//!
73//! * You must enforce Rust's aliasing rules. The exact aliasing rules are not decided yet, so we
74//!   only give a rough overview here. The rules also depend on whether a mutable or a shared
75//!   reference is being created.
76//!   * When creating a mutable reference, then while this reference exists, the memory it points to
77//!     must not get accessed (read or written) through any other pointer or reference not derived
78//!     from this reference.
79//!   * When creating a shared reference, then while this reference exists, the memory it points to
80//!     must not get mutated (except inside `UnsafeCell`).
81//!
82//! If a pointer follows all of these rules, it is said to be
83//! *convertible to a (mutable or shared) reference*.
84// ^ we use this term instead of saying that the produced reference must
85// be valid, as the validity of a reference is easily confused for the
86// validity of the thing it refers to, and while the two concepts are
87// closely related, they are not identical.
88//!
89//! These rules apply even if the result is unused!
90//! (The part about being initialized is not yet fully decided, but until
91//! it is, the only safe approach is to ensure that they are indeed initialized.)
92//!
93//! An example of the implications of the above rules is that an expression such
94//! as `unsafe { &*(0 as *const u8) }` is Immediate Undefined Behavior.
95//!
96//! [valid value]: ../../reference/behavior-considered-undefined.html#invalid-values

98//! ## Allocation
99//!
100//! <a id="allocated-object"></a> <!-- keep old URLs working -->
101//!
102//! An *allocation* is a subset of program memory which is addressable
103//! from Rust, and within which pointer arithmetic is possible. Examples of
104//! allocations include heap allocations, stack-allocated variables,
105//! statics, and consts. The safety preconditions of some Rust operations -
106//! such as `offset` and field projections (`expr.field`) - are defined in
107//! terms of the allocations on which they operate.
108//!
109//! An allocation has a base address, a size, and a set of memory
110//! addresses. It is possible for an allocation to have zero size, but
111//! such an allocation will still have a base address. The base address
112//! of an allocation is not necessarily unique. While it is currently the
113//! case that an allocation always has a set of memory addresses which is
114//! fully contiguous (i.e., has no "holes"), there is no guarantee that this
115//! will not change in the future.
116//!
117//! For any allocation with `base` address, `size`, and a set of
118//! `addresses`, the following are guaranteed:
119//! - For all addresses `a` in `addresses`, `a` is in the range `base .. (base +
120//!   size)` (note that this requires `a < base + size`, not `a <= base + size`)
121//! - `base` is not equal to [`null()`] (i.e., the address with the numerical
122//!   value 0)
123//! - `base + size <= usize::MAX`
124//! - `size <= isize::MAX`
125//!
126//! As a consequence of these guarantees, given any address `a` within the set
127//! of addresses of an allocation:
128//! - It is guaranteed that `a - base` does not overflow `isize`
129//! - It is guaranteed that `a - base` is non-negative
130//! - It is guaranteed that, given `o = a - base` (i.e., the offset of `a` within
131//!   the allocation), `base + o` will not wrap around the address space (in
132//!   other words, will not overflow `usize`)
133//!
134//! [`null()`]: null
135//!
136//! # Provenance
137//!
138//! Pointers are not *simply* an "integer" or "address". For instance, it's uncontroversial
139//! to say that a Use After Free is clearly Undefined Behavior, even if you "get lucky"
140//! and the freed memory gets reallocated before your read/write (in fact this is the
141//! worst-case scenario, UAFs would be much less concerning if this didn't happen!).
142//! As another example, consider that [`wrapping_offset`] is documented to "remember"
143//! the allocation that the original pointer points to, even if it is offset far
144//! outside the memory range occupied by that allocation.
145//! To rationalize claims like this, pointers need to somehow be *more* than just their addresses:
146//! they must have **provenance**.
147//!
148//! A pointer value in Rust semantically contains the following information:
149//!
150//! * The **address** it points to, which can be represented by a `usize`.
151//! * The **provenance** it has, defining the memory it has permission to access. Provenance can be
152//!   absent, in which case the pointer does not have permission to access any memory.
153//!
154//! The exact structure of provenance is not yet specified, but the permission defined by a
155//! pointer's provenance have a *spatial* component, a *temporal* component, and a *mutability*
156//! component:
157//!
158//! * Spatial: The set of memory addresses that the pointer is allowed to access.
159//! * Temporal: The timespan during which the pointer is allowed to access those memory addresses.
160//! * Mutability: Whether the pointer may only access the memory for reads, or also access it for
161//!   writes. Note that this can interact with the other components, e.g. a pointer might permit
162//!   mutation only for a subset of addresses, or only for a subset of its maximal timespan.
163//!
164//! When an [allocation] is created, it has a unique Original Pointer. For alloc
165//! APIs this is literally the pointer the call returns, and for local variables and statics,
166//! this is the name of the variable/static. (This is mildly overloading the term "pointer"
167//! for the sake of brevity/exposition.)
168//!
169//! The Original Pointer for an allocation has provenance that constrains the *spatial*
170//! permissions of this pointer to the memory range of the allocation, and the *temporal*
171//! permissions to the lifetime of the allocation. Provenance is implicitly inherited by all
172//! pointers transitively derived from the Original Pointer through operations like [`offset`],
173//! borrowing, and pointer casts. Some operations may *shrink* the permissions of the derived
174//! provenance, limiting how much memory it can access or how long it's valid for (i.e. borrowing a
175//! subfield and subslicing can shrink the spatial component of provenance, and all borrowing can
176//! shrink the temporal component of provenance). However, no operation can ever *grow* the
177//! permissions of the derived provenance: even if you "know" there is a larger allocation, you
178//! can't derive a pointer with a larger provenance. Similarly, you cannot "recombine" two
179//! contiguous provenances back into one (i.e. with a `fn merge(&[T], &[T]) -> &[T]`).
180//!
181//! A reference to a place always has provenance over at least the memory that place occupies.
182//! A reference to a slice always has provenance over at least the range that slice describes.
183//! Whether and when exactly the provenance of a reference gets "shrunk" to *exactly* fit
184//! the memory it points to is not yet determined.
185//!
186//! A *shared* reference only ever has provenance that permits reading from memory,
187//! and never permits writes, except inside [`UnsafeCell`].
188//!
189//! Provenance can affect whether a program has undefined behavior:
190//!
191//! * It is undefined behavior to access memory through a pointer that does not have provenance over
192//!   that memory. Note that a pointer "at the end" of its provenance is not actually outside its
193//!   provenance, it just has 0 bytes it can load/store. Zero-sized accesses do not require any
194//!   provenance since they access an empty range of memory.
195//!
196//! * It is undefined behavior to [`offset`] a pointer across a memory range that is not contained
197//!   in the allocation it is derived from, or to [`offset_from`] two pointers not derived
198//!   from the same allocation. Provenance is used to say what exactly "derived from" even
199//!   means: the lineage of a pointer is traced back to the Original Pointer it descends from, and
200//!   that identifies the relevant allocation. In particular, it's always UB to offset a
201//!   pointer derived from something that is now deallocated, except if the offset is 0.
202//!
203//! But it *is* still sound to:
204//!
205//! * Create a pointer without provenance from just an address (see [`without_provenance`]). Such a
206//!   pointer cannot be used for memory accesses (except for zero-sized accesses). This can still be
207//!   useful for sentinel values like `null` *or* to represent a tagged pointer that will never be
208//!   dereferenceable. In general, it is always sound for an integer to pretend to be a pointer "for
209//!   fun" as long as you don't use operations on it which require it to be valid (non-zero-sized
210//!   offset, read, write, etc).
211//!
212//! * Forge an allocation of size zero at any sufficiently aligned non-null address.
213//!   i.e. the usual "ZSTs are fake, do what you want" rules apply.
214//!
215//! * [`wrapping_offset`] a pointer outside its provenance. This includes pointers
216//!   which have "no" provenance. In particular, this makes it sound to do pointer tagging tricks.
217//!
218//! * Compare arbitrary pointers by address. Pointer comparison ignores provenance and addresses
219//!   *are* just integers, so there is always a coherent answer, even if the pointers are dangling
220//!   or from different provenances. Note that if you get "lucky" and notice that a pointer at the
221//!   end of one allocation is the "same" address as the start of another allocation,
222//!   anything you do with that fact is *probably* going to be gibberish. The scope of that
223//!   gibberish is kept under control by the fact that the two pointers *still* aren't allowed to
224//!   access the other's allocation (bytes), because they still have different provenance.
225//!
226//! Note that the full definition of provenance in Rust is not decided yet, as this interacts
227//! with the as-yet undecided [aliasing] rules.
228//!
229//! ## Pointers Vs Integers
230//!
231//! From this discussion, it becomes very clear that a `usize` *cannot* accurately represent a pointer,
232//! and converting from a pointer to a `usize` is generally an operation which *only* extracts the
233//! address. Converting this address back into pointer requires somehow answering the question:
234//! which provenance should the resulting pointer have?
235//!
236//! Rust provides two ways of dealing with this situation: *Strict Provenance* and *Exposed Provenance*.
237//!
238//! Note that a pointer *can* represent a `usize` (via [`without_provenance`]), so the right type to
239//! use in situations where a value is "sometimes a pointer and sometimes a bare `usize`" is a
240//! pointer type.
241//!
242//! ## Strict Provenance
243//!
244//! "Strict Provenance" refers to a set of APIs designed to make working with provenance more
245//! explicit. They are intended as substitutes for casting a pointer to an integer and back.
246//!
247//! Entirely avoiding integer-to-pointer casts successfully side-steps the inherent ambiguity of
248//! that operation. This benefits compiler optimizations, and it is pretty much a requirement for
249//! using tools like [Miri] and architectures like [CHERI] that aim to detect and diagnose pointer
250//! misuse.
251//!
252//! The key insight to making programming without integer-to-pointer casts *at all* viable is the
253//! [`with_addr`] method:
254//!
255//! ```text
256//!     /// Creates a new pointer with the given address.
257//!     ///
258//!     /// This performs the same operation as an `addr as ptr` cast, but copies
259//!     /// the *provenance* of `self` to the new pointer.
260//!     /// This allows us to dynamically preserve and propagate this important
261//!     /// information in a way that is otherwise impossible with a unary cast.
262//!     ///
263//!     /// This is equivalent to using `wrapping_offset` to offset `self` to the
264//!     /// given address, and therefore has all the same capabilities and restrictions.
265//!     pub fn with_addr(self, addr: usize) -> Self;
266//! ```
267//!
268//! So you're still able to drop down to the address representation and do whatever
269//! clever bit tricks you want *as long as* you're able to keep around a pointer
270//! into the allocation you care about that can "reconstitute" the provenance.
271//! Usually this is very easy, because you only are taking a pointer, messing with the address,
272//! and then immediately converting back to a pointer. To make this use case more ergonomic,
273//! we provide the [`map_addr`] method.
274//!
275//! To help make it clear that code is "following" Strict Provenance semantics, we also provide an
276//! [`addr`] method which promises that the returned address is not part of a
277//! pointer-integer-pointer roundtrip. In the future we may provide a lint for pointer<->integer
278//! casts to help you audit if your code conforms to strict provenance.
279//!
280//! ### Using Strict Provenance
281//!
282//! Most code needs no changes to conform to strict provenance, as the only really concerning
283//! operation is casts from `usize` to a pointer. For code which *does* cast a `usize` to a pointer,
284//! the scope of the change depends on exactly what you're doing.
285//!
286//! In general, you just need to make sure that if you want to convert a `usize` address to a
287//! pointer and then use that pointer to read/write memory, you need to keep around a pointer
288//! that has sufficient provenance to perform that read/write itself. In this way all of your
289//! casts from an address to a pointer are essentially just applying offsets/indexing.
290//!
291//! This is generally trivial to do for simple cases like tagged pointers *as long as you
292//! represent the tagged pointer as an actual pointer and not a `usize`*. For instance:
293//!
294//! ```
295//! unsafe {
296//!     // A flag we want to pack into our pointer
297//!     static HAS_DATA: usize = 0x1;
298//!     static FLAG_MASK: usize = !HAS_DATA;
299//!
300//!     // Our value, which must have enough alignment to have spare least-significant-bits.
301//!     let my_precious_data: u32 = 17;
302//!     assert!(align_of::<u32>() > 1);
303//!
304//!     // Create a tagged pointer
305//!     let ptr = &my_precious_data as *const u32;
306//!     let tagged = ptr.map_addr(|addr| addr | HAS_DATA);
307//!
308//!     // Check the flag:
309//!     if tagged.addr() & HAS_DATA != 0 {
310//!         // Untag and read the pointer
311//!         let data = *tagged.map_addr(|addr| addr & FLAG_MASK);
312//!         assert_eq!(data, 17);
313//!     } else {
314//!         unreachable!()
315//!     }
316//! }
317//! ```
318//!
319//! (Yes, if you've been using [`AtomicUsize`] for pointers in concurrent datastructures, you should
320//! be using [`AtomicPtr`] instead. If that messes up the way you atomically manipulate pointers,
321//! we would like to know why, and what needs to be done to fix it.)
322//!
323//! Situations where a valid pointer *must* be created from just an address, such as baremetal code
324//! accessing a memory-mapped interface at a fixed address, cannot currently be handled with strict
325//! provenance APIs and should use [exposed provenance](#exposed-provenance).
326//!
327//! ## Exposed Provenance
328//!
329//! As discussed above, integer-to-pointer casts are not possible with Strict Provenance APIs.
330//! This is by design: the goal of Strict Provenance is to provide a clear specification that we are
331//! confident can be formalized unambiguously and can be subject to precise formal reasoning.
332//! Integer-to-pointer casts do not (currently) have such a clear specification.
333//!
334//! However, there exist situations where integer-to-pointer casts cannot be avoided, or
335//! where avoiding them would require major refactoring. Legacy platform APIs also regularly assume
336//! that `usize` can capture all the information that makes up a pointer.
337//! Bare-metal platforms can also require the synthesis of a pointer "out of thin air" without
338//! anywhere to obtain proper provenance from.
339//!
340//! Rust's model for dealing with integer-to-pointer casts is called *Exposed Provenance*. However,
341//! the semantics of Exposed Provenance are on much less solid footing than Strict Provenance, and
342//! at this point it is not yet clear whether a satisfying unambiguous semantics can be defined for
343//! Exposed Provenance. (If that sounds bad, be reassured that other popular languages that provide
344//! integer-to-pointer casts are not faring any better.) Furthermore, Exposed Provenance will not
345//! work (well) with tools like [Miri] and [CHERI].
346//!
347//! Exposed Provenance is provided by the [`expose_provenance`] and [`with_exposed_provenance`] methods,
348//! which are equivalent to `as` casts between pointers and integers.
349//! - [`expose_provenance`] is a lot like [`addr`], but additionally adds the provenance of the
350//!   pointer to a global list of 'exposed' provenances. (This list is purely conceptual, it exists
351//!   for the purpose of specifying Rust but is not materialized in actual executions, except in
352//!   tools like [Miri].)
353//!   Memory which is outside the control of the Rust abstract machine (MMIO registers, for example)
354//!   is always considered to be exposed, so long as this memory is disjoint from memory that will
355//!   be used by the abstract machine such as the stack, heap, and statics.
356//! - [`with_exposed_provenance`] can be used to construct a pointer with one of these previously
357//!   'exposed' provenances. [`with_exposed_provenance`] takes only `addr: usize` as arguments, so
358//!   unlike in [`with_addr`] there is no indication of what the correct provenance for the returned
359//!   pointer is -- and that is exactly what makes integer-to-pointer casts so tricky to rigorously
360//!   specify! The compiler will do its best to pick the right provenance for you, but currently we
361//!   cannot provide any guarantees about which provenance the resulting pointer will have. Only one
362//!   thing is clear: if there is *no* previously 'exposed' provenance that justifies the way the
363//!   returned pointer will be used, the program has undefined behavior.
364//!
365//! If at all possible, we encourage code to be ported to [Strict Provenance] APIs, thus avoiding
366//! the need for Exposed Provenance. Maximizing the amount of such code is a major win for avoiding
367//! specification complexity and to facilitate adoption of tools like [CHERI] and [Miri] that can be
368//! a big help in increasing the confidence in (unsafe) Rust code. However, we acknowledge that this
369//! is not always possible, and offer Exposed Provenance as a way to explicit "opt out" of the
370//! well-defined semantics of Strict Provenance, and "opt in" to the unclear semantics of
371//! integer-to-pointer casts.

450/// Copies `count * size_of::<T>()` bytes from `src` to `dst`. The source
451/// and destination must *not* overlap.
452///
453/// For regions of memory which might overlap, use [`copy`] instead.
454///
455/// `copy_nonoverlapping` is semantically equivalent to C's [`memcpy`], but
456/// with the source and destination arguments swapped,
457/// and `count` counting the number of `T`s instead of bytes.
458///
459/// The copy is "untyped" in the sense that data may be uninitialized or otherwise violate the
460/// requirements of `T`. The initialization state is preserved exactly.
461///
462/// [`memcpy`]: https://en.cppreference.com/w/c/string/byte/memcpy
463///
464/// # Safety
465///
466/// Behavior is undefined if any of the following conditions are violated:
467///
468/// * `src` must be [valid] for reads of `count * size_of::<T>()` bytes.
469///
470/// * `dst` must be [valid] for writes of `count * size_of::<T>()` bytes.
471///
472/// * Both `src` and `dst` must be properly aligned.
473///
474/// * The region of memory beginning at `src` with a size of `count *
475///   size_of::<T>()` bytes must *not* overlap with the region of memory
476///   beginning at `dst` with the same size.
477///
478/// Like [`read`], `copy_nonoverlapping` creates a bitwise copy of `T`, regardless of
479/// whether `T` is [`Copy`]. If `T` is not [`Copy`], using *both* the values
480/// in the region beginning at `*src` and the region beginning at `*dst` can
481/// [violate memory safety][read-ownership].
482///
483/// Note that even if the effectively copied size (`count * size_of::<T>()`) is
484/// `0`, the pointers must be properly aligned.
485///
486/// [`read`]: crate::ptr::read
487/// [read-ownership]: crate::ptr::read#ownership-of-the-returned-value
488/// [valid]: crate::ptr#safety
489///
490/// # Examples
491///
492/// Manually implement [`Vec::append`]:
493///
494/// ```
495/// use std::ptr;
496///
497/// /// Moves all the elements of `src` into `dst`, leaving `src` empty.
498/// fn append<T>(dst: &mut Vec<T>, src: &mut Vec<T>) {
499///     let src_len = src.len();
500///     let dst_len = dst.len();
501///
502///     // Ensure that `dst` has enough capacity to hold all of `src`.
503///     dst.reserve(src_len);
504///
505///     unsafe {
506///         // The call to add is always safe because `Vec` will never
507///         // allocate more than `isize::MAX` bytes.
508///         let dst_ptr = dst.as_mut_ptr().add(dst_len);
509///         let src_ptr = src.as_ptr();
510///
511///         // Truncate `src` without dropping its contents. We do this first,
512///         // to avoid problems in case something further down panics.
513///         src.set_len(0);
514///
515///         // The two regions cannot overlap because mutable references do
516///         // not alias, and two different vectors cannot own the same
517///         // memory.
518///         ptr::copy_nonoverlapping(src_ptr, dst_ptr, src_len);
519///
520///         // Notify `dst` that it now holds the contents of `src`.
521///         dst.set_len(dst_len + src_len);
522///     }
523/// }
524///
525/// let mut a = vec!['r'];
526/// let mut b = vec!['u', 's', 't'];
527///
528/// append(&mut a, &mut b);
529///
530/// assert_eq!(a, &['r', 'u', 's', 't']);
531/// assert!(b.is_empty());
532/// ```
533///
534/// [`Vec::append`]: ../../std/vec/struct.Vec.html#method.append
535#[doc(alias = "memcpy")]
536#[stable(feature = "rust1", since = "1.0.0")]
537#[rustc_const_stable(feature = "const_intrinsic_copy", since = "1.83.0")]
538#[inline(always)]
539#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
540#[rustc_diagnostic_item = "ptr_copy_nonoverlapping"]
541#[cfg(not(feature = "ferrocene_certified"))]
542pub const unsafe fn copy_nonoverlapping<T>(src: *const T, dst: *mut T, count: usize) {
543    ub_checks::assert_unsafe_precondition!(
544        check_language_ub,
545        "ptr::copy_nonoverlapping requires that both pointer arguments are aligned and non-null \
546        and the specified memory ranges do not overlap",
547        (
548            src: *const () = src as *const (),
549            dst: *mut () = dst as *mut (),
550            size: usize = size_of::<T>(),
551            align: usize = align_of::<T>(),
552            count: usize = count,
553        ) => {
554            let zero_size = count == 0 || size == 0;
555            ub_checks::maybe_is_aligned_and_not_null(src, align, zero_size)
556                && ub_checks::maybe_is_aligned_and_not_null(dst, align, zero_size)
557                && ub_checks::maybe_is_nonoverlapping(src, dst, size, count)
558        }
559    );
560
561    // SAFETY: the safety contract for `copy_nonoverlapping` must be
562    // upheld by the caller.
563    unsafe { crate::intrinsics::copy_nonoverlapping(src, dst, count) }
564}

566/// Copies `count * size_of::<T>()` bytes from `src` to `dst`. The source
567/// and destination may overlap.
568///
569/// If the source and destination will *never* overlap,
570/// [`copy_nonoverlapping`] can be used instead.
571///
572/// `copy` is semantically equivalent to C's [`memmove`], but
573/// with the source and destination arguments swapped,
574/// and `count` counting the number of `T`s instead of bytes.
575/// Copying takes place as if the bytes were copied from `src`
576/// to a temporary array and then copied from the array to `dst`.
577///
578/// The copy is "untyped" in the sense that data may be uninitialized or otherwise violate the
579/// requirements of `T`. The initialization state is preserved exactly.
580///
581/// [`memmove`]: https://en.cppreference.com/w/c/string/byte/memmove
582///
583/// # Safety
584///
585/// Behavior is undefined if any of the following conditions are violated:
586///
587/// * `src` must be [valid] for reads of `count * size_of::<T>()` bytes.
588///
589/// * `dst` must be [valid] for writes of `count * size_of::<T>()` bytes, and must remain valid even
590///   when `src` is read for `count * size_of::<T>()` bytes. (This means if the memory ranges
591///   overlap, the `dst` pointer must not be invalidated by `src` reads.)
592///
593/// * Both `src` and `dst` must be properly aligned.
594///
595/// Like [`read`], `copy` creates a bitwise copy of `T`, regardless of
596/// whether `T` is [`Copy`]. If `T` is not [`Copy`], using both the values
597/// in the region beginning at `*src` and the region beginning at `*dst` can
598/// [violate memory safety][read-ownership].
599///
600/// Note that even if the effectively copied size (`count * size_of::<T>()`) is
601/// `0`, the pointers must be properly aligned.
602///
603/// [`read`]: crate::ptr::read
604/// [read-ownership]: crate::ptr::read#ownership-of-the-returned-value
605/// [valid]: crate::ptr#safety
606///
607/// # Examples
608///
609/// Efficiently create a Rust vector from an unsafe buffer:
610///
611/// ```
612/// use std::ptr;
613///
614/// /// # Safety
615/// ///
616/// /// * `ptr` must be correctly aligned for its type and non-zero.
617/// /// * `ptr` must be valid for reads of `elts` contiguous elements of type `T`.
618/// /// * Those elements must not be used after calling this function unless `T: Copy`.
619/// # #[allow(dead_code)]
620/// unsafe fn from_buf_raw<T>(ptr: *const T, elts: usize) -> Vec<T> {
621///     let mut dst = Vec::with_capacity(elts);
622///
623///     // SAFETY: Our precondition ensures the source is aligned and valid,
624///     // and `Vec::with_capacity` ensures that we have usable space to write them.
625///     unsafe { ptr::copy(ptr, dst.as_mut_ptr(), elts); }
626///
627///     // SAFETY: We created it with this much capacity earlier,
628///     // and the previous `copy` has initialized these elements.
629///     unsafe { dst.set_len(elts); }
630///     dst
631/// }
632/// ```
633#[doc(alias = "memmove")]
634#[stable(feature = "rust1", since = "1.0.0")]
635#[rustc_const_stable(feature = "const_intrinsic_copy", since = "1.83.0")]
636#[inline(always)]
637#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
638#[rustc_diagnostic_item = "ptr_copy"]
639#[cfg(not(feature = "ferrocene_certified"))]
640pub const unsafe fn copy<T>(src: *const T, dst: *mut T, count: usize) {
641    // SAFETY: the safety contract for `copy` must be upheld by the caller.
642    unsafe {
643        ub_checks::assert_unsafe_precondition!(
644            check_language_ub,
645            "ptr::copy requires that both pointer arguments are aligned and non-null",
646            (
647                src: *const () = src as *const (),
648                dst: *mut () = dst as *mut (),
649                align: usize = align_of::<T>(),
650                zero_size: bool = T::IS_ZST || count == 0,
651            ) =>
652            ub_checks::maybe_is_aligned_and_not_null(src, align, zero_size)
653                && ub_checks::maybe_is_aligned_and_not_null(dst, align, zero_size)
654        );
655        crate::intrinsics::copy(src, dst, count)
656    }
657}

659/// Sets `count * size_of::<T>()` bytes of memory starting at `dst` to
660/// `val`.
661///
662/// `write_bytes` is similar to C's [`memset`], but sets `count *
663/// size_of::<T>()` bytes to `val`.
664///
665/// [`memset`]: https://en.cppreference.com/w/c/string/byte/memset
666///
667/// # Safety
668///
669/// Behavior is undefined if any of the following conditions are violated:
670///
671/// * `dst` must be [valid] for writes of `count * size_of::<T>()` bytes.
672///
673/// * `dst` must be properly aligned.
674///
675/// Note that even if the effectively copied size (`count * size_of::<T>()`) is
676/// `0`, the pointer must be properly aligned.
677///
678/// Additionally, note that changing `*dst` in this way can easily lead to undefined behavior (UB)
679/// later if the written bytes are not a valid representation of some `T`. For instance, the
680/// following is an **incorrect** use of this function:
681///
682/// ```rust,no_run
683/// unsafe {
684///     let mut value: u8 = 0;
685///     let ptr: *mut bool = &mut value as *mut u8 as *mut bool;
686///     let _bool = ptr.read(); // This is fine, `ptr` points to a valid `bool`.
687///     ptr.write_bytes(42u8, 1); // This function itself does not cause UB...
688///     let _bool = ptr.read(); // ...but it makes this operation UB! ⚠️
689/// }
690/// ```
691///
692/// [valid]: crate::ptr#safety
693///
694/// # Examples
695///
696/// Basic usage:
697///
698/// ```
699/// use std::ptr;
700///
701/// let mut vec = vec![0u32; 4];
702/// unsafe {
703///     let vec_ptr = vec.as_mut_ptr();
704///     ptr::write_bytes(vec_ptr, 0xfe, 2);
705/// }
706/// assert_eq!(vec, [0xfefefefe, 0xfefefefe, 0, 0]);
707/// ```
708#[doc(alias = "memset")]
709#[stable(feature = "rust1", since = "1.0.0")]
710#[rustc_const_stable(feature = "const_ptr_write", since = "1.83.0")]
711#[inline(always)]
712#[cfg_attr(miri, track_caller)] // even without panics, this helps for Miri backtraces
713#[rustc_diagnostic_item = "ptr_write_bytes"]
714#[cfg(not(feature = "ferrocene_certified"))]
715pub const unsafe fn write_bytes<T>(dst: *mut T, val: u8, count: usize) {
716    // SAFETY: the safety contract for `write_bytes` must be upheld by the caller.
717    unsafe {
718        ub_checks::assert_unsafe_precondition!(
719            check_language_ub,
720            "ptr::write_bytes requires that the destination pointer is aligned and non-null",
721            (
722                addr: *const () = dst as *const (),
723                align: usize = align_of::<T>(),
724                zero_size: bool = T::IS_ZST || count == 0,
725            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, zero_size)
726        );
727        crate::intrinsics::write_bytes(dst, val, count)
728    }

1046/// Converts a reference to a raw pointer.
1047///
1048/// For `r: &T`, `from_ref(r)` is equivalent to `r as *const T` (except for the caveat noted below),
1049/// but is a bit safer since it will never silently change type or mutability, in particular if the
1050/// code is refactored.
1051///
1052/// The caller must ensure that the pointee outlives the pointer this function returns, or else it
1053/// will end up dangling.
1054///
1055/// The caller must also ensure that the memory the pointer (non-transitively) points to is never
1056/// written to (except inside an `UnsafeCell`) using this pointer or any pointer derived from it. If
1057/// you need to mutate the pointee, use [`from_mut`]. Specifically, to turn a mutable reference `m:
1058/// &mut T` into `*const T`, prefer `from_mut(m).cast_const()` to obtain a pointer that can later be
1059/// used for mutation.
1060///
1061/// ## Interaction with lifetime extension
1062///
1063/// Note that this has subtle interactions with the rules for lifetime extension of temporaries in
1064/// tail expressions. This code is valid, albeit in a non-obvious way:
1065/// ```rust
1066/// # type T = i32;
1067/// # fn foo() -> T { 42 }
1068/// // The temporary holding the return value of `foo` has its lifetime extended,
1069/// // because the surrounding expression involves no function call.
1070/// let p = &foo() as *const T;
1071/// unsafe { p.read() };
1072/// ```
1073/// Naively replacing the cast with `from_ref` is not valid:
1074/// ```rust,no_run
1075/// # use std::ptr;
1076/// # type T = i32;
1077/// # fn foo() -> T { 42 }
1078/// // The temporary holding the return value of `foo` does *not* have its lifetime extended,
1079/// // because the surrounding expression involves a function call.
1080/// let p = ptr::from_ref(&foo());
1081/// unsafe { p.read() }; // UB! Reading from a dangling pointer ⚠️
1082/// ```
1083/// The recommended way to write this code is to avoid relying on lifetime extension
1084/// when raw pointers are involved:
1085/// ```rust
1086/// # use std::ptr;
1087/// # type T = i32;
1088/// # fn foo() -> T { 42 }
1089/// let x = foo();
1090/// let p = ptr::from_ref(&x);
1091/// unsafe { p.read() };
1092/// ```

1094#[must_use]
1095#[stable(feature = "ptr_from_ref", since = "1.76.0")]
1096#[rustc_const_stable(feature = "ptr_from_ref", since = "1.76.0")]
1097#[rustc_never_returns_null_ptr]
1098#[rustc_diagnostic_item = "ptr_from_ref"]
1099#[cfg(not(feature = "ferrocene_certified"))]
1100pub const fn from_ref<T: ?Sized>(r: &T) -> *const T {
1101    r
1102}

1242/// Swaps the values at two mutable locations of the same type, without
1243/// deinitializing either.
1244///
1245/// But for the following exceptions, this function is semantically
1246/// equivalent to [`mem::swap`]:
1247///
1248/// * It operates on raw pointers instead of references. When references are
1249///   available, [`mem::swap`] should be preferred.
1250///
1251/// * The two pointed-to values may overlap. If the values do overlap, then the
1252///   overlapping region of memory from `x` will be used. This is demonstrated
1253///   in the second example below.
1254///
1255/// * The operation is "untyped" in the sense that data may be uninitialized or otherwise violate
1256///   the requirements of `T`. The initialization state is preserved exactly.
1257///
1258/// # Safety
1259///
1260/// Behavior is undefined if any of the following conditions are violated:
1261///
1262/// * Both `x` and `y` must be [valid] for both reads and writes. They must remain valid even when the
1263///   other pointer is written. (This means if the memory ranges overlap, the two pointers must not
1264///   be subject to aliasing restrictions relative to each other.)
1265///
1266/// * Both `x` and `y` must be properly aligned.
1267///
1268/// Note that even if `T` has size `0`, the pointers must be properly aligned.
1269///
1270/// [valid]: self#safety
1271///
1272/// # Examples
1273///
1274/// Swapping two non-overlapping regions:
1275///
1276/// ```
1277/// use std::ptr;
1278///
1279/// let mut array = [0, 1, 2, 3];
1280///
1281/// let (x, y) = array.split_at_mut(2);
1282/// let x = x.as_mut_ptr().cast::<[u32; 2]>(); // this is `array[0..2]`
1283/// let y = y.as_mut_ptr().cast::<[u32; 2]>(); // this is `array[2..4]`
1284///
1285/// unsafe {
1286///     ptr::swap(x, y);
1287///     assert_eq!([2, 3, 0, 1], array);
1288/// }
1289/// ```
1290///
1291/// Swapping two overlapping regions:
1292///
1293/// ```
1294/// use std::ptr;
1295///
1296/// let mut array: [i32; 4] = [0, 1, 2, 3];
1297///
1298/// let array_ptr: *mut i32 = array.as_mut_ptr();
1299///
1300/// let x = array_ptr as *mut [i32; 3]; // this is `array[0..3]`
1301/// let y = unsafe { array_ptr.add(1) } as *mut [i32; 3]; // this is `array[1..4]`
1302///
1303/// unsafe {
1304///     ptr::swap(x, y);
1305///     // The indices `1..3` of the slice overlap between `x` and `y`.
1306///     // Reasonable results would be for to them be `[2, 3]`, so that indices `0..3` are
1307///     // `[1, 2, 3]` (matching `y` before the `swap`); or for them to be `[0, 1]`
1308///     // so that indices `1..4` are `[0, 1, 2]` (matching `x` before the `swap`).
1309///     // This implementation is defined to make the latter choice.
1310///     assert_eq!([1, 0, 1, 2], array);
1311/// }
1312/// ```
1313#[inline]
1314#[stable(feature = "rust1", since = "1.0.0")]
1315#[rustc_const_stable(feature = "const_swap", since = "1.85.0")]
1316#[rustc_diagnostic_item = "ptr_swap"]
1317#[cfg(not(feature = "ferrocene_certified"))]
1318pub const unsafe fn swap<T>(x: *mut T, y: *mut T) {
1319    // Give ourselves some scratch space to work with.
1320    // We do not have to worry about drops: `MaybeUninit` does nothing when dropped.
1321    let mut tmp = MaybeUninit::<T>::uninit();
1322
1323    // Perform the swap
1324    // SAFETY: the caller must guarantee that `x` and `y` are
1325    // valid for writes and properly aligned. `tmp` cannot be
1326    // overlapping either `x` or `y` because `tmp` was just allocated
1327    // on the stack as a separate allocation.
1328    unsafe {
1329        copy_nonoverlapping(x, tmp.as_mut_ptr(), 1);
1330        copy(y, x, 1); // `x` and `y` may overlap
1331        copy_nonoverlapping(tmp.as_ptr(), y, 1);
1332    }
1333}

1335/// Swaps `count * size_of::<T>()` bytes between the two regions of memory
1336/// beginning at `x` and `y`. The two regions must *not* overlap.
1337///
1338/// The operation is "untyped" in the sense that data may be uninitialized or otherwise violate the
1339/// requirements of `T`. The initialization state is preserved exactly.
1340///
1341/// # Safety
1342///
1343/// Behavior is undefined if any of the following conditions are violated:
1344///
1345/// * Both `x` and `y` must be [valid] for both reads and writes of `count *
1346///   size_of::<T>()` bytes.
1347///
1348/// * Both `x` and `y` must be properly aligned.
1349///
1350/// * The region of memory beginning at `x` with a size of `count *
1351///   size_of::<T>()` bytes must *not* overlap with the region of memory
1352///   beginning at `y` with the same size.
1353///
1354/// Note that even if the effectively copied size (`count * size_of::<T>()`) is `0`,
1355/// the pointers must be properly aligned.
1356///
1357/// [valid]: self#safety
1358///
1359/// # Examples
1360///
1361/// Basic usage:
1362///
1363/// ```
1364/// use std::ptr;
1365///
1366/// let mut x = [1, 2, 3, 4];
1367/// let mut y = [7, 8, 9];
1368///
1369/// unsafe {
1370///     ptr::swap_nonoverlapping(x.as_mut_ptr(), y.as_mut_ptr(), 2);
1371/// }
1372///
1373/// assert_eq!(x, [7, 8, 3, 4]);
1374/// assert_eq!(y, [1, 2, 9]);
1375/// ```
1376///
1377/// # Const evaluation limitations
1378///
1379/// If this function is invoked during const-evaluation, the current implementation has a small (and
1380/// rarely relevant) limitation: if `count` is at least 2 and the data pointed to by `x` or `y`
1381/// contains a pointer that crosses the boundary of two `T`-sized chunks of memory, the function may
1382/// fail to evaluate (similar to a panic during const-evaluation). This behavior may change in the
1383/// future.
1384///
1385/// The limitation is illustrated by the following example:
1386///
1387/// ```
1388/// use std::mem::size_of;
1389/// use std::ptr;
1390///
1391/// const { unsafe {
1392///     const PTR_SIZE: usize = size_of::<*const i32>();
1393///     let mut data1 = [0u8; PTR_SIZE];
1394///     let mut data2 = [0u8; PTR_SIZE];
1395///     // Store a pointer in `data1`.
1396///     data1.as_mut_ptr().cast::<*const i32>().write_unaligned(&42);
1397///     // Swap the contents of `data1` and `data2` by swapping `PTR_SIZE` many `u8`-sized chunks.
1398///     // This call will fail, because the pointer in `data1` crosses the boundary
1399///     // between several of the 1-byte chunks that are being swapped here.
1400///     //ptr::swap_nonoverlapping(data1.as_mut_ptr(), data2.as_mut_ptr(), PTR_SIZE);
1401///     // Swap the contents of `data1` and `data2` by swapping a single chunk of size
1402///     // `[u8; PTR_SIZE]`. That works, as there is no pointer crossing the boundary between
1403///     // two chunks.
1404///     ptr::swap_nonoverlapping(&mut data1, &mut data2, 1);
1405///     // Read the pointer from `data2` and dereference it.
1406///     let ptr = data2.as_ptr().cast::<*const i32>().read_unaligned();
1407///     assert!(*ptr == 42);
1408/// } }
1409/// ```
1410#[inline]
1411#[stable(feature = "swap_nonoverlapping", since = "1.27.0")]
1412#[rustc_const_stable(feature = "const_swap_nonoverlapping", since = "1.88.0")]
1413#[rustc_diagnostic_item = "ptr_swap_nonoverlapping"]
1414#[rustc_allow_const_fn_unstable(const_eval_select)] // both implementations behave the same
1415#[track_caller]
1416#[cfg(not(feature = "ferrocene_certified"))]
1417pub const unsafe fn swap_nonoverlapping<T>(x: *mut T, y: *mut T, count: usize) {
1418    ub_checks::assert_unsafe_precondition!(
1419        check_library_ub,
1420        "ptr::swap_nonoverlapping requires that both pointer arguments are aligned and non-null \
1421        and the specified memory ranges do not overlap",
1422        (
1423            x: *mut () = x as *mut (),
1424            y: *mut () = y as *mut (),
1425            size: usize = size_of::<T>(),
1426            align: usize = align_of::<T>(),
1427            count: usize = count,
1428        ) => {
1429            let zero_size = size == 0 || count == 0;
1430            ub_checks::maybe_is_aligned_and_not_null(x, align, zero_size)
1431                && ub_checks::maybe_is_aligned_and_not_null(y, align, zero_size)
1432                && ub_checks::maybe_is_nonoverlapping(x, y, size, count)
1433        }
1434    );
1435
1436    const_eval_select!(
1437        @capture[T] { x: *mut T, y: *mut T, count: usize }:
1438        if const {
1439            // At compile-time we want to always copy this in chunks of `T`, to ensure that if there
1440            // are pointers inside `T` we will copy them in one go rather than trying to copy a part
1441            // of a pointer (which would not work).
1442            // SAFETY: Same preconditions as this function
1443            unsafe { swap_nonoverlapping_const(x, y, count) }
1444        } else {
1445            // Going though a slice here helps codegen know the size fits in `isize`
1446            let slice = slice_from_raw_parts_mut(x, count);
1447            // SAFETY: This is all readable from the pointer, meaning it's one
1448            // allocation, and thus cannot be more than isize::MAX bytes.
1449            let bytes = unsafe { mem::size_of_val_raw::<[T]>(slice) };
1450            if let Some(bytes) = NonZero::new(bytes) {
1451                // SAFETY: These are the same ranges, just expressed in a different
1452                // type, so they're still non-overlapping.
1453                unsafe { swap_nonoverlapping_bytes(x.cast(), y.cast(), bytes) };
1454            }
1455        }
1456    )
1457}

1459/// Same behavior and safety conditions as [`swap_nonoverlapping`]
1460#[inline]
1461#[cfg(not(feature = "ferrocene_certified"))]
1462const unsafe fn swap_nonoverlapping_const<T>(x: *mut T, y: *mut T, count: usize) {
1463    let mut i = 0;
1464    while i < count {
1465        // SAFETY: By precondition, `i` is in-bounds because it's below `n`
1466        let x = unsafe { x.add(i) };
1467        // SAFETY: By precondition, `i` is in-bounds because it's below `n`
1468        // and it's distinct from `x` since the ranges are non-overlapping
1469        let y = unsafe { y.add(i) };
1470
1471        // SAFETY: we're only ever given pointers that are valid to read/write,
1472        // including being aligned, and nothing here panics so it's drop-safe.
1473        unsafe {
1474            // Note that it's critical that these use `copy_nonoverlapping`,
1475            // rather than `read`/`write`, to avoid #134713 if T has padding.
1476            let mut temp = MaybeUninit::<T>::uninit();
1477            copy_nonoverlapping(x, temp.as_mut_ptr(), 1);
1478            copy_nonoverlapping(y, x, 1);
1479            copy_nonoverlapping(temp.as_ptr(), y, 1);
1480        }
1481
1482        i += 1;
1483    }
1484}

1486// Don't let MIR inline this, because we really want it to keep its noalias metadata
1487#[rustc_no_mir_inline]
1488#[inline]
1489#[cfg(not(feature = "ferrocene_certified"))]
1490fn swap_chunk<const N: usize>(x: &mut MaybeUninit<[u8; N]>, y: &mut MaybeUninit<[u8; N]>) {
1491    let a = *x;
1492    let b = *y;
1493    *x = b;
1494    *y = a;
1495}
1496
1497#[inline]
1498#[cfg(not(feature = "ferrocene_certified"))]
1499unsafe fn swap_nonoverlapping_bytes(x: *mut u8, y: *mut u8, bytes: NonZero<usize>) {
1500    // Same as `swap_nonoverlapping::<[u8; N]>`.
1501    unsafe fn swap_nonoverlapping_chunks<const N: usize>(
1502        x: *mut MaybeUninit<[u8; N]>,
1503        y: *mut MaybeUninit<[u8; N]>,
1504        chunks: NonZero<usize>,
1505    ) {
1506        let chunks = chunks.get();
1507        for i in 0..chunks {
1508            // SAFETY: i is in [0, chunks) so the adds and dereferences are in-bounds.
1509            unsafe { swap_chunk(&mut *x.add(i), &mut *y.add(i)) };
1510        }
1511    }
1512
1513    // Same as `swap_nonoverlapping_bytes`, but accepts at most 1+2+4=7 bytes
1514    #[inline]
1515    unsafe fn swap_nonoverlapping_short(x: *mut u8, y: *mut u8, bytes: NonZero<usize>) {
1516        // Tail handling for auto-vectorized code sometimes has element-at-a-time behaviour,
1517        // see <https://github.com/rust-lang/rust/issues/134946>.
1518        // By swapping as different sizes, rather than as a loop over bytes,
1519        // we make sure not to end up with, say, seven byte-at-a-time copies.
1520
1521        let bytes = bytes.get();
1522        let mut i = 0;
1523        macro_rules! swap_prefix {
1524            ($($n:literal)+) => {$(
1525                if (bytes & $n) != 0 {
1526                    // SAFETY: `i` can only have the same bits set as those in bytes,
1527                    // so these `add`s are in-bounds of `bytes`.  But the bit for
1528                    // `$n` hasn't been set yet, so the `$n` bytes that `swap_chunk`
1529                    // will read and write are within the usable range.
1530                    unsafe { swap_chunk::<$n>(&mut*x.add(i).cast(), &mut*y.add(i).cast()) };
1531                    i |= $n;
1532                }
1533            )+};
1534        }
1535        swap_prefix!(4 2 1);
1536        debug_assert_eq!(i, bytes);
1537    }
1538
1539    const CHUNK_SIZE: usize = size_of::<*const ()>();
1540    let bytes = bytes.get();
1541
1542    let chunks = bytes / CHUNK_SIZE;
1543    let tail = bytes % CHUNK_SIZE;
1544    if let Some(chunks) = NonZero::new(chunks) {
1545        // SAFETY: this is bytes/CHUNK_SIZE*CHUNK_SIZE bytes, which is <= bytes,
1546        // so it's within the range of our non-overlapping bytes.
1547        unsafe { swap_nonoverlapping_chunks::<CHUNK_SIZE>(x.cast(), y.cast(), chunks) };
1548    }
1549    if let Some(tail) = NonZero::new(tail) {
1550        const { assert!(CHUNK_SIZE <= 8) };
1551        let delta = chunks * CHUNK_SIZE;
1552        // SAFETY: the tail length is below CHUNK SIZE because of the remainder,
1553        // and CHUNK_SIZE is at most 8 by the const assert, so tail <= 7
1554        unsafe { swap_nonoverlapping_short(x.add(delta), y.add(delta), tail) };
1555    }
1556}

1562/// This function is semantically equivalent to [`mem::replace`] except that it
1563/// operates on raw pointers instead of references. When references are
1564/// available, [`mem::replace`] should be preferred.
1565///
1566/// # Safety
1567///
1568/// Behavior is undefined if any of the following conditions are violated:
1569///
1570/// * `dst` must be [valid] for both reads and writes.
1571///
1572/// * `dst` must be properly aligned.
1573///
1574/// * `dst` must point to a properly initialized value of type `T`.
1575///
1576/// Note that even if `T` has size `0`, the pointer must be properly aligned.
1577///
1578/// [valid]: self#safety
1579///
1580/// # Examples
1581///
1582/// ```
1583/// use std::ptr;
1584///
1585/// let mut rust = vec!['b', 'u', 's', 't'];
1586///
1587/// // `mem::replace` would have the same effect without requiring the unsafe
1588/// // block.
1589/// let b = unsafe {
1590///     ptr::replace(&mut rust[0], 'r')
1591/// };
1592///
1593/// assert_eq!(b, 'b');
1594/// assert_eq!(rust, &['r', 'u', 's', 't']);
1595/// ```
1596#[inline]
1597#[stable(feature = "rust1", since = "1.0.0")]
1598#[rustc_const_stable(feature = "const_replace", since = "1.83.0")]
1599#[rustc_diagnostic_item = "ptr_replace"]
1600#[track_caller]
1601#[cfg(not(feature = "ferrocene_certified"))]
1602pub const unsafe fn replace<T>(dst: *mut T, src: T) -> T {
1603    // SAFETY: the caller must guarantee that `dst` is valid to be
1604    // cast to a mutable reference (valid for writes, aligned, initialized),
1605    // and cannot overlap `src` since `dst` must point to a distinct
1606    // allocation.
1607    unsafe {
1608        ub_checks::assert_unsafe_precondition!(
1609            check_language_ub,
1610            "ptr::replace requires that the pointer argument is aligned and non-null",
1611            (
1612                addr: *const () = dst as *const (),
1613                align: usize = align_of::<T>(),
1614                is_zst: bool = T::IS_ZST,
1615            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, is_zst)
1616        );
1617        mem::replace(&mut *dst, src)
1618    }
1619}

1621/// Reads the value from `src` without moving it. This leaves the
1622/// memory in `src` unchanged.
1623///
1624/// # Safety
1625///
1626/// Behavior is undefined if any of the following conditions are violated:
1627///
1628/// * `src` must be [valid] for reads.
1629///
1630/// * `src` must be properly aligned. Use [`read_unaligned`] if this is not the
1631///   case.
1632///
1633/// * `src` must point to a properly initialized value of type `T`.
1634///
1635/// Note that even if `T` has size `0`, the pointer must be properly aligned.
1636///
1637/// # Examples
1638///
1639/// Basic usage:
1640///
1641/// ```
1642/// let x = 12;
1643/// let y = &x as *const i32;
1644///
1645/// unsafe {
1646///     assert_eq!(std::ptr::read(y), 12);
1647/// }
1648/// ```
1649///
1650/// Manually implement [`mem::swap`]:
1651///
1652/// ```
1653/// use std::ptr;
1654///
1655/// fn swap<T>(a: &mut T, b: &mut T) {
1656///     unsafe {
1657///         // Create a bitwise copy of the value at `a` in `tmp`.
1658///         let tmp = ptr::read(a);
1659///
1660///         // Exiting at this point (either by explicitly returning or by
1661///         // calling a function which panics) would cause the value in `tmp` to
1662///         // be dropped while the same value is still referenced by `a`. This
1663///         // could trigger undefined behavior if `T` is not `Copy`.
1664///
1665///         // Create a bitwise copy of the value at `b` in `a`.
1666///         // This is safe because mutable references cannot alias.
1667///         ptr::copy_nonoverlapping(b, a, 1);
1668///
1669///         // As above, exiting here could trigger undefined behavior because
1670///         // the same value is referenced by `a` and `b`.
1671///
1672///         // Move `tmp` into `b`.
1673///         ptr::write(b, tmp);
1674///
1675///         // `tmp` has been moved (`write` takes ownership of its second argument),
1676///         // so nothing is dropped implicitly here.
1677///     }
1678/// }
1679///
1680/// let mut foo = "foo".to_owned();
1681/// let mut bar = "bar".to_owned();
1682///
1683/// swap(&mut foo, &mut bar);
1684///
1685/// assert_eq!(foo, "bar");
1686/// assert_eq!(bar, "foo");
1687/// ```
1688///
1689/// ## Ownership of the Returned Value
1690///
1691/// `read` creates a bitwise copy of `T`, regardless of whether `T` is [`Copy`].
1692/// If `T` is not [`Copy`], using both the returned value and the value at
1693/// `*src` can violate memory safety. Note that assigning to `*src` counts as a
1694/// use because it will attempt to drop the value at `*src`.
1695///
1696/// [`write()`] can be used to overwrite data without causing it to be dropped.
1697///
1698/// ```
1699/// use std::ptr;
1700///
1701/// let mut s = String::from("foo");
1702/// unsafe {
1703///     // `s2` now points to the same underlying memory as `s`.
1704///     let mut s2: String = ptr::read(&s);
1705///
1706///     assert_eq!(s2, "foo");
1707///
1708///     // Assigning to `s2` causes its original value to be dropped. Beyond
1709///     // this point, `s` must no longer be used, as the underlying memory has
1710///     // been freed.
1711///     s2 = String::default();
1712///     assert_eq!(s2, "");
1713///
1714///     // Assigning to `s` would cause the old value to be dropped again,
1715///     // resulting in undefined behavior.
1716///     // s = String::from("bar"); // ERROR
1717///
1718///     // `ptr::write` can be used to overwrite a value without dropping it.
1719///     ptr::write(&mut s, String::from("bar"));
1720/// }
1721///
1722/// assert_eq!(s, "bar");
1723/// ```
1724///
1725/// [valid]: self#safety
1726#[inline]
1727#[stable(feature = "rust1", since = "1.0.0")]
1728#[rustc_const_stable(feature = "const_ptr_read", since = "1.71.0")]
1729#[track_caller]
1730#[rustc_diagnostic_item = "ptr_read"]
1731pub const unsafe fn read<T>(src: *const T) -> T {
1732    // It would be semantically correct to implement this via `copy_nonoverlapping`
1733    // and `MaybeUninit`, as was done before PR #109035. Calling `assume_init`
1734    // provides enough information to know that this is a typed operation.
1735
1736    // However, as of March 2023 the compiler was not capable of taking advantage
1737    // of that information. Thus, the implementation here switched to an intrinsic,
1738    // which lowers to `_0 = *src` in MIR, to address a few issues:
1739    //
1740    // - Using `MaybeUninit::assume_init` after a `copy_nonoverlapping` was not
1741    //   turning the untyped copy into a typed load. As such, the generated
1742    //   `load` in LLVM didn't get various metadata, such as `!range` (#73258),
1743    //   `!nonnull`, and `!noundef`, resulting in poorer optimization.
1744    // - Going through the extra local resulted in multiple extra copies, even
1745    //   in optimized MIR.  (Ignoring StorageLive/Dead, the intrinsic is one
1746    //   MIR statement, while the previous implementation was eight.)  LLVM
1747    //   could sometimes optimize them away, but because `read` is at the core
1748    //   of so many things, not having them in the first place improves what we
1749    //   hand off to the backend.  For example, `mem::replace::<Big>` previously
1750    //   emitted 4 `alloca` and 6 `memcpy`s, but is now 1 `alloc` and 3 `memcpy`s.
1751    // - In general, this approach keeps us from getting any more bugs (like
1752    //   #106369) that boil down to "`read(p)` is worse than `*p`", as this
1753    //   makes them look identical to the backend (or other MIR consumers).
1754    //
1755    // Future enhancements to MIR optimizations might well allow this to return
1756    // to the previous implementation, rather than using an intrinsic.
1757
1758    // SAFETY: the caller must guarantee that `src` is valid for reads.
1759    unsafe {
1760        #[cfg(debug_assertions)] // Too expensive to always enable (for now?)
1761        ub_checks::assert_unsafe_precondition!(
1762            check_language_ub,
1763            "ptr::read requires that the pointer argument is aligned and non-null",
1764            (
1765                addr: *const () = src as *const (),
1766                align: usize = align_of::<T>(),
1767                is_zst: bool = T::IS_ZST,
1768            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, is_zst)
1769        );
1770        crate::intrinsics::read_via_copy(src)
1771    }
1772}

1774/// Reads the value from `src` without moving it. This leaves the
1775/// memory in `src` unchanged.
1776///
1777/// Unlike [`read`], `read_unaligned` works with unaligned pointers.
1778///
1779/// # Safety
1780///
1781/// Behavior is undefined if any of the following conditions are violated:
1782///
1783/// * `src` must be [valid] for reads.
1784///
1785/// * `src` must point to a properly initialized value of type `T`.
1786///
1787/// Like [`read`], `read_unaligned` creates a bitwise copy of `T`, regardless of
1788/// whether `T` is [`Copy`]. If `T` is not [`Copy`], using both the returned
1789/// value and the value at `*src` can [violate memory safety][read-ownership].
1790///
1791/// [read-ownership]: read#ownership-of-the-returned-value
1792/// [valid]: self#safety
1793///
1794/// ## On `packed` structs
1795///
1796/// Attempting to create a raw pointer to an `unaligned` struct field with
1797/// an expression such as `&packed.unaligned as *const FieldType` creates an
1798/// intermediate unaligned reference before converting that to a raw pointer.
1799/// That this reference is temporary and immediately cast is inconsequential
1800/// as the compiler always expects references to be properly aligned.
1801/// As a result, using `&packed.unaligned as *const FieldType` causes immediate
1802/// *undefined behavior* in your program.
1803///
1804/// Instead you must use the `&raw const` syntax to create the pointer.
1805/// You may use that constructed pointer together with this function.
1806///
1807/// An example of what not to do and how this relates to `read_unaligned` is:
1808///
1809/// ```
1810/// #[repr(packed, C)]
1811/// struct Packed {
1812///     _padding: u8,
1813///     unaligned: u32,
1814/// }
1815///
1816/// let packed = Packed {
1817///     _padding: 0x00,
1818///     unaligned: 0x01020304,
1819/// };
1820///
1821/// // Take the address of a 32-bit integer which is not aligned.
1822/// // In contrast to `&packed.unaligned as *const _`, this has no undefined behavior.
1823/// let unaligned = &raw const packed.unaligned;
1824///
1825/// let v = unsafe { std::ptr::read_unaligned(unaligned) };
1826/// assert_eq!(v, 0x01020304);
1827/// ```
1828///
1829/// Accessing unaligned fields directly with e.g. `packed.unaligned` is safe however.
1830///
1831/// # Examples
1832///
1833/// Read a `usize` value from a byte buffer:
1834///
1835/// ```
1836/// fn read_usize(x: &[u8]) -> usize {
1837///     assert!(x.len() >= size_of::<usize>());
1838///
1839///     let ptr = x.as_ptr() as *const usize;
1840///
1841///     unsafe { ptr.read_unaligned() }
1842/// }
1843/// ```
1844#[inline]
1845#[stable(feature = "ptr_unaligned", since = "1.17.0")]
1846#[rustc_const_stable(feature = "const_ptr_read", since = "1.71.0")]
1847#[track_caller]
1848#[rustc_diagnostic_item = "ptr_read_unaligned"]
1849#[cfg(not(feature = "ferrocene_certified"))]
1850pub const unsafe fn read_unaligned<T>(src: *const T) -> T {
1851    let mut tmp = MaybeUninit::<T>::uninit();
1852    // SAFETY: the caller must guarantee that `src` is valid for reads.
1853    // `src` cannot overlap `tmp` because `tmp` was just allocated on
1854    // the stack as a separate allocation.
1855    //
1856    // Also, since we just wrote a valid value into `tmp`, it is guaranteed
1857    // to be properly initialized.
1858    unsafe {
1859        copy_nonoverlapping(src as *const u8, tmp.as_mut_ptr() as *mut u8, size_of::<T>());
1860        tmp.assume_init()
1861    }
1862}

2064/// Performs a volatile read of the value from `src` without moving it. This
2065/// leaves the memory in `src` unchanged.
2066///
2067/// Volatile operations are intended to act on I/O memory, and are guaranteed
2068/// to not be elided or reordered by the compiler across other volatile
2069/// operations.
2070///
2071/// # Notes
2072///
2073/// Rust does not currently have a rigorously and formally defined memory model,
2074/// so the precise semantics of what "volatile" means here is subject to change
2075/// over time. That being said, the semantics will almost always end up pretty
2076/// similar to [C11's definition of volatile][c11].
2077///
2078/// The compiler shouldn't change the relative order or number of volatile
2079/// memory operations. However, volatile memory operations on zero-sized types
2080/// (e.g., if a zero-sized type is passed to `read_volatile`) are noops
2081/// and may be ignored.
2082///
2083/// [c11]: http://www.open-std.org/jtc1/sc22/wg14/www/docs/n1570.pdf
2084///
2085/// # Safety
2086///
2087/// Behavior is undefined if any of the following conditions are violated:
2088///
2089/// * `src` must be [valid] for reads.
2090///
2091/// * `src` must be properly aligned.
2092///
2093/// * `src` must point to a properly initialized value of type `T`.
2094///
2095/// Like [`read`], `read_volatile` creates a bitwise copy of `T`, regardless of
2096/// whether `T` is [`Copy`]. If `T` is not [`Copy`], using both the returned
2097/// value and the value at `*src` can [violate memory safety][read-ownership].
2098/// However, storing non-[`Copy`] types in volatile memory is almost certainly
2099/// incorrect.
2100///
2101/// Note that even if `T` has size `0`, the pointer must be properly aligned.
2102///
2103/// [valid]: self#safety
2104/// [read-ownership]: read#ownership-of-the-returned-value
2105///
2106/// Just like in C, whether an operation is volatile has no bearing whatsoever
2107/// on questions involving concurrent access from multiple threads. Volatile
2108/// accesses behave exactly like non-atomic accesses in that regard. In particular,
2109/// a race between a `read_volatile` and any write operation to the same location
2110/// is undefined behavior.
2111///
2112/// # Examples
2113///
2114/// Basic usage:
2115///
2116/// ```
2117/// let x = 12;
2118/// let y = &x as *const i32;
2119///
2120/// unsafe {
2121///     assert_eq!(std::ptr::read_volatile(y), 12);
2122/// }
2123/// ```
2124#[inline]
2125#[stable(feature = "volatile", since = "1.9.0")]
2126#[track_caller]
2127#[rustc_diagnostic_item = "ptr_read_volatile"]
2128#[cfg(not(feature = "ferrocene_certified"))]
2129pub unsafe fn read_volatile<T>(src: *const T) -> T {
2130    // SAFETY: the caller must uphold the safety contract for `volatile_load`.
2131    unsafe {
2132        ub_checks::assert_unsafe_precondition!(
2133            check_language_ub,
2134            "ptr::read_volatile requires that the pointer argument is aligned and non-null",
2135            (
2136                addr: *const () = src as *const (),
2137                align: usize = align_of::<T>(),
2138                is_zst: bool = T::IS_ZST,
2139            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, is_zst)
2140        );
2141        intrinsics::volatile_load(src)
2142    }
2143}

2161/// Rust does not currently have a rigorously and formally defined memory model,
2162/// so the precise semantics of what "volatile" means here is subject to change
2163/// over time. That being said, the semantics will almost always end up pretty
2164/// similar to [C11's definition of volatile][c11].
2165///
2166/// The compiler shouldn't change the relative order or number of volatile
2167/// memory operations. However, volatile memory operations on zero-sized types
2168/// (e.g., if a zero-sized type is passed to `write_volatile`) are noops
2169/// and may be ignored.
2170///
2171/// [c11]: http://www.open-std.org/jtc1/sc22/wg14/www/docs/n1570.pdf
2172///
2173/// # Safety
2174///
2175/// Behavior is undefined if any of the following conditions are violated:
2176///
2177/// * `dst` must be [valid] for writes.
2178///
2179/// * `dst` must be properly aligned.
2180///
2181/// Note that even if `T` has size `0`, the pointer must be properly aligned.
2182///
2183/// [valid]: self#safety
2184///
2185/// Just like in C, whether an operation is volatile has no bearing whatsoever
2186/// on questions involving concurrent access from multiple threads. Volatile
2187/// accesses behave exactly like non-atomic accesses in that regard. In particular,
2188/// a race between a `write_volatile` and any other operation (reading or writing)
2189/// on the same location is undefined behavior.
2190///
2191/// # Examples
2192///
2193/// Basic usage:
2194///
2195/// ```
2196/// let mut x = 0;
2197/// let y = &mut x as *mut i32;
2198/// let z = 12;
2199///
2200/// unsafe {
2201///     std::ptr::write_volatile(y, z);
2202///     assert_eq!(std::ptr::read_volatile(y), 12);
2203/// }
2204/// ```
2205#[inline]
2206#[stable(feature = "volatile", since = "1.9.0")]
2207#[rustc_diagnostic_item = "ptr_write_volatile"]
2208#[track_caller]
2209#[cfg(not(feature = "ferrocene_certified"))]
2210pub unsafe fn write_volatile<T>(dst: *mut T, src: T) {
2211    // SAFETY: the caller must uphold the safety contract for `volatile_store`.
2212    unsafe {
2213        ub_checks::assert_unsafe_precondition!(
2214            check_language_ub,
2215            "ptr::write_volatile requires that the pointer argument is aligned and non-null",
2216            (
2217                addr: *mut () = dst as *mut (),
2218                align: usize = align_of::<T>(),
2219                is_zst: bool = T::IS_ZST,
2220            ) => ub_checks::maybe_is_aligned_and_not_null(addr, align, is_zst)
2221        );
2222        intrinsics::volatile_store(dst, src);
2223    }
2224}

2226/// Align pointer `p`.
2227///
2228/// Calculate offset (in terms of elements of `size_of::<T>()` stride) that has to be applied
2229/// to pointer `p` so that pointer `p` would get aligned to `a`.
2230///
2231/// # Safety
2232/// `a` must be a power of two.
2233///
2234/// # Notes
2235/// This implementation has been carefully tailored to not panic. It is UB for this to panic.
2236/// The only real change that can be made here is change of `INV_TABLE_MOD_16` and associated
2237/// constants.
2238///
2239/// If we ever decide to make it possible to call the intrinsic with `a` that is not a
2240/// power-of-two, it will probably be more prudent to just change to a naive implementation rather
2241/// than trying to adapt this to accommodate that change.
2242///
2243/// Any questions go to @nagisa.
2244#[allow(ptr_to_integer_transmute_in_consts)]
2245#[cfg(not(feature = "ferrocene_certified"))]
2246pub(crate) unsafe fn align_offset<T: Sized>(p: *const T, a: usize) -> usize {
2247    // FIXME(#75598): Direct use of these intrinsics improves codegen significantly at opt-level <=
2248    // 1, where the method versions of these operations are not inlined.
2249    use intrinsics::{
2250        assume, cttz_nonzero, exact_div, mul_with_overflow, unchecked_rem, unchecked_shl,
2251        unchecked_shr, unchecked_sub, wrapping_add, wrapping_mul, wrapping_sub,
2252    };
2253
2254    /// Calculate multiplicative modular inverse of `x` modulo `m`.
2255    ///
2256    /// This implementation is tailored for `align_offset` and has following preconditions:
2257    ///
2258    /// * `m` is a power-of-two;
2259    /// * `x < m`; (if `x ≥ m`, pass in `x % m` instead)
2260    ///
2261    /// Implementation of this function shall not panic. Ever.
2262    #[inline]
2263    const unsafe fn mod_inv(x: usize, m: usize) -> usize {
2264        /// Multiplicative modular inverse table modulo 2⁴ = 16.
2265        ///
2266        /// Note, that this table does not contain values where inverse does not exist (i.e., for
2267        /// `0⁻¹ mod 16`, `2⁻¹ mod 16`, etc.)
2268        const INV_TABLE_MOD_16: [u8; 8] = [1, 11, 13, 7, 9, 3, 5, 15];
2269        /// Modulo for which the `INV_TABLE_MOD_16` is intended.
2270        const INV_TABLE_MOD: usize = 16;
2271
2272        // SAFETY: `m` is required to be a power-of-two, hence non-zero.
2273        let m_minus_one = unsafe { unchecked_sub(m, 1) };
2274        let mut inverse = INV_TABLE_MOD_16[(x & (INV_TABLE_MOD - 1)) >> 1] as usize;
2275        let mut mod_gate = INV_TABLE_MOD;
2276        // We iterate "up" using the following formula:
2277        //
2278        // $$ xy ≡ 1 (mod 2ⁿ) → xy (2 - xy) ≡ 1 (mod 2²ⁿ) $$
2279        //
2280        // This application needs to be applied at least until `2²ⁿ ≥ m`, at which point we can
2281        // finally reduce the computation to our desired `m` by taking `inverse mod m`.
2282        //
2283        // This computation is `O(log log m)`, which is to say, that on 64-bit machines this loop
2284        // will always finish in at most 4 iterations.
2285        loop {
2286            // y = y * (2 - xy) mod n
2287            //
2288            // Note, that we use wrapping operations here intentionally – the original formula
2289            // uses e.g., subtraction `mod n`. It is entirely fine to do them `mod
2290            // usize::MAX` instead, because we take the result `mod n` at the end
2291            // anyway.
2292            if mod_gate >= m {
2293                break;
2294            }
2295            inverse = wrapping_mul(inverse, wrapping_sub(2usize, wrapping_mul(x, inverse)));
2296            let (new_gate, overflow) = mul_with_overflow(mod_gate, mod_gate);
2297            if overflow {
2298                break;
2299            }
2300            mod_gate = new_gate;
2301        }
2302        inverse & m_minus_one
2303    }
2304
2305    let stride = size_of::<T>();
2306
2307    let addr: usize = p.addr();
2308
2309    // SAFETY: `a` is a power-of-two, therefore non-zero.
2310    let a_minus_one = unsafe { unchecked_sub(a, 1) };
2311
2312    if stride == 0 {
2313        // SPECIAL_CASE: handle 0-sized types. No matter how many times we step, the address will
2314        // stay the same, so no offset will be able to align the pointer unless it is already
2315        // aligned. This branch _will_ be optimized out as `stride` is known at compile-time.
2316        let p_mod_a = addr & a_minus_one;
2317        return if p_mod_a == 0 { 0 } else { usize::MAX };
2318    }
2319
2320    // SAFETY: `stride == 0` case has been handled by the special case above.
2321    let a_mod_stride = unsafe { unchecked_rem(a, stride) };
2322    if a_mod_stride == 0 {
2323        // SPECIAL_CASE: In cases where the `a` is divisible by `stride`, byte offset to align a
2324        // pointer can be computed more simply through `-p (mod a)`. In the off-chance the byte
2325        // offset is not a multiple of `stride`, the input pointer was misaligned and no pointer
2326        // offset will be able to produce a `p` aligned to the specified `a`.
2327        //
2328        // The naive `-p (mod a)` equation inhibits LLVM's ability to select instructions
2329        // like `lea`. We compute `(round_up_to_next_alignment(p, a) - p)` instead. This
2330        // redistributes operations around the load-bearing, but pessimizing `and` instruction
2331        // sufficiently for LLVM to be able to utilize the various optimizations it knows about.
2332        //
2333        // LLVM handles the branch here particularly nicely. If this branch needs to be evaluated
2334        // at runtime, it will produce a mask `if addr_mod_stride == 0 { 0 } else { usize::MAX }`
2335        // in a branch-free way and then bitwise-OR it with whatever result the `-p mod a`
2336        // computation produces.
2337
2338        let aligned_address = wrapping_add(addr, a_minus_one) & wrapping_sub(0, a);
2339        let byte_offset = wrapping_sub(aligned_address, addr);
2340        // FIXME: Remove the assume after <https://github.com/llvm/llvm-project/issues/62502>
2341        // SAFETY: Masking by `-a` can only affect the low bits, and thus cannot have reduced
2342        // the value by more than `a-1`, so even though the intermediate values might have
2343        // wrapped, the byte_offset is always in `[0, a)`.
2344        unsafe { assume(byte_offset < a) };
2345
2346        // SAFETY: `stride == 0` case has been handled by the special case above.
2347        let addr_mod_stride = unsafe { unchecked_rem(addr, stride) };
2348
2349        return if addr_mod_stride == 0 {
2350            // SAFETY: `stride` is non-zero. This is guaranteed to divide exactly as well, because
2351            // addr has been verified to be aligned to the original type’s alignment requirements.
2352            unsafe { exact_div(byte_offset, stride) }
2353        } else {
2354            usize::MAX
2355        };
2356    }
2357
2358    // GENERAL_CASE: From here on we’re handling the very general case where `addr` may be
2359    // misaligned, there isn’t an obvious relationship between `stride` and `a` that we can take an
2360    // advantage of, etc. This case produces machine code that isn’t particularly high quality,
2361    // compared to the special cases above. The code produced here is still within the realm of
2362    // miracles, given the situations this case has to deal with.
2363
2364    // SAFETY: a is power-of-two hence non-zero. stride == 0 case is handled above.
2365    // FIXME(const-hack) replace with min
2366    let gcdpow = unsafe {
2367        let x = cttz_nonzero(stride);
2368        let y = cttz_nonzero(a);
2369        if x < y { x } else { y }
2370    };
2371    // SAFETY: gcdpow has an upper-bound that’s at most the number of bits in a `usize`.
2372    let gcd = unsafe { unchecked_shl(1usize, gcdpow) };
2373    // SAFETY: gcd is always greater or equal to 1.
2374    if addr & unsafe { unchecked_sub(gcd, 1) } == 0 {
2375        // This branch solves for the following linear congruence equation:
2376        //
2377        // ` p + so = 0 mod a `
2378        //
2379        // `p` here is the pointer value, `s` - stride of `T`, `o` offset in `T`s, and `a` - the
2380        // requested alignment.
2381        //
2382        // With `g = gcd(a, s)`, and the above condition asserting that `p` is also divisible by
2383        // `g`, we can denote `a' = a/g`, `s' = s/g`, `p' = p/g`, then this becomes equivalent to:
2384        //
2385        // ` p' + s'o = 0 mod a' `
2386        // ` o = (a' - (p' mod a')) * (s'^-1 mod a') `
2387        //
2388        // The first term is "the relative alignment of `p` to `a`" (divided by the `g`), the
2389        // second term is "how does incrementing `p` by `s` bytes change the relative alignment of
2390        // `p`" (again divided by `g`). Division by `g` is necessary to make the inverse well
2391        // formed if `a` and `s` are not co-prime.
2392        //
2393        // Furthermore, the result produced by this solution is not "minimal", so it is necessary
2394        // to take the result `o mod lcm(s, a)`. This `lcm(s, a)` is the same as `a'`.
2395
2396        // SAFETY: `gcdpow` has an upper-bound not greater than the number of trailing 0-bits in
2397        // `a`.
2398        let a2 = unsafe { unchecked_shr(a, gcdpow) };
2399        // SAFETY: `a2` is non-zero. Shifting `a` by `gcdpow` cannot shift out any of the set bits
2400        // in `a` (of which it has exactly one).
2401        let a2minus1 = unsafe { unchecked_sub(a2, 1) };
2402        // SAFETY: `gcdpow` has an upper-bound not greater than the number of trailing 0-bits in
2403        // `a`.
2404        let s2 = unsafe { unchecked_shr(stride & a_minus_one, gcdpow) };
2405        // SAFETY: `gcdpow` has an upper-bound not greater than the number of trailing 0-bits in
2406        // `a`. Furthermore, the subtraction cannot overflow, because `a2 = a >> gcdpow` will
2407        // always be strictly greater than `(p % a) >> gcdpow`.
2408        let minusp2 = unsafe { unchecked_sub(a2, unchecked_shr(addr & a_minus_one, gcdpow)) };
2409        // SAFETY: `a2` is a power-of-two, as proven above. `s2` is strictly less than `a2`
2410        // because `(s % a) >> gcdpow` is strictly less than `a >> gcdpow`.
2411        return wrapping_mul(minusp2, unsafe { mod_inv(s2, a2) }) & a2minus1;
2412    }
2413
2414    // Cannot be aligned at all.
2415    usize::MAX
2416}

2418/// Compares raw pointers for equality.
2419///
2420/// This is the same as using the `==` operator, but less generic:
2421/// the arguments have to be `*const T` raw pointers,
2422/// not anything that implements `PartialEq`.
2423///
2424/// This can be used to compare `&T` references (which coerce to `*const T` implicitly)
2425/// by their address rather than comparing the values they point to
2426/// (which is what the `PartialEq for &T` implementation does).
2427///
2428/// When comparing wide pointers, both the address and the metadata are tested for equality.
2429/// However, note that comparing trait object pointers (`*const dyn Trait`) is unreliable: pointers
2430/// to values of the same underlying type can compare inequal (because vtables are duplicated in
2431/// multiple codegen units), and pointers to values of *different* underlying type can compare equal
2432/// (since identical vtables can be deduplicated within a codegen unit).
2433///
2434/// # Examples
2435///
2436/// ```
2437/// use std::ptr;
2438///
2439/// let five = 5;
2440/// let other_five = 5;
2441/// let five_ref = &five;
2442/// let same_five_ref = &five;
2443/// let other_five_ref = &other_five;
2444///
2445/// assert!(five_ref == same_five_ref);
2446/// assert!(ptr::eq(five_ref, same_five_ref));
2447///
2448/// assert!(five_ref == other_five_ref);
2449/// assert!(!ptr::eq(five_ref, other_five_ref));
2450/// ```
2451///
2452/// Slices are also compared by their length (fat pointers):
2453///
2454/// ```
2455/// let a = [1, 2, 3];
2456/// assert!(std::ptr::eq(&a[..3], &a[..3]));
2457/// assert!(!std::ptr::eq(&a[..2], &a[..3]));
2458/// assert!(!std::ptr::eq(&a[0..2], &a[1..3]));
2459/// ```
2460#[stable(feature = "ptr_eq", since = "1.17.0")]
2461#[inline(always)]
2462#[must_use = "pointer comparison produces a value"]
2463#[rustc_diagnostic_item = "ptr_eq"]
2464#[allow(ambiguous_wide_pointer_comparisons)] // it's actually clear here
2465#[cfg(not(feature = "ferrocene_certified"))]
2466pub fn eq<T: ?Sized>(a: *const T, b: *const T) -> bool {
2467    a == b
2468}

2471/// ignoring any metadata in fat pointers.
2472///
2473/// If the arguments are thin pointers of the same type,
2474/// then this is the same as [`eq`].
2475///
2476/// # Examples
2477///
2478/// ```
2479/// use std::ptr;
2480///
2481/// let whole: &[i32; 3] = &[1, 2, 3];
2482/// let first: &i32 = &whole[0];

2495/// Compares the *addresses* of the two function pointers for equality.
2496///
2497/// This is the same as `f == g`, but using this function makes clear that the potentially
2498/// surprising semantics of function pointer comparison are involved.
2499///
2500/// There are **very few guarantees** about how functions are compiled and they have no intrinsic
2501/// “identity”; in particular, this comparison:
2502///
2503/// * May return `true` unexpectedly, in cases where functions are equivalent.
2504///
2505///   For example, the following program is likely (but not guaranteed) to print `(true, true)`
2506///   when compiled with optimization:
2507///
2508///   ```
2509///   let f: fn(i32) -> i32 = |x| x;
2510///   let g: fn(i32) -> i32 = |x| x + 0;  // different closure, different body
2511///   let h: fn(u32) -> u32 = |x| x + 0;  // different signature too
2512///   dbg!(std::ptr::fn_addr_eq(f, g), std::ptr::fn_addr_eq(f, h)); // not guaranteed to be equal
2513///   ```
2514///
2515/// * May return `false` in any case.
2516///
2517///   This is particularly likely with generic functions but may happen with any function.
2518///   (From an implementation perspective, this is possible because functions may sometimes be
2519///   processed more than once by the compiler, resulting in duplicate machine code.)
2520///
2521/// Despite these false positives and false negatives, this comparison can still be useful.
2522/// Specifically, if
2523///
2524/// * `T` is the same type as `U`, `T` is a [subtype] of `U`, or `U` is a [subtype] of `T`, and
2525/// * `ptr::fn_addr_eq(f, g)` returns true,
2526///
2527/// then calling `f` and calling `g` will be equivalent.
2528///
2529///
2530/// # Examples
2531///
2532/// ```
2533/// use std::ptr;
2534///
2535/// fn a() { println!("a"); }
2536/// fn b() { println!("b"); }
2537/// assert!(!ptr::fn_addr_eq(a as fn(), b as fn()));
2538/// ```
2539///
2540/// [subtype]: https://doc.rust-lang.org/reference/subtyping.html
2541#[stable(feature = "ptr_fn_addr_eq", since = "1.85.0")]
2542#[inline(always)]
2543#[must_use = "function pointer comparison produces a value"]
2544#[cfg(not(feature = "ferrocene_certified"))]
2545pub fn fn_addr_eq<T: FnPtr, U: FnPtr>(f: T, g: U) -> bool {
2546    f.addr() == g.addr()
2547}

2634/// Creates a `const` raw pointer to a place, without creating an intermediate reference.
2635///
2636/// `addr_of!(expr)` is equivalent to `&raw const expr`. The macro is *soft-deprecated*;
2637/// use `&raw const` instead.
2638///
2639/// It is still an open question under which conditions writing through an `addr_of!`-created
2640/// pointer is permitted. If the place `expr` evaluates to is based on a raw pointer, then the
2641/// result of `addr_of!` inherits all permissions from that raw pointer. However, if the place is
2642/// based on a reference, local variable, or `static`, then until all details are decided, the same
2643/// rules as for shared references apply: it is UB to write through a pointer created with this
2644/// operation, except for bytes located inside an `UnsafeCell`. Use `&raw mut` (or [`addr_of_mut`])
2645/// to create a raw pointer that definitely permits mutation.
2646///
2647/// Creating a reference with `&`/`&mut` is only allowed if the pointer is properly aligned
2648/// and points to initialized data. For cases where those requirements do not hold,
2649/// raw pointers should be used instead. However, `&expr as *const _` creates a reference
2650/// before casting it to a raw pointer, and that reference is subject to the same rules
2651/// as all other references. This macro can create a raw pointer *without* creating
2652/// a reference first.
2653///
2654/// See [`addr_of_mut`] for how to create a pointer to uninitialized data.
2655/// Doing that with `addr_of` would not make much sense since one could only
2656/// read the data, and that would be Undefined Behavior.
2657///
2658/// # Safety
2659///
2660/// The `expr` in `addr_of!(expr)` is evaluated as a place expression, but never loads from the
2661/// place or requires the place to be dereferenceable. This means that `addr_of!((*ptr).field)`
2662/// still requires the projection to `field` to be in-bounds, using the same rules as [`offset`].
2663/// However, `addr_of!(*ptr)` is defined behavior even if `ptr` is null, dangling, or misaligned.
2664///
2665/// Note that `Deref`/`Index` coercions (and their mutable counterparts) are applied inside
2666/// `addr_of!` like everywhere else, in which case a reference is created to call `Deref::deref` or
2667/// `Index::index`, respectively. The statements above only apply when no such coercions are
2668/// applied.
2669///
2670/// [`offset`]: pointer::offset
2671///
2672/// # Example
2673///
2674/// **Correct usage: Creating a pointer to unaligned data**
2675///
2676/// ```
2677/// use std::ptr;
2678///
2679/// #[repr(packed)]
2680/// struct Packed {
2681///     f1: u8,
2682///     f2: u16,
2683/// }
2684///
2685/// let packed = Packed { f1: 1, f2: 2 };
2686/// // `&packed.f2` would create an unaligned reference, and thus be Undefined Behavior!
2687/// let raw_f2 = ptr::addr_of!(packed.f2);
2688/// assert_eq!(unsafe { raw_f2.read_unaligned() }, 2);
2689/// ```
2690///
2691/// **Incorrect usage: Out-of-bounds fields projection**
2692///
2693/// ```rust,no_run
2694/// use std::ptr;
2695///
2696/// #[repr(C)]
2697/// struct MyStruct {
2698///     field1: i32,
2699///     field2: i32,
2700/// }
2701///
2702/// let ptr: *const MyStruct = ptr::null();
2703/// let fieldptr = unsafe { ptr::addr_of!((*ptr).field2) }; // Undefined Behavior ⚠️
2704/// ```
2705///
2706/// The field projection `.field2` would offset the pointer by 4 bytes,
2707/// but the pointer is not in-bounds of an allocation for 4 bytes,
2708/// so this offset is Undefined Behavior.
2709/// See the [`offset`] docs for a full list of requirements for inbounds pointer arithmetic; the
2710/// same requirements apply to field projections, even inside `addr_of!`. (In particular, it makes
2711/// no difference whether the pointer is null or dangling.)
2712#[stable(feature = "raw_ref_macros", since = "1.51.0")]
2713#[rustc_macro_transparency = "semitransparent"]
2714#[cfg(not(feature = "ferrocene_certified"))]
2715pub macro addr_of($place:expr) {
2716    &raw const $place
2717}

2719/// Creates a `mut` raw pointer to a place, without creating an intermediate reference.
2720///
2721/// `addr_of_mut!(expr)` is equivalent to `&raw mut expr`. The macro is *soft-deprecated*;
2722/// use `&raw mut` instead.
2723///
2724/// Creating a reference with `&`/`&mut` is only allowed if the pointer is properly aligned
2725/// and points to initialized data. For cases where those requirements do not hold,
2726/// raw pointers should be used instead. However, `&mut expr as *mut _` creates a reference
2727/// before casting it to a raw pointer, and that reference is subject to the same rules
2728/// as all other references. This macro can create a raw pointer *without* creating
2729/// a reference first.
2730///
2731/// # Safety
2732///
2733/// The `expr` in `addr_of_mut!(expr)` is evaluated as a place expression, but never loads from the
2734/// place or requires the place to be dereferenceable. This means that `addr_of_mut!((*ptr).field)`
2735/// still requires the projection to `field` to be in-bounds, using the same rules as [`offset`].
2736/// However, `addr_of_mut!(*ptr)` is defined behavior even if `ptr` is null, dangling, or misaligned.
2737///
2738/// Note that `Deref`/`Index` coercions (and their mutable counterparts) are applied inside
2739/// `addr_of_mut!` like everywhere else, in which case a reference is created to call `Deref::deref`
2740/// or `Index::index`, respectively. The statements above only apply when no such coercions are
2741/// applied.
2742///
2743/// [`offset`]: pointer::offset
2744///
2745/// # Examples
2746///
2747/// **Correct usage: Creating a pointer to unaligned data**
2748///
2749/// ```
2750/// use std::ptr;
2751///
2752/// #[repr(packed)]
2753/// struct Packed {
2754///     f1: u8,
2755///     f2: u16,
2756/// }
2757///
2758/// let mut packed = Packed { f1: 1, f2: 2 };
2759/// // `&mut packed.f2` would create an unaligned reference, and thus be Undefined Behavior!
2760/// let raw_f2 = ptr::addr_of_mut!(packed.f2);
2761/// unsafe { raw_f2.write_unaligned(42); }
2762/// assert_eq!({packed.f2}, 42); // `{...}` forces copying the field instead of creating a reference.
2763/// ```
2764///
2765/// **Correct usage: Creating a pointer to uninitialized data**
2766///
2767/// ```rust
2768/// use std::{ptr, mem::MaybeUninit};
2769///
2770/// struct Demo {
2771///     field: bool,
2772/// }
2773///
2774/// let mut uninit = MaybeUninit::<Demo>::uninit();
2775/// // `&uninit.as_mut().field` would create a reference to an uninitialized `bool`,
2776/// // and thus be Undefined Behavior!
2777/// let f1_ptr = unsafe { ptr::addr_of_mut!((*uninit.as_mut_ptr()).field) };
2778/// unsafe { f1_ptr.write(true); }
2779/// let init = unsafe { uninit.assume_init() };
2780/// ```
2781///
2782/// **Incorrect usage: Out-of-bounds fields projection**
2783///
2784/// ```rust,no_run
2785/// use std::ptr;
2786///
2787/// #[repr(C)]
2788/// struct MyStruct {
2789///     field1: i32,
2790///     field2: i32,
2791/// }
2792///
2793/// let ptr: *mut MyStruct = ptr::null_mut();
2794/// let fieldptr = unsafe { ptr::addr_of_mut!((*ptr).field2) }; // Undefined Behavior ⚠️
2795/// ```
2796///

2798/// but the pointer is not in-bounds of an allocation for 4 bytes,
2799/// so this offset is Undefined Behavior.
